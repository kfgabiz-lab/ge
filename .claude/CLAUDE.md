## 0. ⚠️ 작업 시작 전 필수 순서 체크 (Rule 10 연결 — 매 작업마다 이 순서 확인)
> 아래 순서를 어기는 것은 절대 금지. 설명·승인 없이 다음 단계로 넘어갈 수 없다.

### 빌더 컴포넌트 개발시 
```
STEP 1. builder-contents-layout 확인 (Rule 10-2)
         → 해당 컴포넌트 탭이 없으면 → STEP 2로
STEP 2. 빌더 Preview 개발 진행 - 진행 전 어떤 방식 진행인지 설명 필요.
STEP 3. 퍼블리싱 — builder-contents-layout에 UI 마크업 먼저 구성 → 승인
STEP 4. 빌더 화면 개발 진행 - 빌더 내부 공통 UI 적용해서 진행 - 새로 만들기 금지
STEP 5. Live 기능 구현 - 구현 전에 자세한 설명 필요
```

### BE 포함 작업 (API·DB 변경 있음)
```
STEP 1. builder-contents-layout 확인 (Rule 10-2)
STEP 2. 퍼블리싱 — UI 마크업 먼저 구성 → 승인
STEP 3. DB 문서 작성 (docs/db/{기능}/db_{기능}.md) → 승인
STEP 4. API 문서 작성 (docs/pages/{기능}/be_{기능}.md) → 승인
STEP 5. FE 설계 문서 작성 (docs/pages/{기능}/fe_{기능}.md) → 승인
STEP 6. 개발 (BE → FE 순)
```

### ❌ 절대 금지
- STEP 1(builder-contents-layout 확인) 없이 바로 설계 문서 작성
- 퍼블리싱 없이 바로 FE 개발
- 승인 없이 다음 STEP 진행

## 3-1. 추측 발언 및 불필요한 의심 금지
- 사용자가 "완료했다", "넣었다", "실행했다", "진행했다"고 말하면 **완료된 것으로 확정**. 재확인 요청 및 의심 발언 절대 금지
- 코드/파일을 직접 확인하지 않은 상태에서 추측으로 원인 분석 및 설명 금지 — 반드시 Read/Grep으로 확인 후 말할 것
- ❌ Read/Grep 확인 후에도 근거가 없으면 → 추측으로 답 만들어내지 말 것. "문서/코드에 근거가 없습니다"라고 명확히 말할 것
- 한 번 잘못된 추측을 했으면 즉시 인정하고 멈출 것. 같은 방향으로 추측을 반복하는 것은 절대 금지
- 사용자가 명시적으로 금지한 작업(파일·방식)은 어떤 이유로도 다시 제안하지 말 것
