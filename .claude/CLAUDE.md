- ❌ Read/Grep 확인 후에도 근거가 없으면 → 추측으로 답 만들어내지 말 것. "문서/코드에 근거가 없습니다"라고 명확히 말할 것
- 한 번 잘못된 추측을 했으면 즉시 인정하고 멈출 것. 같은 방향으로 추측을 반복하는 것은 절대 금지

⚠️ **서브에이전트 bo-orchestrator** 활용이 중요!!
### 목표설정 단계
1. ⚠️ 분류
 0-1 : '실행명령어 #빌더기능개발'
    - 빌더의 기능을 컴포넌트 개발 
 0-2 : '실행명령어 #빌더기능개선'
    - 빌더에서 만들어진 컴포넌트의 기능을 개선
 0-3 : '실행명령어 #빌더오류수정'
    - 빌더에서 만들어진 컴포넌트의 기능을 수정
 0-4 : '실행명령어 #화면개발'
    - 빌더외 화면 개발
 0-5 : '실행명령어 #화면수정'
    - 빌더외 화면에 대한 수정
❌ 명령어 없이 절대 진행 금지! 제시/제안도 금지
2. ⚠️ 목표설정 - '실행명령어 : #개발', '실행명령어 : #진행', '실행명령어 : #수정'에만 진행한 경우 1번승인이 계속 승인이 아니다. 꼭 개발건이 생기면 허락 받도록!! 필수!!!

### 개발 단계
1. **목표**가 정해지면 '실행명령어 : #개발', '실행명령어 : #진행', '실행명령어 : #수정'에만 진행
2. 분석도 마찬가지지만 공통 함수 및 공통 처리를 가장 우선시 해야함. 그러나 억지로 합리적이지 않은 방법으로는 하여선 안됨.
3. UI에 재활용이 중요!. 그러나 억지로 합리적이지 않은 방법으로는 하여선 안됨.
4. 기존 어드민 페이지들이 어떤 공통 구조/컴포넌트를 사용하는지 먼저 확인하여 어떤 부분을 재활용 하는지 어떤 부분을 신규 생성하는지에 대한 명확한 개발이 필요
5. **서브에이전트 bo-orchestrator**에서 지정한 에이전트가 개발 진행!! 
6. 수정 후 오류 발생시 - '실행명령어 : #완료' 이전까지 계속 수행
    ⚠️ 문제 분석 강제 규칙 - 어떤 문제든 원인 발표 전 전체 흐름 직접 확인 필수:
    - Explore 에이전트 와 **서브에이전트 bo-orchestrator** 오류 분석
    - 데이터 흐름 시작 → 끝까지 각 단계 코드 직접 Read
    - 한 단계라도 건너뛰면 원인 발표 금지
    - 에이전트 분석 결과 = 참고용. 핵심 파일은 반드시 직접 확인
    - 추측으로 원인 말하기 금지
⚠️ 개발/수정을 진행하더라도 '실행명령어 : #완료' 사용자가 입력하기 전까지 끝난게 아니다. 
⚠️ '실행명령어 : #완료' 이전까지 완료가 아니니 계속 ** 목표 제시해서 대기
⚠️ 꼭 현재 에이전트와 **서브에이전트 bo-orchestrator**의 내용이 합쳐져서 합리적인 제안을 제시!!!
⚠️ 에이전트 검증시 로그인이 안된 경우 로그인 요청!!
❌ 개발 진행 유도하는 문구 절대 금지!!!
### 완료 단계
1. '실행명령어 : #완료' 전까지 완료가 절대 아니니 계속 수정이 필요.
2. '실행명령어 : #완료' 이후에 빌더에 대한 작업인 경우 가이드 문서에 변경이 필요한지 확인
3. 가이드 문서 변경시 가이드에 잘못된 부분이 있다면 무조건 개선 필요
4. 가이드 문서 변경이 완료 되면 /compact 후 룰 재확인이 필요


## 0. ⚠️ 작업 시작 전 필수 순서 체크 (Rule 10 연결 — 매 작업마다 이 순서 확인)
> 아래 순서를 어기는 것은 절대 금지. 설명·승인 없이 다음 단계로 넘어갈 수 없다.
### 빌더 컴포넌트 개발시 
```
STEP 1. builder-contents-layout 확인 (Rule 10-2)
         → 해당 컴포넌트 탭이 없으면 → STEP 2로
STEP 2. 빌더 Preview 개발 진행 - 진행 전 어떤 방식 진행인지 설명 필요.
STEP 3. 퍼블리싱 — builder-contents-layout에 UI 마크업 먼저 구성 → 승인
STEP 4. 빌더 화면 개발 진행 - 빌더 내부 공통 UI 적용해서 진행 - 새로 만들기 금지
STEP 5. Live 기능 구현 - 구현 전에 자세한 설명 필요
```

### BE 포함 작업 (API·DB 변경 있음)
```
STEP 1. builder-contents-layout 확인 (Rule 10-2)
STEP 2. 퍼블리싱 — UI 마크업 먼저 구성 → 승인
STEP 3. DB 문서 작성 (docs/db/{기능}/db_{기능}.md) → 승인
STEP 4. API 문서 작성 (docs/pages/{기능}/be_{기능}.md) → 승인
STEP 5. FE 설계 문서 작성 (docs/pages/{기능}/fe_{기능}.md) → 승인
STEP 6. 개발 (BE → FE 순)
```
### ❌ 절대 금지
- STEP 1(builder-contents-layout 확인) 없이 바로 설계 문서 작성
- 퍼블리싱 없이 바로 FE 개발
- 승인 없이 다음 STEP 진행
