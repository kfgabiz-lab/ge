# 빌더 다국어(i18n) 적용 가이드

> 작성일: 2026-06-04  
> 대상: 빌더 위젯/렌더러를 신규 추가하거나 수정하는 모든 에이전트  
> 연관 문서: `docs/ge_guide/multi-launguage-guide_v1.0.md`, `docs/db/i18n/i18n-key-convention_v2.md`

---

## 0. ⭐ 핵심 원칙 — 반드시 숙지

빌더 다국어 처리는 **세 가지 패턴**으로 완전히 분리된다.

| 구분 | 패턴 | 대상 텍스트 | 처리 방법 |
|---|---|---|---|
| **패턴 A** | `labelMsgKey` 방식 | 빌더에서 관리자가 직접 입력/선택하는 텍스트 | `widget.*MsgKey ? t(widget.*MsgKey) : widget.*` |
| **패턴 B** | `t()` 방식 | 렌더러 코드에 고정된 텍스트 (버튼, 안내문구, toast 등) | `t('common.*')` |
| **패턴 C** | 공통코드 값 방식 | codeGroupCode로 연동된 값(코드→라벨) | `resolveCodeLabel()` / `codeDetailToLabel()` (§3) |

> **잘못된 접근**: 렌더러에 `"삭제"`, `"항목이 없습니다"` 같은 한국어 문자열을 하드코딩하는 것은 절대 금지.

## 1. 패턴 A — 빌더 입력값 다국어 처리 (`labelMsgKey` 방식)

### 1-1. 개념

관리자가 빌더에서 설정하는 텍스트(라벨, 제목, 버튼명 등)는  
**일반 텍스트 입력 필드** + **다국어 키 선택 필드** 두 가지를 동시에 저장한다.

```
관리자 빌더 설정:
  label      = "카테고리명"          ← 한국어 텍스트 (fallback용)
  labelMsgKey = "category.field.name" ← message_resource.key

렌더러 출력:
  labelMsgKey 있으면 → t('category.field.name') = 현재 언어 번역
  labelMsgKey 없으면 → label = "카테고리명" (한국어 고정)
```

### 1-2. 빌더 패널 UI — `BuilderI18nModeContext` + `MessageKeySelector`

빌더 좌측 패널에서 i18n 모드 토글 버튼(`Globe` 아이콘)이 ON일 때  
텍스트 입력 대신 `MessageKeySelector`를 표시한다.

#### BuilderI18nModeProvider 동작 방식

```typescript
// useBuilderI18nMode() 반환값 전체
interface BuilderI18nModeContextValue {
    i18nMode: boolean;          // true: 다국어 키 모드 / false: 직접 입력 모드
    toggleI18nMode: () => void; // 모드 토글 함수
}
```

**WidgetI18nToggle 컴포넌트**:
- `SizeSettingPanel` 헤더에 배치 (`e.stopPropagation()`으로 헤더 클릭과 분리)
- 파란색 배경 = 다국어 키 모드 ON / 투명 = 직접 입력 모드 OFF

**i18nMode 복원 로직**:
- 위젯 저장 시 `widgetItem.i18nMode`에 현재 모드 저장
- 불러오기 시 기존 `*MsgKey` 값 감지 → 자동으로 모드 복원
- 탭 전환 / 다른 위젯 선택 시 모드 초기화되지 않음

```tsx
/* 빌더 컴포넌트 내부 패턴 */
import { MessageKeySelector } from '@/components/i18n/message-key-selector';
import { useBuilderI18nMode } from '../../contexts/BuilderI18nModeContext';

export function MyBuilder({ widget, onChange }: Props) {
  const { i18nMode } = useBuilderI18nMode();

  return (
    <div>
      <label className={LABEL_CLS}>라벨</label>
      {i18nMode ? (
        /* i18n 모드: 다국어 키 선택 */
        <MessageKeySelector
          value={widget.labelMsgKey ?? ''}
          onChange={key => onChange({ ...widget, labelMsgKey: key || undefined })}
          resourceType="WORD"   // 단어/라벨 → WORD, 문장/설명 → SENTENCE
          size="sm"
        />
      ) : (
        /* 일반 모드: 텍스트 직접 입력 */
        <input
          type="text"
          value={widget.label ?? ''}
          onChange={e => onChange({ ...widget, label: e.target.value })}
          className={INPUT_CLS}
        />
      )}
    </div>
  );
}
```

> **`resourceType` 구분 규칙**:
> - `"WORD"` : 버튼명, 컬럼 헤더, 탭 레이블 등 짧은 라벨
> - `"SENTENCE"` : 안내 문구, 설명 등 문장형 텍스트

#### MessageKeySelector 상세

```typescript
// Props 명세
interface MessageKeySelectorProps {
    value: string;
    onChange?: (key: string) => void;
    disabled?: boolean;               // 비활성화 여부
    resourceType?: 'WORD' | 'SENTENCE'; // 필터 (미지정 시 전체 조회)
    size?: 'sm' | 'md';
}
```

**동작 특성**:
- API: `GET /message-resources?active=true&resourceType=...&size=9999`
- **Portal 렌더링** — `body`에 직접 마운트하여 overflow 컨테이너 회피
- 검색 시 `key` / `ko` / `en` 세 필드 모두 검색 가능
- X 버튼으로 선택값 제거 → `onChange('')` 호출 → `|| undefined` 처리로 `undefined` 저장

### 1-3. types.ts 타입 선언 — `*MsgKey?` 필드 필수 추가

위젯/컬럼 타입에 반드시 `*MsgKey?: string` 선택 필드를 추가한다.

```typescript
/* renderer/types.ts 예시 패턴 */
export interface MyWidget {
  type: 'mywidget';
  widgetId: string;
  label?: string;
  labelMsgKey?: string;              // ← 라벨 다국어 키
  label2?: string;                   // ← dateRange 두 번째 라벨
  label2MsgKey?: string;             // ← dateRange 두 번째 라벨 다국어 키
  title?: string;
  titleMsgKey?: string;              // ← 타이틀 다국어 키
  description?: string;
  descriptionMsgKey?: string;        // ← 설명 다국어 키
  placeholder?: string;
  placeholderMsgKey?: string;        // ← placeholder 다국어 키
  defaultValue?: string;
  defaultValueMsgKey?: string;       // ← 기본값 다국어 키 (input/select/textarea)
}
```

### 1-4. 빌더 → 렌더러 변환 함수 (`toFieldValues` / `fromFieldValues`)

`SubListBuilder.tsx` 등 변환 함수를 사용하는 빌더에서는  
`*MsgKey` 필드를 양방향으로 매핑해야 한다.

```typescript
/* toFieldValues: 위젯 데이터 → 빌더 입력 필드값 변환 */
function toFieldValues(col: SubListColumn) {
  return {
    label: col.label ?? '',
    labelMsgKey: col.labelMsgKey ?? '',       // ← 반드시 추가
    placeholder: col.placeholder ?? '',
    placeholderMsgKey: col.placeholderMsgKey ?? '',  // ← 반드시 추가
    description: col.description ?? '',
    descriptionMsgKey: col.descriptionMsgKey ?? '',  // ← 반드시 추가
    // ...
  };
}

/* fromFieldValues: 빌더 입력 필드값 → 위젯 데이터 변환 */
function fromFieldValues(vals: ReturnType<typeof toFieldValues>): Partial<SubListColumn> {
  return {
    label: vals.label,
    labelMsgKey: vals.labelMsgKey || undefined,       // 빈 문자열 → undefined
    placeholder: vals.placeholder || undefined,
    placeholderMsgKey: vals.placeholderMsgKey || undefined,
    description: vals.description || undefined,
    descriptionMsgKey: vals.descriptionMsgKey || undefined,
    // ...
  };
}
```

### 1-5. FieldBase 컴포넌트 — 필드 라벨/설명 MsgKey 전달

필드 컴포넌트(`InputField`, `SelectField`, `FileField` 등)는 모두  
`FieldBase`를 통해 라벨과 설명을 렌더링한다.  
**반드시 `labelMsgKey`와 `descriptionMsgKey`를 함께 전달해야 한다.**

```tsx
/* FieldBase 전달 패턴 — 모든 필드 컴포넌트 동일 */
export const MyField = (props: FieldEditProps) => {
  const { values, onChange } = props;
  return (
    <FieldBase
      {...props}
      label={values.label}
      labelMsgKey={values.labelMsgKey}           // ← 반드시 전달
      description={values.description}
      descriptionMsgKey={values.descriptionMsgKey}  // ← 반드시 전달
      // placeholder 있는 경우
      // placeholder={values.placeholder}
      // placeholderMsgKey={values.placeholderMsgKey}
    >
      {/* 필드 본문 */}
    </FieldBase>
  );
};
```

> **누락 시 증상**: 빌더에서 다국어 키를 설정해도 렌더러에서 번역이 적용되지 않고 한국어 텍스트만 표시된다.

### 1-6. 렌더러에서 MsgKey 분기 패턴

```tsx
/* 렌더러 내 MsgKey 분기 — 반드시 이 패턴 사용 */
const { t } = useI18n();

// 라벨 표시
{widget.labelMsgKey ? t(widget.labelMsgKey) : (widget.label || '기본값')}

// 접혔을 때 헤더 라벨 (SubListBuilder SortableColumnItem 예시)
{col.labelMsgKey ? t(col.labelMsgKey) : (col.label || <span className="text-slate-300 italic">라벨 없음</span>)}
```

## 2. 패턴 B — 렌더러 고정 텍스트 처리 (`t()` 방식)

### 2-1. 기본 사용법

렌더러 파일에서 UI 고정 텍스트(버튼, 안내문구, toast, confirm 등)는  
**반드시 `useI18n()` 훅의 `t()` 함수를 경유**해야 한다.

```tsx
import { useI18n } from '@/hooks/use-i18n';

export function MyRenderer({ widget, mode }: Props) {
  const { t } = useI18n();

  return (
    <div>
      {/* ✅ 올바른 방법 */}
      <button>{t('common.btn.add')}</button>
      <p>{t('common.table.no_data')}</p>

      {/* ❌ 절대 금지 — 하드코딩 */}
      <button>추가</button>
      <p>데이터가 없습니다.</p>
    </div>
  );
}
```

### 2-2. 변수 치환 (카운트 등)

```tsx
// DB 키: "common.multiselect.selected_count"
// ko: "{count}개 선택됨"
// en: "{count} items selected"

{t('common.multiselect.selected_count', { count: selected.length })}
```

### 2-3. toast 메시지 다국어 처리

```tsx
import { toast } from 'sonner';
import { useI18n } from '@/hooks/use-i18n';

const { t } = useI18n();

// ✅ 올바른 방법
toast.success(t('common.saved'));
toast.error(t('common.error.unknown'));
toast.warning(t('common.validation.required'));

// confirm 다이얼로그
if (!confirm(t('common.confirm.delete'))) return;
```

## 3. 패턴 C — 공통코드 값 표시 다국어 처리

`codeGroupCode`로 연동된 값(공통코드)은 패턴 A/B와 다르다 — 빌더가 만든 텍스트도, 렌더러 고정 텍스트도 아니라 **DB 공통코드 테이블(`CodeDetail`)에서 조회한 값**이기 때문이다.

```typescript
// utils.ts — 공통코드 라벨 변환 공통함수 (반드시 이 함수 경유)
codeDetailToLabel(detail, t)   // 단건: nameMsgKey 있으면 t(nameMsgKey), 없으면 name
resolveCodeLabel(strVal, codeGroupCode, displayAs, codeGroups, t, requireActive)
                                // 콤마 구분 다중값 지원 (내부적으로 codeDetailToLabel 반복 호출)
```

**적용 위치 3곳 — 반드시 동일 함수 공유** (하나라도 다른 방식을 쓰면 화면·엑셀 라벨이 서로 어긋난다):
- `TableCellRenderer.tsx` — 테이블 셀 표시 (`requireActive` 생략, 기존부터 비활성 코드도 표시)
- `FieldRenderer.tsx` — 폼 읽기전용 표시 (`requireActive=true`, 비활성 코드는 매칭 제외 — 두 렌더러의 `requireActive` 차이는 기존부터 있던 동작이니 통일하지 말 것)
- `WidgetRenderer.tsx` `doExcelDownload()` — 엑셀 export용 `codeMaps` 딕셔너리 생성 (§`02.builder_data_process.md` 9-3 정책 6 참고)

> **누락 시 증상**: 테이블/엑셀에 사람이 읽는 라벨 대신 코드값(`COMM00101` 등)이 그대로 노출된다.

## 5. 절대 금지 패턴 ❌

### 금지 1: 렌더러에 한국어 텍스트 하드코딩

```tsx
// ❌ 금지
<p>데이터가 없습니다.</p>
<button>추가</button>
toast.success('등록되었습니다.');
confirm('삭제하시겠습니까?');

// ✅ 올바른 방법
<p>{t('common.table.no_data')}</p>
<button>{t('common.btn.add')}</button>
toast.success(t('common.saved'));
confirm(t('common.confirm.delete'));
```

### 금지 2: resourceType 오류 (WORD ↔ SENTENCE 혼용)

```tsx
// ❌ 금지 — 제목/라벨에 SENTENCE 사용
<MessageKeySelector resourceType="SENTENCE" />  // 버튼명/라벨에는 WORD 사용

// ✅ 올바른 방법
<MessageKeySelector resourceType="WORD" />       // 단어/라벨
<MessageKeySelector resourceType="SENTENCE" />   // 문장/설명
```

### 금지 3: MsgKey 없이 위젯 타입 추가

```typescript
// ❌ 금지 — MsgKey 없이 타입만 추가
export interface NewWidget {
  type: 'new';
  label: string;     // MsgKey 없음 → 다국어 적용 불가
  title: string;
}

// ✅ 올바른 방법 — 모든 사용자 노출 텍스트에 MsgKey 추가
export interface NewWidget {
  type: 'new';
  label?: string;
  labelMsgKey?: string;
  title?: string;
  titleMsgKey?: string;
}
```

### 금지 4: `t()` 없이 `useI18n` import만 하고 미사용

```tsx
// ❌ 금지 — import만 하고 t() 미사용
import { useI18n } from '@/hooks/use-i18n';
// ...
<p>항목이 없습니다</p>  // t() 누락

// ✅ 올바른 방법
import { useI18n } from '@/hooks/use-i18n';
const { t } = useI18n();
<p>{t('common.table.no_data')}</p>
```

### 금지 5: FieldBase에 MsgKey 전달 누락

```tsx
// ❌ 금지 — label만 전달, labelMsgKey 누락
<FieldBase
  label={values.label}
  // labelMsgKey 없음 → 다국어 키 설정해도 미적용
/>

// ✅ 올바른 방법
<FieldBase
  label={values.label}
  labelMsgKey={values.labelMsgKey}         // 필수
  description={values.description}
  descriptionMsgKey={values.descriptionMsgKey}  // 필수
/>
```
