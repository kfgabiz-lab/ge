## 0. ⭐ 핵심 개념 — 반드시 숙지

### 0-1. Slug Entity란?

빌더에서 Form 위젯을 구성할 때 **반복적으로 사용되는 데이터 필드 집합을 미리 정의**해두는 메타 정보다.

예를 들어 "회원" entity를 등록해두면, 빌더에서 "회원" entity를 선택하고 빌드 버튼을 누르는 것만으로 회원 관련 Form 필드(이름, 이메일, 연락처 등)가 자동 생성된다.

```
[관리 화면] Slug Entity 정의
   └── entity: member (회원)
       ├── field: name      (label: 이름,     columnType: VARCHAR,   fieldType: input)
       ├── field: email     (label: 이메일,   columnType: VARCHAR,   fieldType: input)
       ├── field: gender    (label: 성별,     columnType: VARCHAR,   fieldType: select,   codeGroupCode: GENDER)
       └── field: join_date (label: 가입일,   columnType: DATE,      fieldType: date)

[빌더] entity 선택 → 빌드 버튼 클릭
   └── Form 위젯에 필드 4개 자동 생성
```

---

### 0-2. 필드 속성 구조

`SlugEntityFieldItem` 타입 (정의 위치: `EntityList.tsx`)

| 속성 | 타입 | 필수 | 설명 |
|---|---|---|---|
| `key` | `string \| null` | 권장 | 빌더 fieldKey 매핑 기준 (snake_case, 예: `member_id`) |
| `label` | `string` | ✅ | 화면 표시명 (예: 회원 ID) |
| `columnType` | `string` | ✅ | DB 타입 (`VARCHAR`, `BIGINT`, `INT`, `BOOLEAN`, `TIMESTAMPTZ`, `DATE`, `JSONB`, `TEXT`) |
| `fieldType` | `string \| null` | - | 빌더 필드 타입 직접 지정 (1순위 매핑) |
| `codeGroupCode` | `string \| null` | - | 공통코드 그룹 코드 — select/radio/checkbox 옵션 자동 연결 |
| `isNullable` | `boolean` | ✅ | NULL 허용 여부 → `false`면 Form 필드 `required: true` 자동 설정 |
| `description` | `string \| null` | - | 필드 설명 |

## 1. Entity 필드 타입 매핑 규칙

빌더에서 Entity 빌드 시 **빌더 필드 타입(type)** 결정 우선순위는 다음과 같다.

```
1순위: entity.fieldType 직접 지정값 사용
         ↓ (없을 경우)
2순위: entity.columnType(DB 타입) → 자동 매핑
```

### 1-1. DB 타입 자동 매핑표 (2순위 fallback)

| DB 타입 (`columnType`) | 빌더 필드 타입 (`type`) |
|---|---|
| `VARCHAR` | `input` |
| `BIGINT`, `INT` | `input` |
| `BOOLEAN` | `checkbox` |
| `DATE`, `TIMESTAMPTZ` | `date` |
| `TEXT`, `JSONB`, 기타 | `textarea` |

> ⚠️ `fieldType`이 entity 관리 화면에서 미리 설정되어 있으면 위 자동 매핑은 **무시**된다.

### 1-2. Entity 관리 화면 자동 제안

Entity 관리 화면(`/admin/settings/slug-entity`)에서 DB 타입 드롭다운을 변경하면 `fieldType`이 자동으로 제안된다 (`EntityFieldEditor.tsx` — `mapDbTypeToFieldType`).  
사용자가 이후 수동으로 변경 가능하므로 제안값일 뿐이다.

---

## 2. colSpan 자동 결정 규칙

빌드 시 필드 타입에 따라 `colSpan`이 자동으로 결정된다.

| 빌더 필드 타입 | colSpan |
|---|---|
| `textarea` | **2** (넓은 입력 필요) |
| `dateRange` | **2** (시작일·종료일 범위) |
| `yearMonthRange` | **2** (시작월·종료월 범위) |
| 그 외 모든 타입 | **1** |

---

## 3. 공통코드 자동 연결 규칙

Entity 필드에 `codeGroupCode`가 설정되어 있으면, 빌드 시 `FormFieldItem`에 `codeGroupCode`가 자동으로 연결된다.

```typescript
// 빌드 결과 예시: gender 필드에 GENDER 공통코드 자동 연결
{
    type: 'select',
    fieldKey: 'gender',
    label: '성별',
    codeGroupCode: 'GENDER',  // ← entity.codeGroupCode에서 자동 주입
    colSpan: 1,
    required: false,
}
```

> `codeGroupCode`가 있으면 select/radio/checkbox 타입에서 옵션 목록을 공통코드 API(`/codes`)에서 자동으로 불러온다.

---

## 4. Key 매칭 규칙 (`getKeyVariants`)

Entity 필드의 `key`는 snake_case로 저장되지만 빌더 내부에서는 다양한 형태로 참조된다.  
빌더는 다음 4가지 변형을 모두 매핑에 등록해 **어떤 형태로 fieldKey를 설정해도 entity 필드를 정확히 찾는다.**

```
원본 key:           disp_at_from
camelCase:          dispAtFrom
suffix 제거:        disp_at       ← _from / _to suffix 제거
suffix 제거 + camelCase: dispAt
```

> `dateRange` 타입 필드는 FE에서 `fieldKey` + `_from` / `_to` 형태로 실제 값을 분리 저장한다.  
> Entity에는 `disp_at_from`, `disp_at_to`로 등록해두어도 빌더는 `disp_at` 하나로 인식한다.

## 5. 빌드 흐름 전체 (`handleBuildFromEntity`)

```
빌드 버튼 클릭
    │
    ▼
slugEntityFields 로드 확인
    │
    ▼
entityFieldMap 생성 (key 4가지 변형 모두 등록)
    │
    ├── [Form 위젯 이미 있는 경우]
    │       ├── 기존 필드: label / required 업데이트만
    │       └── 신규 필드: 기존에 없는 entity 필드만 하단 추가
    │
    └── [Form 위젯 없는 경우]
            └── Form 위젯 + 컨텐츠 + 필드 전체 신규 생성
```

## 6. 코드 위치 — `buildFieldItem` 헬퍼

> 파일: `bo/src/app/admin/templates/make/widget/page.tsx`

```typescript
const buildFieldItem = (f: SlugEntityFieldItem): FormFieldItem => {
    // 1순위: entity fieldType 직접 사용 / 2순위: DB 타입 자동 매핑
    const type = (f.fieldType as SearchFieldType | undefined) ?? mapColumnTypeToFieldType(f.columnType);
    const wideTypes = ['textarea', 'dateRange', 'yearMonthRange'];
    return {
        id: uid(),
        type,
        label: f.label,
        fieldKey: f.key!,
        colSpan: wideTypes.includes(type) ? 2 : 1,
        rowSpan: 1,
        required: f.isNullable === false,
        ...(type === 'date' && { dateSubType: 'date' as const }),
        ...(f.codeGroupCode ? { codeGroupCode: f.codeGroupCode } : {}),
    } as FormFieldItem;
};
```

## 7. Entity 관리 화면 사용 방법

**경로:** `/admin/settings/slug-entity`

### 7-1. Entity 등록

1. 좌측 패널 [등록] 버튼 클릭
2. slug(영문/숫자/하이픈/언더스코어), 표시명, 설명 입력
3. 등록 완료 후 자동 선택

### 7-2. 필드 편집

| 컬럼 | 설명 |
|---|---|
| `key` | 빌더 fieldKey 기준값 (snake_case 권장) |
| `label` | Form 필드 표시명 |
| `DB 타입` | 실제 DB 컬럼 타입 선택 — 변경 시 빌더필드타입 자동 제안 |
| `빌더필드타입` | 빌더에서 실제 사용할 필드 타입 (1순위) |
| `공통코드` | select/radio/checkbox 타입일 때 연결할 공통코드 그룹 |
| `길이` | VARCHAR 등 길이 지정 타입에만 입력 |
| `NULL` | 체크 해제 시 빌드된 Form 필드에 `required: true` 자동 설정 |

### 7-3. 빌더에서 사용

1. Widget 빌더에서 좌측 패널 → [Entity] 탭 선택
2. 연동할 Slug Entity 선택
3. [Entity로 빌드] 버튼 클릭
4. Entity 필드 기반으로 Form 위젯 자동 생성

---

## 8. 절대 금지 패턴

- `buildFieldItem` 내에서 `fieldType`을 무시하고 `mapColumnTypeToFieldType`만 사용 — **1순위 fieldType 우선 원칙 위반**
- Entity 필드에 `key` 없이 빌드 시도 — key 없는 필드는 빌드에서 제외됨 (`f.key`가 null이면 skip)
- `codeGroupCode`가 있는데 빌더 필드 타입이 `select/radio/checkbox`가 아닌 경우 — 의미 없는 연결. entity 관리에서 타입 확인 필요
- `getKeyVariants` 없이 entity 필드 매핑 — snake_case/camelCase/dateRange suffix 불일치로 매핑 누락 발생
