## 0. ⭐ 핵심 개념 — 반드시 숙지

### 0-1. Slug Entity란?

빌더에서 Form 위젯을 구성할 때 **반복적으로 사용되는 데이터 필드 집합을 미리 정의**해두는 메타 정보다.

예를 들어 "회원" entity를 등록해두면, 빌더에서 "회원" entity를 선택하고 빌드 버튼을 누르는 것만으로 회원 관련 Form 필드(이름, 이메일, 연락처 등)가 자동 생성된다.

```
[관리 화면] Slug Entity 정의
   └── entity: member (회원)
       ├── field: name      (label: 이름,     columnType: VARCHAR,   fieldType: input)
       ├── field: email     (label: 이메일,   columnType: VARCHAR,   fieldType: input)
       ├── field: gender    (label: 성별,     columnType: VARCHAR,   fieldType: select,   codeGroupCode: GENDER)
       └── field: join_date (label: 가입일,   columnType: DATE,      fieldType: date)

[빌더] entity 선택 → 빌드 버튼 클릭
   └── Form 위젯에 필드 4개 자동 생성
```

### 0-1-1. 저장소는 그대로다 — Slug Entity ≠ Data Entity

빌더 패널 "타입" selectbox에는 `Slug` / `Slug Entity` / `Data Entity` 세 가지가 있다. 이름이 비슷해서 혼동하기 쉽지만 저장소 관점에서는 완전히 다르다.

| 타입 | 저장/조회 대상 | 핵심 목적 |
|---|---|---|
| **Slug** | 기존 page_data(SlugRegistry) | 그대로 |
| **Slug Entity** | 기존 page_data(SlugRegistry) — **동일** | entity는 저장소가 아니라, 빌더에서 필드 key를 임의로 오타·제각각 입력하지 않도록 selectbox로 제안해주는 **편의 기능**일 뿐 |
| **Data Entity** | SlugEntity 코드생성이 만든 **자체 REST API**(`/api/v1/hero-data` 등, Controller/Service/Repository/Entity 파일) | page_data가 아니라 entity 자신의 API를 빌더와 실제로 연동하는 개념 |

즉 이 문서(04번)가 다루는 "Slug Entity"는 **page_data 저장 흐름을 바꾸지 않는다** — `slug_entity`/`slug_entity_field`는 필드 정의 메타 정보일 뿐이고, 실제 데이터는 여전히 `page_data`(SlugRegistry로 연결된 slug)에 저장·조회된다.

entity 자신이 생성한 REST API(예: `/api/v1/hero-data`)를 빌더가 직접 호출해서 쓰는 "Data Entity" 개념은 이 문서의 범위가 아니다 — [06. Entity 기반 API 자동생성 → 빌더 연동 통합 가이드](./06.builder_entity_to_api_guide.md)를 참고한다.

### 0-2. 필드 속성 구조

`SlugEntityFieldItem` 타입 (정의 위치: `EntityList.tsx`)

| 속성 | 타입 | 필수 | 설명 |
|---|---|---|---|
| `id` | `number` | - | 필드 PK (신규 등록 시 미지정) |
| `key` | `string \| null` | 권장 | 빌더 fieldKey 매핑 기준 (snake_case, 예: `member_id`) |
| `label` | `string` | ✅ | 화면 표시명 (예: 회원 ID) |
| `columnType` | `string` | ✅ | DB 타입 (`VARCHAR`, `BIGINT`, `INT`, `BOOLEAN`, `TIMESTAMPTZ`, `DATE`, `JSONB`, `TEXT`, `FILE`, `ENTITY_REF`) |
| `columnLength` | `number \| null` | - | 컬럼 길이 (`VARCHAR` 등 길이 지정 타입에만 사용) |
| `connectedEntityId` | `number \| null` | - | `columnType=ENTITY_REF`일 때 연동 대상 Slug Entity id (§0-5). 다른 타입에서는 사용하지 않음 |
| `fieldType` | `string \| null` | - | 빌더 필드 타입 직접 지정 (1순위 매핑) |
| `codeGroupCode` | `string \| null` | - | 공통코드 그룹 코드 — select/radio/checkbox 옵션 자동 연결 |
| `defaultValue` | `string \| null` | - | 필드 기본값 — Slug Entity 관리화면에서 입력·저장·조회만 지원. **`buildFieldItem`(§6)은 이 값을 참조하지 않으므로 [Entity로 빌드] 시 Form 필드에 자동 반영되지 않는다** (§8 절대 금지 패턴 참고) |
| `isNullable` | `boolean` | ✅ | NULL 허용 여부 → `false`면 Form 필드 `required: true` 자동 설정 |
| `description` | `string \| null` | - | 필드 설명 |
| `sortOrder` | `number` | ✅ | 필드 표시 순서 |

### 0-3. 마스터(부모) Entity 연결

Entity 하나가 다른 Entity를 "마스터(부모)"로 참조하는 구조를 정의할 수 있다. Form+SubList처럼 마스터-자식 관계를 가진 화면을 모델링할 때 사용한다.

```
[Entity] member (회원) — 마스터, 독립 Entity
[Entity] member-address (회원 주소) — 마스터 Entity로 member 선택
   └── 등록/저장 시 필드 목록에 member_id(BIGINT, 필수)가 자동 추가됨
```

- Entity 등록 모달 또는 EntityFieldEditor 헤더의 "마스터 Entity" select에서 지정
- 지정하면 자식 Entity의 필드 목록에 `{마스터slug의 하이픈→언더스코어}_id`(BIGINT, 필수) 필드가 **자동으로 추가**된다 (이미 동일 key 필드가 있으면 중복 추가하지 않음)
- 자동 추가된 필드도 이후 일반 필드와 동일하게 직접 수정·삭제 가능
- 자기 자신을 마스터로 지정할 수 없음(select 옵션에서 제외 + API에서도 차단)
- 자식 Entity가 하나라도 있으면 마스터 Entity는 삭제할 수 없음(409 에러)

> ⚠️ **참조무결성/CASCADE는 보장되지 않는다.** `slug_entity.parent_entity_id`(메타데이터 테이블 자기참조)에는 실제 DB FK가 걸려있어 자식이 있으면 부모 삭제가 막히지만, 자동 추가되는 `{마스터slug}_id` 필드는 **업무 테이블의 평범한 BIGINT 컬럼**일 뿐이다. codegen이 이 필드를 `@ManyToOne`/FK 연관관계로 만들어주지 않는다 — 상세는 §10-5 참고.

> ⚠️ **Entity 삭제 사전가드는 두 종류가 나란히 존재한다.** 위 마스터(부모) 참조 외에, 다른 Entity의 필드가 `columnType=ENTITY_REF`로 이 Entity를 연동 대상(`connected_entity_id`, §0-5)으로 지정하고 있으면 그 Entity도 삭제할 수 없다. 두 가드 모두 삭제 시점에 참조 존재 여부를 먼저 확인하는 동일한 사전가드 패턴이다.
>
> | 차단 상황 | 에러코드 | HTTP |
> |---|---|---|
> | 자식 Entity가 마스터로 참조 중(§0-3) | `SLUG_ENTITY_HAS_CHILDREN` | 409 |
> | 다른 필드가 ENTITY_REF로 연동 참조 중(§0-5) | `SLUG_ENTITY_REFERENCED` | 409 |

### 0-4. 첨부파일(FILE) 타입 필드

Entity 필드의 `columnType`을 `FILE`로 지정하면, 생성되는 Entity/DTO에 그 필드가 **`List<Long>`(DDL `BIGINT[]`, 배열) 컬럼**으로 만들어진다. 배열의 각 값은 파일의 실제 내용(바이트)이 아니라 **별도 파일 메타정보 테이블 `file_meta`의 `id`(참조값)**다.

```
[Entity] banner (배너)
   └── field: image_file_id (label: 이미지, columnType: FILE, fieldType: file)
        → 생성 코드: private List<Long> imageFileId;
          (@JdbcTypeCode(SqlTypes.ARRAY), banner.image_file_id 컬럼 = bigint[])
```

**왜 단일 값이 아니라 배열인가**: 빌더 Form 위젯의 `file`/`image`/`video`/`media` 타입 필드는 첨부 파일 개수와 무관하게 **항상 배열로 값을 저장·전송**한다(`existingFileMeta`/`fileList`가 전부 배열 구조 — `maxFileCount=1`이어도 길이 1짜리 배열). FILE 컬럼을 단일 `Long`으로 만들면 Form이 보내는 배열 값을 Jackson이 역직렬화하지 못해 저장 자체가 실패하므로, 이 계약에 맞춰 배열로 생성한다.

파일의 실제 업로드/조회/다운로드/삭제는 아래 범용 API로 수행한다 (`file_meta` 테이블 스펙은 §9-3 참고). **FILE 필드 전용 업로드 엔드포인트는 생성되지 않는다** — Controller는 다른 타입과 동일하게 일반 CRUD 5종만 생성되고, 파일 자체는 entity와 독립적인 `file_meta` 공용 API가 전담한다.

| 메서드 | 경로 | 설명 |
|---|---|---|
| `POST` | `/api/v1/file-meta/upload` | multipart 파일 업로드 → `file_meta` insert → `FileMetaResponse`(id 포함) 반환 |
| `GET` | `/api/v1/file-meta/{id}` | 파일 메타정보 단건 조회 |
| `GET` | `/api/v1/file-meta?ids=1,2,3` | 파일 메타정보 일괄 조회(존재하지 않는 id는 결과에서 자동 제외, 에러 아님) |
| `GET` | `/api/v1/file-meta/{id}/download` | 파일 실제 다운로드/blob 스트리밍 |
| `DELETE` | `/api/v1/file-meta/{id}` | 파일 삭제(DB 레코드 + 물리 파일. local 저장소만, Azure Blob 실삭제는 미지원 — page_file과 동일한 기존 갭) |

업로드로 받은 `id`들을 배열로 모아 해당 Entity의 일반 Create/Update 요청 바디(예: `imageFileId: [12, 15]`)에 그대로 실으면 저장된다 — 필드별 전용 연결 절차는 없다. Update 시에는 항상 전체 배열을 덮어쓴다(기존 값에 append하지 않음).

**실제 화면 연동 — 검증 완료(로드맵 3단계).** entity 연결(Data Entity) Form 위젯의 파일 첨부 UI가 위 5개 API로 업로드·수정모드 복원·다운로드·삭제까지 전부 실측 검증됐다(banner-data 기준). 연동 로직은 `bo/src/app/admin/templates/make/_shared/hooks/useWidgetPageState.ts`가 `connectedType`(entity/data 여부)에 따라 `file_meta` 계열 API와 기존 `page_file` 계열 API를 분기 호출하는 방식이며, 분기 판단은 `entityApi.ts`의 유틸을 거치지 않고 훅 내부에서 직접 이뤄진다.

### 0-5. 연동 Entity(ENTITY_REF) 타입 필드

Entity 필드의 `columnType`을 `ENTITY_REF`로 지정하면, 그 필드는 **다른 Slug Entity의 레코드들을 가리키는 참조 필드**가 된다. 즉 "이 필드에는 어떤 Entity의 어떤 레코드들이 연결되어 있는가"를 저장한다.

> ⚠️ **§0-3의 마스터(부모) Entity와는 완전히 별개 개념이다.** 아래처럼 참조 레벨·카디널리티가 다르다.
>
> | 구분 | 마스터(부모) Entity (§0-3) | 연동 Entity / ENTITY_REF (§0-5) |
> |---|---|---|
> | 참조 레벨 | **Entity 전체 레벨** — Entity 하나가 다른 Entity 하나를 부모로 가짐 | **필드 레벨** — Entity 안의 개별 필드가 참조를 가짐 |
> | 카디널리티 | **단일 참조** — 부모는 하나(`{마스터slug}_id` 단일 BIGINT) | **다중 참조** — 연결 레코드 여러 개(`List<Long>`/`BIGINT[]`) |
> | 저장 위치 | 자식 Entity 필드 목록에 `{마스터slug}_id` 필드 자동 추가 | 해당 필드 값 자체가 참조 id 배열 |

**매핑 방식** — FILE 타입(§0-4)과 동일하게 `List<Long>`(DDL `BIGINT[]`, 배열)으로 매핑한다. FILE의 배열 매핑 방식을 그대로 재사용한다.

```
[Entity] product (상품)
   └── field: related_option_ids (label: 연동 옵션, columnType: ENTITY_REF,
              connected_entity_id → option Entity)
        → 생성 코드: private List<Long> relatedOptionIds;
          (@JdbcTypeCode(SqlTypes.ARRAY), product.related_option_ids 컬럼 = bigint[])
```

**두 가지 정보가 나뉘어 저장된다** — 무엇을 가리키는지(값)와, 어느 Entity를 대상으로 삼는지(메타)가 분리된다.

| 저장 대상 | 저장 위치 | 내용 |
|---|---|---|
| 필드 값 자체 (`BIGINT[]`) | 생성 Entity의 업무 테이블 컬럼 | **연동된 레코드들의 id 배열** (예: `[12, 15]`) |
| 연동 대상 Entity | `slug_entity_field.connected_entity_id` (FK → `slug_entity.id`, §9-2) | 이 필드가 **어느 Slug Entity**를 가리키는지 지정하는 별도 메타 컬럼 |

즉 값(`BIGINT[]`)은 "몇 번 레코드들과 연결되었나"만 담고, "그 레코드들이 속한 Entity가 무엇인가"는 필드 정의 메타(`connected_entity_id`)가 별도로 관리한다.

**관리 화면 설정 방식** — `EntityFieldEditor.tsx`에서 필드의 `columnType`을 `ENTITY_REF`로 선택하면 "연동 Entity" autocomplete selectbox가 나타난다. 이 selectbox에서 대상 Entity를 검색·선택하면 그 Entity의 id가 `connectedEntityId`(→ `connected_entity_id`)에 저장된다.

> ⚠️ **ENTITY_REF로 참조되는 Entity는 삭제할 수 없다.** 어떤 필드가 `connected_entity_id`로 이 Entity를 가리키고 있으면 삭제 시 `SLUG_ENTITY_REFERENCED`(409)로 차단된다. §0-3의 마스터 참조 차단(`SLUG_ENTITY_HAS_CHILDREN`, 409)과 동일한 사전가드 패턴이다(§0-3 표 참고).

## 1. Entity 필드 타입 매핑 규칙

빌더에서 Entity 빌드 시 **빌더 필드 타입(type)** 결정 우선순위는 다음과 같다.

```
1순위: entity.fieldType 직접 지정값 사용
         ↓ (없을 경우)
2순위: entity.columnType(DB 타입) → 자동 매핑
```

### 1-1. DB 타입 자동 매핑표 (2순위 fallback)

| DB 타입 (`columnType`) | 빌더 필드 타입 (`type`) |
|---|---|
| `VARCHAR` | `input` |
| `BIGINT`, `INT` | `input` |
| `BOOLEAN` | `checkbox` |
| `DATE`, `TIMESTAMPTZ` | `date` |
| `FILE` | `file` |
| `TEXT`, `JSONB`, 기타 | `textarea` |

> ⚠️ `fieldType`이 entity 관리 화면에서 미리 설정되어 있으면 위 자동 매핑은 **무시**된다.

### 1-2. Entity 관리 화면 자동 제안

Entity 관리 화면(`/admin/settings/slug-entity`)에서 DB 타입 드롭다운을 변경하면 `fieldType`이 자동으로 제안된다 (`EntityFieldEditor.tsx` — `mapDbTypeToFieldType`).  
사용자가 이후 수동으로 변경 가능하므로 제안값일 뿐이다.

> ⚠️ **`ENTITY_REF`는 의도적으로 미제안 처리된다.** `mapDbTypeToFieldType('ENTITY_REF')`는 빈 문자열을 반환해 "빌더필드타입"을 자동 채우지 않는다. 대신 DB 타입을 `ENTITY_REF`로 선택하면 "빌더필드타입" 셀 자체가 일반 selectbox가 아니라 **"연동 Entity" autocomplete selectbox**로 전환되어(§0-5), 여기서 대상 Entity를 선택하게 된다. 다중 엔티티 링크(연동 레코드 여러 개)에 정확히 대응하는 빌더 위젯 필드타입은 아직 정의되어 있지 않으므로, 이 부분은 **확인 필요/향후 결정 필요** 항목이다.

## 2. colSpan 자동 결정 규칙

빌드 시 필드 타입에 따라 `colSpan`이 자동으로 결정된다.

| 빌더 필드 타입 | colSpan |
|---|---|
| `textarea` | **2** (넓은 입력 필요) |
| `dateRange` | **2** (시작일·종료일 범위) |
| `yearMonthRange` | **2** (시작월·종료월 범위) |
| 그 외 모든 타입 | **1** |

## 3. 공통코드 자동 연결 규칙

Entity 필드에 `codeGroupCode`가 설정되어 있으면, 빌드 시 `FormFieldItem`에 `codeGroupCode`가 자동으로 연결된다.

```typescript
// 빌드 결과 예시: gender 필드에 GENDER 공통코드 자동 연결
{
    type: 'select',
    fieldKey: 'gender',
    label: '성별',
    codeGroupCode: 'GENDER',  // ← entity.codeGroupCode에서 자동 주입
    colSpan: 1,
    required: false,
}
```

> `codeGroupCode`가 있으면 select/radio/checkbox 타입에서 옵션 목록을 공통코드 API(`/codes`)에서 자동으로 불러온다.

## 4. Key 매칭 규칙 (`getKeyVariants`)

Entity 필드의 `key`는 snake_case로 저장되지만 빌더 내부에서는 다양한 형태로 참조된다.  
빌더는 다음 4가지 변형을 모두 매핑에 등록해 **어떤 형태로 fieldKey를 설정해도 entity 필드를 정확히 찾는다.**

```
원본 key:           disp_at_from
camelCase:          dispAtFrom
suffix 제거:        disp_at       ← _from / _to suffix 제거
suffix 제거 + camelCase: dispAt
```

> `dateRange` 타입 필드는 FE에서 `fieldKey` + `_from` / `_to` 형태로 실제 값을 분리 저장한다.  
> Entity에는 `disp_at_from`, `disp_at_to`로 등록해두어도 빌더는 `disp_at` 하나로 인식한다.

## 5. 빌드 흐름 전체 (`handleBuildFromEntity`)

```
빌드 버튼 클릭
    │
    ▼
slugEntityFields 로드 확인
    │
    ▼
entityFieldMap 생성 (key 4가지 변형 모두 등록)
    │
    ├── [Form 위젯 이미 있는 경우]
    │       ├── 기존 필드: label / required 업데이트만
    │       └── 신규 필드: 기존에 없는 entity 필드만 하단 추가
    │
    └── [Form 위젯 없는 경우]
            └── Form 위젯 + 컨텐츠 + 필드 전체 신규 생성
```

## 6. 코드 위치 — `buildFieldItem` 헬퍼

> 파일: `bo/src/app/admin/templates/make/widget/page.tsx`

```typescript
const buildFieldItem = (f: SlugEntityFieldItem): FormFieldItem => {
    // 1순위: entity fieldType 직접 사용 / 2순위: DB 타입 자동 매핑
    const type = (f.fieldType as SearchFieldType | undefined) ?? mapColumnTypeToFieldType(f.columnType);
    const wideTypes = ['textarea', 'dateRange', 'yearMonthRange'];
    return {
        id: uid(),
        type,
        label: f.label,
        fieldKey: f.key!,
        colSpan: wideTypes.includes(type) ? 2 : 1,
        rowSpan: 1,
        required: f.isNullable === false,
        ...(type === 'date' && { dateSubType: 'date' as const }),
        ...(f.codeGroupCode ? { codeGroupCode: f.codeGroupCode } : {}),
    } as FormFieldItem;
};
```

## 8. 절대 금지 패턴

- `buildFieldItem` 내에서 `fieldType`을 무시하고 `mapColumnTypeToFieldType`만 사용 — **1순위 fieldType 우선 원칙 위반**
- Entity 필드에 `key` 없이 빌드 시도 — key 없는 필드는 빌드에서 제외됨 (`f.key`가 null이면 skip)
- `codeGroupCode`가 있는데 빌더 필드 타입이 `select/radio/checkbox`가 아닌 경우 — 의미 없는 연결. entity 관리에서 타입 확인 필요
- `getKeyVariants` 없이 entity 필드 매핑 — snake_case/camelCase/dateRange suffix 불일치로 매핑 누락 발생
- Entity에 `defaultValue`를 넣으면 [Entity로 빌드] 시 Form 필드에도 자동 반영된다고 가정 — **잘못된 가정**. 현재 `buildFieldItem`(`widget/page.tsx`, §6)은 `defaultValue`를 참조하지 않는다. Entity 관리화면의 `defaultValue`는 저장·조회 용도로만 존재하며, 빌더 필드에 기본값을 반영하려면 `buildFieldItem` 별도 개선이 필요하다

## 10. 파일생성(코드 생성) — Slug Entity → BE Java 파일 자동생성

### 10-1. 개요

Entity 관리 화면(`/admin/settings/slug-entity`)에서 정의한 Slug Entity + 필드 목록을 기반으로,
BE의 CRUD 6종 파일(Entity/Request DTO/Response DTO/Repository/Service/Controller)을
자동 조립해 실제 `bo-api` 소스 디렉토리에 생성하는 기능이다.

> ⚠️ **local 프로파일 전용 기능**이다. dev/운영 프로파일에서는 차단된다.

### 10-2. 선행 조건 — `table_name`

파일생성 버튼은 Entity에 **DB 테이블명(`tableName`)이 입력되어 있어야만 활성화**된다.
테이블명은 `docs/데이터베이스표준.md` 규칙(단수형 snake_case, `TB_` 접두어 금지)을 따라 입력한다.

### 10-3. 2단계 플로우 — 미리보기 → 저장

```
[Entity 필드 편집 화면] → [파일생성] 버튼
    │
    ▼
POST /slug-entity/{id}/generate-preview   ← 파일시스템에 아무것도 쓰지 않음
    │
    ▼
CodeGenerateModal — 6개 파일(+DDL) 탭 미리보기
    │
    ▼ [저장] 클릭 (미리보기 데이터를 그대로 재전송)
    │
POST /slug-entity/{id}/generate-save
    │
    ▼
실제 파일 기록 (기존 파일 있으면 *.bak.{timestamp} 백업 후 덮어씀)
```

### 10-4. 생성되는 파일 6종

| 파일 | 경로(`application-local.yml` `codegen.entity.*-dir`) |
|---|---|
| Entity | `entity-dir` (`entity/`) |
| Request DTO | `dto-dir` (`dto/`) |
| Response DTO | `dto-dir` (`dto/`) |
| Repository | `repository-dir` (`repository/`) |
| Service | `service-dir` (`service/`) |
| Controller | `controller-dir` (`controller/`), **`@ApiLinkedEntity(entityName)` 자동 부여** |

DDL(CREATE TABLE)도 함께 생성되지만 **자동 실행되지 않는다** — `hibernate.ddl-auto: update`(local 프로파일)에 의해 재기동 시 테이블이 자동 생성된다.

> FILE 타입 필드(§0-4)가 있어도 Controller는 일반 CRUD 5종만 생성된다 — 별도의 업로드 전용 엔드포인트는 만들어지지 않는다.
>
> ⚠️ **기존에 이미 `BIGINT`(단일)로 만들어져 있던 컬럼을 FILE 타입으로 재생성하면 주의가 필요하다.** `hibernate.ddl-auto: update`는 `bigint` → `bigint[]` 컬럼 타입 자동 변환을 지원하지 않는다(PostgreSQL이 `USING` 절 없는 변환을 거부). 재기동 로그에는 에러가 남지 않고 넘어가지만 컬럼은 그대로 남아 저장 시점에 타입 불일치로 실패한다. 이 경우 수동으로 `ALTER TABLE {table} ALTER COLUMN {column} TYPE bigint[] USING {column}::bigint[];`를 실행해야 한다(기존 데이터가 있다면 백업 후 진행). 처음부터 FILE로 생성하는 신규 테이블에는 해당하지 않는다.

### 10-5. 필드 → Java 매핑 시 주의사항 (실제 발생했던 버그로 확정된 규칙)

- Entity `key`는 camelCase로 정규화되어 Java 필드명/컬럼명(snake_case)에 각각 사용된다 (`default_val` → Java 필드 `defaultVal`, 컬럼 `default_val`)
- DATE/TIMESTAMPTZ 필드가 있으면 `java.time.LocalDate`/`OffsetDateTime` import가 자동 포함된다
- 마스터 Entity 연결(§0-3)로 자동 추가된 `{마스터slug}_id` 필드도 다른 필드와 동일하게 처리되어 **평범한 BIGINT 컬럼**으로 생성된다 — `@ManyToOne`/`@JoinColumn` 같은 연관관계 매핑은 생성되지 않으며 참조무결성/CASCADE도 없다. (`slug_entity.parent_entity_id` 자체는 메타데이터 테이블의 자기참조 FK로 실제 제약이 걸려있는 것과는 별개 — §0-3 참고)
- FILE 타입 필드(§0-4)는 `List<Long>`(`BIGINT[]`) 컬럼으로 생성된다. Form 위젯이 파일 필드 값을 파일 개수와 무관하게 항상 배열로 전송하는 계약에 맞춘 것이다. 파일 업로드는 `POST /api/v1/file-meta/upload`로 별도 수행하고, 반환된 `id`들을 배열로 모아 일반 Create/Update 요청 바디에 그대로 포함시키면 저장된다 — 필드별 전용 연결 엔드포인트는 생성되지 않는다.
- ENTITY_REF 타입 필드(§0-5)도 FILE과 동일하게 `List<Long>`(`BIGINT[]`) 컬럼으로 생성된다. 배열 각 값은 연동 대상 Entity 레코드의 id다. 어느 Entity를 대상으로 삼는지는 업무 테이블 컬럼이 아니라 필드 정의 메타 `slug_entity_field.connected_entity_id`(§9-2)에 저장되므로, 생성 컬럼 자체는 대상 Entity 정보를 담지 않는다.

### 10-6. 재기동 후 다음 단계 — API 자동 등록

파일생성만으로는 API가 시스템에 등록되지 않는다. **`bo-api` 재기동 후**, `/admin/system/api` 화면([05.builder_api_sync_guide.md](./05.builder_api_sync_guide.md) 참고)에서 **[API 동기화]**를 눌러야 신규 컨트롤러가 `api_info` 테이블에 자동 등록된다.
