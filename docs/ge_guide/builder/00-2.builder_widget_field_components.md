## 0. 문서 목적

00-1 문서의 계층 구조에서 이 문서가 다루는 범위:

```
위젯 컴포넌트 (컨텐츠 컴포넌트)  ← 00-1 문서
└── 필드 컴포넌트                 ← 이 문서의 대상
    └── input, select, date, file, ...
```

필드 컴포넌트는 Form·Search·SubList·Space 위젯 내부의 개별 입력 요소다.  
**위젯 컴포넌트(WidgetRenderer)와 혼동하지 말 것** — 필드 컴포넌트는 `FieldRenderer`가 담당한다.

---

## 1. 왜 필요한가 (Why)

필드 컴포넌트 구조를 모르면 다음 버그가 반드시 발생한다:

- **빌더에서 보이는데 화면에서 안 나옴** — 위젯 Builder 파일(`SearchBuilder.tsx`/`FormBuilder.tsx` 등)의 switch에만 추가하고 `FieldRenderer`에 추가 안 한 경우
- **위젯에서 필드 선택 불가** — 필드 타입을 추가했지만 해당 위젯 팔레트 상수에 추가 안 한 경우
- **FormFieldItem 타입을 못 찾음** — `types.ts`가 아닌 `FormBuilder.tsx`에 정의되어 있기 때문
- **위젯마다 지원 타입 수가 다른 이유 모름** — Search·Form·SubList가 각자 실제로 필요한 타입만 팔레트와 자체 switch에 등록하는 구조적 이유 미파악

---

## 2. 핵심 개념 (What)

### 2-1. 용어 정의

| 용어 | 정의 |
|---|---|
| 필드 컴포넌트 | 위젯 내부 개별 입력 요소. `SearchFieldType`으로 타입 식별 |
| 필드 타입 | `SearchFieldType` union 값 (input, select, date 등 22종) |
| 빌더 측 필드 컴포넌트 | `builder/fields/*.tsx` — 설정 UI (빌더 좌측 패널) |
| 렌더러 측 필드 컴포넌트 | `FieldRenderer.tsx` 내부 렌더링 — 출력 UI (미리보기·운영 페이지) |
| 팔레트 상수 | 각 위젯 Builder 파일에서 선택 가능한 필드 타입 목록을 정의한 배열 |
| `_FieldBase` | 모든 빌더 필드 컴포넌트의 공통 베이스 (라벨·Key·ColSpan·RowSpan·조건) |

### 2-2. 빌더 측 vs 렌더러 측 구조

```
[빌더 좌측 패널]                              [미리보기 · 운영 페이지]
SearchBuilder.tsx / FormBuilder.tsx 등        FieldRenderer
   ↓ 위젯별 자체 switch(field.type)              ↓ switch(field.type)
   위젯마다 필요한 타입만 처리                     22개 case 전체 처리
   (각 *Field.tsx 컴포넌트는 위젯 간 공용)         직접 JSX 렌더링
```

빌더 측은 **위젯(Search/Form/SubList)마다 자체 switch문**을 갖는다. `renderer` 측은 `FieldRenderer` 하나가 22종 전체를 처리한다. 개별 `*Field.tsx` 컴포넌트(InputField, SelectField 등)는 여러 위젯이 공용으로 가져다 쓰지만, "이 타입이 왔을 때 어떤 컴포넌트를 렌더링할지 결정하는 switch"는 위젯마다 따로 있다. 새 필드 타입을 추가할 때는 **그 타입을 쓸 위젯의 Builder 파일 switch**와 **`FieldRenderer`** 양쪽을 모두 수정해야 한다.

|---|---|---|---|---|
| `input` | 텍스트 단일행 입력 | ✅ InputField | ✅ | Search, Form, SubList |
| `select` | 셀렉트박스 (수동/공통코드) | ✅ SelectField | ✅ | Search, Form, SubList |
| `radio` | 라디오 단일선택 | ✅ RadioField | ✅ | Search, Form, SubList |
| `checkbox` | 체크박스 복수선택 | ✅ CheckboxField | ✅ | Search, Form, SubList |
| `button` | 선택 버튼 그룹 | ✅ ButtonField | ✅ | Search, Form |
| `date` | 단일 날짜 (date/yearMonth/datetime) | ✅ DateField | ✅ | Search, Form, SubList |

| `time` | 시간 선택 HH:MM | ✅ TimeField | ✅ | Search, Form, SubList |
| `textarea` | 여러 줄 텍스트 표시/입력 | ✅ TextareaField | ✅ | Space, Form |
| `action-button` | 액션 버튼 (connType 연결) | ✅ ActionButtonField | ✅ | Space |
| `file` | 파일 업로드 | ✅ FileField | ✅ | Form, SubList |
| `image` | 이미지 업로드 (격자 미리보기) | ✅ ImageField | ✅ | Form |
| `video` | 비디오 업로드/URL | ✅ VideoField | ✅ | Form |
| `media` | 이미지+동영상 통합 (1개) | ✅ MediaField | ✅ | Form |
| `editor` | 위지윅 에디터 (dynamic import) | ✅ EditorField | ✅ | Form |
| `color` | Preset 원형 버튼 색상 선택 | ✅ ColorField | ✅ | Form |
| `hidden` | 화면 미노출, 저장 시 defaultValue 포함 | ❌ (설정 불필요) | ✅ null 렌더 | Form |

## 4. 필드 dispatch 구조 (How)

빌더 측은 **위젯 Builder 파일마다 자체 switch**를 갖고, 렌더러 측은 `FieldRenderer` **하나**가 22종 전체를 처리한다. 새 필드 타입을 추가할 때는 **그 타입을 쓸 위젯의 Builder switch**와 `FieldRenderer` 양쪽을 모두 수정해야 한다.

### 4-1. 위젯별 빌더 switch (빌더 측)

Search와 Form은 각자 자신에게 필요한 타입만 처리한다. (SubList는 팔레트 상수 + `toFieldValues`/`fromFieldValues` 변환 함수 방식 — §10 참고)

```typescript
// bo/src/app/admin/templates/make/_shared/components/builder/SearchBuilder.tsx
switch (field.type) {
    case 'input':            return <InputField ... />;
    case 'select':           return <SelectField ... />;
    case 'date':             return <DateField ... />;
    case 'dateRange':        return <DateRangeField ... />;
    case 'radio':            return <RadioField ... />;
    case 'checkbox':         return <CheckboxField ... />;
    case 'button':           return <ButtonField ... />;
    case 'category':         return <CategoryField ... />;
    case 'dateRangeStatus':  return <DateRangeStatusSearchField ... />;
}
```

```typescript
// bo/src/app/admin/templates/make/_shared/components/builder/FormBuilder.tsx
switch (field.type) {
    case 'input':      return <InputField ... />;
    case 'select':     return <SelectField ... />;
    case 'date':       return <DateField ... />;
    case 'dateRange':  return <DateRangeField ... />;
    case 'radio':      return <RadioField ... />;
    case 'checkbox':   return <CheckboxField ... />;
    case 'button':     return <ButtonField ... />;
    case 'editor':     return <EditorField ... />;
    case 'file':       return <FileField ... />;
    case 'image':      return <ImageField ... />;
    case 'video':      return <VideoField ... />;
    case 'media':      return <MediaField ... />;
    case 'textarea':   return <FormTextareaField ... />;
    case 'time':       return <TimeField ... />;
    // hidden은 설정 UI 없음 (defaultValue만 사용)
}
```

> ⚠️ 같은 타입(`input`, `select`, `date` 등)이라도 각 위젯 Builder 파일이 **개별적으로** switch에 등록한다. `*Field.tsx` 컴포넌트 자체는 위젯 간 공용이지만, "이 타입이 오면 이 컴포넌트를 쓴다"는 연결부는 위젯마다 따로 있다. 위젯이 실제로 지원하지 않는 타입(예: Search의 `editor`)은 그 위젯 switch에 애초에 없다.

### 4-2. FieldRenderer (렌더러 측 — 22종 전체를 한 곳에서 처리)

```typescript
// bo/src/app/admin/templates/make/_shared/components/renderer/FieldRenderer.tsx
switch (field.type) {
    case 'input':                                             return <input ... />;
    case 'select':                                            return SelectField 계열 렌더링;
    case 'radio': case 'checkbox': case 'button':             return 선택형 UI;
    case 'date': case 'yearMonth':                            return <input type="date|month" ... />;
    case 'dateRange': case 'yearMonthRange':                  return 시작~종료 input 2개;
    case 'time':                                              return <input type="time" ... />;
    case 'textarea':                                          return 표시/입력 textarea;
    case 'editor':                                            return TipTap/TOAST 에디터 (dynamic import);
    case 'file': case 'image': case 'video': case 'media':    return 업로드 UI;
    case 'color':                                             return ColorPresetSelector;
    case 'action-button':      return <button ...>액션 버튼</button>;
    case 'hidden':             return null;
    case 'message-key-select': return <MessageKeySelector ... />;
    case 'category':           return <CategorySearchField ... />;
    case 'dateRangeStatus':    return 상태 select/radio;
    // textarea는 SpaceRenderer에서도 별도 처리
}
```

> `FieldRenderer`는 22종 전체를 이 하나의 switch에서 직접 처리하며, 빌더 측 위젯별 switch와는 코드를 공유하지 않는 독립된 구현이다.

**preview vs live 모드 차이**:
- `preview`: `disabled=true`, 샘플 옵션 3~4개, pointer-events 차단
- `live`: 실제 입력 가능, 공통코드 전체 옵션, onChange 작동

## 6. 빌더 측 공통 베이스 구조 (How)

모든 빌더 필드 컴포넌트는 공통 베이스 컴포넌트 3개를 조합해서 구성한다.

### 6-1. _FieldBase (가장 중요)

```typescript
// bo/src/app/admin/templates/make/_shared/components/builder/fields/_FieldBase.tsx
<FieldBase
    label={values.label}
    labelMsgKey={values.labelMsgKey}
    fieldKey={values.fieldKey}
    colSpan={values.colSpan}
    colSpanMode={colSpanMode}       // 'button'(1~5) | 'input'(1~12)
    rowSpan={values.rowSpan}
    rowSpanConfig={rowSpanConfig}   // Form 전용 { min:1, max:30 }
    required={values.required}
    readonly={values.readonly}
    hideCondition={values.hideCondition}
    disableCondition={values.disableCondition}
    onChange={onChange}
/>
```

`_FieldBase`가 처리하는 공통 속성:

| 속성 | 설명 |
|---|---|
| 라벨 + labelMsgKey | 직접 입력 또는 MessageKeySelector (다국어 모드) |
| fieldKey | 직접 입력 또는 Slug Entity selectbox (Form 전용) |
| colSpan | button 모드: 1~5 버튼 / input 모드: 1~12 숫자 |
| rowSpan | Form·Layer 전용, rowSpanConfig 전달 시만 표시 |
| required | 필수 항목 토글 |
| readonly | 읽기 전용 토글 (Form 전용) |
| hideCondition | 동적 HIDE 조건 (예: "status=1,type=Y") |
| disableCondition | 동적 Disable 조건 |

### 6-2. _FieldOptions (select·radio·checkbox·button 전용)

```typescript
// bo/src/app/admin/templates/make/_shared/components/builder/fields/_FieldOptions.tsx
<FieldOptions
    options={values.options}
    codeGroupCode={values.codeGroupCode}
    codeGroups={codeGroups}
    codeGroupsLoading={codeGroupsLoading}
    onChange={onChange}
/>
```

수동 입력 탭 / 공통코드 탭 2가지 모드 전환을 처리한다.

### 6-3. _ToggleRow (토글 UI 공용)

```typescript
// bo/src/app/admin/templates/make/_shared/components/builder/fields/_ToggleRow.tsx
<ToggleRow
    label="필수 항목"
    value={values.required ?? false}
    onChange={v => onChange({ required: v || undefined })}
/>
```

boolean 설정을 토글 버튼 한 줄로 표현하는 공통 컴포넌트.

### 6-4. FieldEditProps 프로토콜

모든 빌더 필드 컴포넌트가 받는 공통 props 타입:

```typescript
// bo/src/app/admin/templates/make/_shared/components/builder/fields/types.ts
interface FieldEditProps {
    values: FieldEditValues;                  // 현재 필드 설정값
    onChange: (updates: Partial<FieldEditValues>) => void; // 변경 핸들러
    colSpanMode: ColSpanMode;                 // 'button' | 'input' 모드
    rowSpanConfig?: { min: number; max: number };
    codeGroups: CodeGroupDef[];               // 공통코드 목록
    codeGroupsLoading: boolean;
    autoFocus?: boolean;
    compact?: boolean;                        // 한 줄 배치 (Space 전용)
    hideColSpan?: boolean;                    // SubList 컬럼 등에서 숨김
    hideConditionFields?: boolean;            // 동적 조건 입력 숨김
    onLabelKeyDown?: (e: KeyboardEvent) => void;
    slugEntityFields?: { key: string|null; label: string }[]; // Form 전용
}
```

> **`TableColumnConfig.data` (컬럼 Data 표현식)**  
> `accessor`(Key)로 단순히 값을 꺼내는 대신, 여러 컬럼을 조합·분기해 셀 값을 만드는 표현식 속성.  
> `types.ts`의 `TableColumnConfig`에 `data?: string`으로 정의되며, 렌더러는 `evalColumnDataExpr(col.data, row)`(`_shared/utils.ts`)로 평가한다.  
> 문법(조건 `=`·분기 `?/\|`·연결 `+`·리터럴 `'...'`)과 preview/live 동작 규칙은 **`02.input_builder_outputpanel.md` §4-11** 참조.

## 8. 렌더러 측 필드 처리 (How)

### 8-1. FieldRenderer 진입점

```typescript
// bo/src/app/admin/templates/make/_shared/components/renderer/FieldRenderer.tsx
<FieldRenderer
    field={SearchFieldConfig}
    value={...}
    onChange={...}
    mode={'preview' | 'live'}
    codeGroups={...}
/>
```

### 8-2. 특이 처리 항목

| 타입 | 특이사항 |
|---|---|
| `editor` | `dynamic import` (SSR 불가) — Next.js `ssr: false` 필수 |
| `file` / `image` / `video` / `media` | Edge 브라우저 overflow 버그 우회를 위해 `FileInput` 컴포넌트 사용 |
| `hidden` | `return null` — 화면에 렌더링 없음, 저장 시 defaultValue 포함 |
| `textarea` | Space 위젯에서는 표시 전용 (onChange 없음), Form에서는 입력 가능 |
| `category` | `CategorySearchField` 별도 컴포넌트로 분리 |
| `yearMonth` / `yearMonthRange` | 레거시 타입 — 내부적으로 date/dateRange로 fallthrough 처리 |

## 10. 새 필드 타입 추가 체크리스트 (How)

새 필드 타입을 추가할 때 수정해야 할 파일은 최소 6개다. 순서대로 진행한다.

| 순서 | 파일 | 작업 |
|---|---|---|
| 1 | `types.ts` | `SearchFieldType` union에 새 타입 추가 |
| 2 | `types.ts` | `SearchFieldConfig`에 전용 속성 추가 (있는 경우) |
| 3 | `builder/fields/{Name}Field.tsx` | 빌더 설정 UI 컴포넌트 신규 생성 (`_FieldBase` 사용) |
| 4 | `builder/fields/index.ts` | 신규 컴포넌트 export 등록 |
| 5 | 해당 위젯 Builder 파일(예: `SearchBuilder.tsx`, `FormBuilder.tsx`)의 자체 switch | `switch` case 추가 |
| 6 | `renderer/FieldRenderer.tsx` | `switch` case 추가 |
| 7 | 해당 위젯 Builder 팔레트 상수 | 사용할 위젯의 타입 배열에 추가 |

> ⚠️ **새 속성 추가 시 전 빌더 체크 필수**: SearchBuilder·FormBuilder·SubListBuilder·SpaceBuilder 각각 별도 매핑 로직이 있다. SubListBuilder는 `toFieldValues`/`fromFieldValues` 변환 함수도 수정해야 한다.

## 11. 절대 금지 패턴

### 금지 1. 빌더 측 switch만 수정하고 렌더러 측 누락

```typescript
// ❌ 금지 — 위젯 Builder 파일에만 추가하고 FieldRenderer 누락
// builder/FormBuilder.tsx (또는 SearchBuilder.tsx)에만 case 추가

// ✅ 올바른 방법 — 반드시 2곳 동시 수정
// 1. bo/.../builder/{위젯}Builder.tsx — switch case 추가
// 2. bo/.../renderer/FieldRenderer.tsx — switch case 추가
```

### 금지 2. 팔레트 상수 누락

```typescript
// ❌ 금지 — 새 필드 타입 추가 후 위젯 팔레트 상수 미등록
// → 빌더에서 새 타입 선택 불가능

// ✅ 올바른 방법 — 해당 위젯 Builder 파일의 팔레트 상수 배열에 추가
// bo/.../builder/FormBuilder.tsx → FORM_FIELD_TYPES 배열에 추가
```

### 금지 3. 동적 Tailwind 클래스 사용

```typescript
// ❌ 금지 — 런타임 문자열 조합 (Tailwind purge에서 제거됨)
className={`bg-${color}-500`}

// ✅ 올바른 방법 — 정적 맵 사용
const BG_COLOR_MAP = { black: 'bg-slate-900', green: 'bg-green-600', ... }
className={BG_COLOR_MAP[color]}
```

### 금지 4. 개별 Field 컴포넌트 직접 렌더링

```typescript
// ❌ 금지 — FieldRenderer를 거치지 않고 직접 렌더링
import { InputField } from '../builder/fields/InputField';
<InputField values={...} />  // 빌더 설정 컴포넌트를 출력 UI로 사용

// ✅ 올바른 방법
// bo/.../renderer/FieldRenderer.tsx
<FieldRenderer field={fieldConfig} value={val} onChange={...} mode="live" />
```

### 금지 5. editor 타입 SSR 렌더링

```typescript
// ❌ 금지 — editor 필드를 SSR 컴포넌트에서 직접 import
import WysiwygEditor from './WysiwygEditor';

// ✅ 올바른 방법 — dynamic import + ssr:false
const WysiwygEditor = dynamic(() => import('./WysiwygEditor'), { ssr: false });
```

## 12. 혼동하기 쉬운 케이스

| 케이스 | 원인 | 해결 |
|---|---|---|
| 빌더에서 보이는데 운영 페이지에서 안 나옴 | `FieldRenderer`에 case 누락 | FieldRenderer switch에 새 타입 추가 |
| 운영 페이지에서 보이는데 빌더 설정 UI가 없음 | 해당 위젯 Builder 파일의 switch에 case 누락 | 해당 위젯 Builder switch에 추가 |
| `FormFieldItem` 타입을 `types.ts`에서 못 찾음 | FormFieldItem은 `FormBuilder.tsx`에 정의 | `FormBuilder.tsx`에서 import |
| 위젯에서 새 필드 타입 선택 불가 | 해당 위젯 팔레트 상수에 미추가 | FormBuilder.tsx의 FORM_FIELD_TYPES 등에 추가 |
| `textarea`가 Space에서 편집 안 됨 | Space의 textarea는 표시 전용 의도적 동작 | Space에서 textarea는 content 표시용 — 정상 |
| `yearMonth`로 저장된 데이터가 date로 처리됨 | yearMonth는 레거시 타입, DateField로 fallthrough 처리 | dateSubType: 'yearMonth'로 확인 |
| SubListColumn 타입이 SearchFieldType과 다름 | SubListColumnType은 `renderer/types.ts`에 별도 정의 | `renderer/types.ts`에서 SubListColumnType 참조 |
