# 빌더 데이터 처리 가이드 (Data Process)

> 작성일: 2026-05-27  
> 대상: 운영 페이지(`/admin/widgetSub/[slug]`)의 데이터 호출·저장·삭제 흐름을 이해하거나 수정하는 모든 에이전트

> - `bo/src/app/admin/widgetSub/[slug]/page.tsx` — 운영 페이지 (모든 위젯 live 모드, page 모드 저장/읽기)
> - `bo/src/app/admin/templates/make/_shared/components/renderer/WidgetRenderer.tsx` — 팝업 모드 (handleInternalPopupOpen 읽기 + handlePopupContentAction 저장)
> - `bo/src/app/admin/templates/make/_shared/utils.ts` — validateFormFields, **buildDataJson**, getSpaceGridColumn 등
> - `bo/src/app/admin/templates/make/_shared/templateApi.ts` — 템플릿 조회·저장 공통 유틸
> - `bo-api/.../controller/PageDataController.java` — 데이터 CRUD API
> - `bo-api/.../service/PageDataService.java` — 데이터 비즈니스 로직
> - `bo-api/.../controller/PageFileController.java` — 파일 업로드·다운로드 API

> [`06.builder_entity_to_api_guide.md`](./06.builder_entity_to_api_guide.md) §1 참고.

## 0. ⭐ 핵심 아키텍처 원칙 — 반드시 숙지

### 0-1. slug = 데이터 네임스페이스

`page_data` 테이블의 모든 데이터는 `template_slug` 컬럼으로 격리된다.  
하나의 운영 페이지에서 여러 위젯이 각자 다른 `connectedSlug`를 가질 수 있다.

```
/admin/widgetSub/user-form
  └── Form 위젯 (connectedSlug: 'user-info')     → page_data WHERE template_slug = 'user-info'
  └── SubList 위젯 (connectedSlug: 'user-roles') → page_data WHERE template_slug = 'user-roles'
  └── Table 위젯 (connectedSlug: 'user-list')    → page_data WHERE template_slug = 'user-list'
```

> **slug 없이는 어떤 저장도 불가능하다.** `connectedSlug`가 비어있으면 해당 위젯의 저장 로직은 전부 건너뜀.

### 0-2. widgetId = FE 상태 키

동일 타입의 위젯이 한 페이지에 여러 개 있어도 `widgetId`로 구분한다.  
FE의 모든 상태 맵(formValuesMap, tableDataMap, …)은 widgetId를 키로 사용한다.

```typescript
formValuesMap[widgetId]        // 해당 Form 위젯의 입력값
tableDataMap[widgetId]         // 해당 Table 위젯의 데이터
subListRowsMap[widgetId]       // 해당 SubList 위젯의 행 목록
```

### 0-3. contentKey = dataJson 내부 섹션 구분자

하나의 `page_data` 레코드에 여러 위젯 데이터를 통합 저장할 때 `contentKey`로 섹션을 분리한다.

```json
{
    "id": 1,
    "userInfo": { "name": "홍길동", "email": "hong@test.com" },
    "orderHistory": { "rows": [{ "product": "노트북", "qty": 1 }] },
    "relatedTags": [3, 7, 12]
}
```

| contentKey 유무 | dataJson 저장 방식 |
|---|---|
| `contentKey` 없음 | 필드값을 최상위에 flat하게 저장 |
| `contentKey` 있음 | `{ [contentKey]: { 필드값들 } }` 중첩 구조로 저장 |

### 0-4. groupId = 다중 slug 묶음 키

한 페이지에서 서로 다른 `connectedSlug`에 동시 저장할 때 FE에서 UUID를 생성하여 모든 레코드에 동일한 `groupId`를 심는다.  
수정/삭제 시 `groupId`만 알면 연관 레코드 전체를 한 번에 처리할 수 있다.

```
신규 저장 (다중 slug):
  crypto.randomUUID() → groupId 생성
  POST /page-data/user-info    { dataJson, groupId: "uuid-abc" }
  POST /page-data/user-roles   { dataJson, groupId: "uuid-abc" }
  POST /page-data/user-address { dataJson, groupId: "uuid-abc" }

수정 모드 진입:
  URL: ?group_id=uuid-abc
  GET /page-data/user-info/group/uuid-abc
  GET /page-data/user-roles/group/uuid-abc
  GET /page-data/user-address/group/uuid-abc

삭제:
  DELETE /page-data/user-info/group/uuid-abc  (← 첫 번째 slug만 호출, BE에서 전체 삭제)
```

### 0-5. slug_relation = 슬러그 간 관계 매핑

서로 다른 slug에 저장된 데이터를 연결해야 할 때 `slug_relation` 테이블을 사용한다.  
하드코딩 없이 설정 기반으로 슬러그 간 관계를 정의하고, BE가 이를 참조해 조회한다.

> **관리 화면**: `/admin/system/slug-relation` — 설정 CRUD UI

#### 매핑 테이블 구조 (slug_relation)

| 컬럼 | 타입 | 설명 |
|---|---|---|
| `id` | BIGINT | PK, GENERATED ALWAYS AS IDENTITY |
| `master_slug` | TEXT | 기준 슬러그 (조회 대상) |
| `slave_slug` | TEXT | 관계 슬러그 (필터링 또는 fetch 기준) |
| `master_key` | TEXT | master에서 비교할 필드 경로 (예: `id`, `rel.product-data`) |
| `slave_key` | TEXT | slave에서 master를 참조하는 필드 경로 (예: `product.id`) — ⚠️ 아래 폴백 미지원, 항상 전체 경로 필수 |
| `join_type` | TEXT | `EQ` (단순 동등) / `ARRAY_CONTAINS` (배열 포함) |
| `slave_filter` | TEXT | slave 조회 시 고정 조건 (예: `depth=3`). null이면 조건 없음 |
| `relation_dir` | TEXT | `FILTER` (slave→master 필터링) / `FETCH` (master 조회 시 slave 데이터 포함) |
| `fetch_fields` | TEXT | FETCH 방향 시 표시할 slave 필드 경로 (예: `form1.title`) |
| `fetch_separator` | TEXT | FETCH 결과 구분자 (기본 `,`) |
| `slave_type` | TEXT | `TABLE` (직접 추출) / `CATEGORY` (계층 순회 추출) — 기본 `TABLE` |
| `category_depth` | INT | CATEGORY 타입 전용. N번째 단계 이름 1개 반환 (기본 `1` = 최상위) |
| `description` | TEXT | 관계 설명 (선택) |

```sql
-- DB 생성 DDL
CREATE TABLE slug_relation (
    id               BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    master_slug      TEXT NOT NULL,
    slave_slug       TEXT NOT NULL,
    master_key       TEXT NOT NULL DEFAULT 'id',
    slave_key        TEXT NOT NULL,
    join_type        TEXT NOT NULL DEFAULT 'EQ',        -- EQ | ARRAY_CONTAINS
    slave_filter     TEXT,                              -- null이면 조건 없음
    relation_dir     TEXT NOT NULL DEFAULT 'FILTER',    -- FILTER | FETCH
    fetch_fields     TEXT,
    fetch_separator  TEXT NOT NULL DEFAULT ',',
    slave_type       TEXT NOT NULL DEFAULT 'TABLE',     -- TABLE | CATEGORY
    category_depth   INT  NOT NULL DEFAULT 1,           -- CATEGORY 타입에서만 의미 있음
    description      TEXT,
    created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

#### FILTER 방향 — slave를 통한 master 필터링

카테고리 선택 시 해당 카테고리에 속한 제품만 조회하는 케이스:

```
-- slug_relation 설정 값
master_slug:  product-data
slave_slug:   cate1-data
master_key:   id
slave_key:    product.id          -- cate1-data depth=3 항목의 dataJson.product.id
join_type:    EQ
slave_filter: depth=3             -- cate1-data에서 depth=3인 항목만 참조
relation_dir: FILTER
```

```
조회 흐름 (FE 검색 파라미터 → BE 처리):
  FE: product-data 목록 조회 + eq_categoryId=695 파라미터 전달
    → BE: slug_relation 조회 (master_slug=product-data, relation_dir=FILTER)
    → cate1-data WHERE depth=3 AND parentId=695 조회 → dataJson.product.id 목록 [684, 723]
    → product-data WHERE dataJson.id IN (684, 723) 반환
```

> **Master Key bare fieldKey 폴백**: contentKey를 포함한 전체 경로(`form1.title`)가 기본이지만,
> contentKey 없이 필드명만 적어도(`title`) 동작한다 — BE가 해당 레코드의 dataJson 전체를 재귀
> 탐색해 그 이름이 **유일하게 1곳에서만** 발견되면 자동으로 그 값을 사용한다.
> 같은 이름이 2곳 이상에 있으면 모호하므로 매칭하지 않는다(null) — 안전을 위해 되도록
> 전체 경로를 명시하는 것을 권장한다.

---

#### FILTER 방향 — 자기참조(Self-Reference) 케이스

`master_slug`와 `slave_slug`가 **같은 슬러그**인 FILTER 관계도 있다. 빌더 검색필드 타입
`category`(대분류→중분류 계단식 선택)가 이 패턴을 사용한다.

```
-- slug_relation 설정 예시
master_slug:  category-data
slave_slug:   category-data   -- master와 동일(자기참조)
master_key:   id              -- 자기참조에서는 참조되지 않음
slave_key:    product.id      -- 연결 레코드 판별 + 부모경로 유도에 사용
join_type:    EQ
slave_filter: depth=3         -- 연결 레코드 판별 조건 (예: dataJson.product.depth = "3")
relation_dir: FILTER
```

**일반 FILTER와의 차이**

- 일반 FILTER는 `slave_key` 경로의 값을 그대로 master id로 사용하지만, 자기참조에서는
  `master_key`를 쓰지 않는다 — **매칭된 slave 레코드 자신의 id**가 곧 필터 대상 id다.
  slave 레코드 자체가 목록에 표시되는 행(카테고리와 연결된 레코드)이기 때문이다.
- `slave_key`의 마지막 세그먼트를 `parentId`로 치환한 경로(예: `product.id` → `product.parentId`)로
  카테고리 계층을 타고 올라가며, 선택한 카테고리 + 모든 하위 카테고리의 id를 먼저 수집한다.
  이때 `slave_key` 경로 값이 **NULL인 레코드만 "카테고리 노드"**로 취급해 순회하고,
  값이 있는 레코드(연결 레코드)는 순회 대상에서 제외한다.
- 수집한 카테고리 id 목록으로 `parentId 경로 IN (...)` + `slave_filter` 조건에 맞는 연결
  레코드를 찾아, 그 레코드들의 id를 최종 필터 id로 사용한다.

**빌더 연동**: 검색필드 타입을 `category`로 설정하고 `relationSlugId`에 위 slug_relation id를
지정하면, FE가 선택값(가장 깊이 선택한 카테고리의 id)을 `rel_{relationSlugId}` 파라미터로
전송한다.

> ⚠️ **필드 경로는 실제 데이터 구조와 반드시 일치해야 한다**: `slave_key`(및 그로부터 유도되는
> parentId 경로)가 데이터 마이그레이션 등으로 실제 필드 위치와 어긋나면, BE는 에러 없이
> **매칭 0건**을 반환하고 검색이 조용히 실패한다(전체 목록 또는 0건으로 보임 — 에러 메시지가
> 없어 원인 파악이 어렵다). 관계 설정을 새로 만들거나 데이터 구조가 바뀐 뒤에는 실제
> `page_data.data_json` 샘플과 `slave_key` 경로를 반드시 대조 확인할 것.

#### FETCH 방향 — master 조회 시 slave 데이터 포함

FETCH 결과는 **`_fetchedRel{relationId}`** 키로 master row에 주입된다.

> **FE 빌더 연동**: Table 위젯 컬럼 설정에서 "연결 Slug"를 선택하면  
> `accessor`가 자동으로 `_fetchedRel{id}`(관계 PK 기반)로 설정된다.  
> 예: slug_relation id=2 선택 → accessor = `_fetchedRel2`

**`fetch_fields` 유무에 따른 반환 방식 요약**

| fetch_fields | BE 반환값 | FE accessor 예시 | 비고 |
|---|---|---|---|
| 있음 | 문자열 (`"값1, 값2"`) | `_fetchedRel2` | 여러 결과는 fetch_separator로 합침 |
| 없음 | Map 객체 전체 | `_fetchedRel2.form1.title` | slug_relation 1개로 여러 컬럼 커버 가능 |

> **⚠️ FE 주의**: accessor를 `_fetchedRel{id}`(Map 전체)로 설정하면 `[object Object]`가 표시된다.  
> fetch_fields 없이 사용할 때는 반드시 accessor를 `_fetchedRel{id}.필드경로` 형태로 직접 입력해야 한다.

> **`_fetchedRel{id}` 내부 contentKey 폴백**: `_fetchedRel{id}.form1.title`처럼 contentKey를 포함한
> 전체 경로가 기본이지만, 같은 relation 안에서 필드명이 유일하면 `_fetchedRel{id}.title`처럼
> contentKey를 생략해도 동작한다. `_fetchedRel{id}` 접두사 자체는 "연동 slug에서 온 값"을 구분하는
> 필수 식별자라 생략 불가 — 그 뒤 contentKey 세그먼트만 생략 가능하다.
> 같은 relation 안에 동일 필드명이 여러 섹션에 있으면 모호하므로 생성되지 않는다.
>
> **⚠️ Slave Key는 이 폴백을 지원하지 않는다** — Slave Key는 slave 테이블을 SQL로 직접 조회하는
> 구조라 필드명 자동 탐색이 불가능하다(전체 스캔 성능 저하 + 레코드마다 경로가 다를 경우 조인
> 정합성 문제). Slave Key는 항상 정확한 전체 경로를 명시해야 한다.

##### ① TABLE 타입 — slave에서 직접 값 추출 (slave_type: TABLE)

slave 슬러그에 저장된 레코드 자체에 원하는 값이 있을 때 사용한다.

**fetch_fields 있음 — 특정 필드 값을 문자열로 추출**

여러 slave 레코드가 있으면 `fetch_separator`로 합쳐 반환한다.

```
-- slug_relation 설정 예시 (id=5)
master_slug:     제품담당자slug
slave_slug:      product-data
master_key:      rel.product-data   -- 담당자 dataJson의 rel.product-data (ID 배열)
slave_key:       id
join_type:       ARRAY_CONTAINS
relation_dir:    FETCH
fetch_fields:    product.title       -- 추출할 필드 경로
fetch_separator: ,
slave_type:      TABLE
```

```
조회 결과:
  "_fetchedRel5": "제품1-1, 제품2"

FE accessor: _fetchedRel5
```

**fetch_fields 없음 — slave 레코드 전체를 Map으로 반환 (첫 번째 레코드)**

slug_relation 1개로 여러 필드를 커버할 때 사용한다.  
FE에서 accessor를 dot notation으로 직접 지정하면 원하는 필드에 접근할 수 있다.

```
-- slug_relation 설정 예시 (id=5, fetch_fields 비워둠)
slave_type:  TABLE
fetch_fields: (비워둠)
```

```
조회 결과:
  "_fetchedRel5": { "product": { "title": "제품1-1", "code": "P001", ... } }

FE accessor 예시:
  제품명 컬럼  → accessor: _fetchedRel5.product.title
  제품코드 컬럼 → accessor: _fetchedRel5.product.code
```

##### ② CATEGORY 타입 — 카테고리 계층을 거슬러 올라가며 추출 (slave_type: CATEGORY)

slave 슬러그 안에 **연결 레코드**(링크)와 **카테고리 레코드**(실제 이름)가 섞여 있을 때 사용한다.

**두 종류의 레코드 구조**

| 구분 | dataJson 예시 | 설명 |
|---|---|---|
| 연결 레코드 | `{ "product": { "id": "723", "parentId": "700" } }` | master와 카테고리를 연결하는 링크 |
| 카테고리 레코드 | `{ "form1": { "title": "중분류명", "parentId": "698" } }` | 실제 카테고리 이름·데이터 보유 |

**순회 흐름 (fetch_fields 있음/없음 공통)**

```
slave_key = "product.id" → 연결 레코드 탐색
                          → product.parentId 로 카테고리 레코드 이동 (1회)
                          → 카테고리 레코드 내 parentId 경로 자동 탐지로 상위 이동 반복
                          → parentId 빈 문자열 = 최상위 도달

수집된 전체 경로 (최상위 → 하위 순):
  index 0 = 대분류 레코드 (depth=1)
  index 1 = 중분류 레코드 (depth=2)
  index 2 = 소분류 레코드 (depth=3)
```

> **parentId 자동 탐지**: fetch_fields 없을 때 BE가 카테고리 레코드에서  
> `parentId` 키를 가진 섹션을 자동으로 찾아 상위 이동 경로를 결정한다.  
> (예: `{ form1: { parentId: "..." } }` → `form1.parentId` 자동 탐지)

**fetch_fields 있음 — category_depth에 해당하는 이름을 문자열로 반환**

한 master 레코드에 연결 레코드가 여러 개이면 각 카테고리 이름을 `fetch_separator`로 합쳐 반환한다.

```
-- 설정 예시 (id=2 — 대분류 이름 표시)
fetch_fields:    form1.title   -- 카테고리 이름 필드 + parentId 경로 파생에 사용
category_depth:  1
```

```
조회 결과:
  "_fetchedRel2": "대분류A"          (연결 레코드 1개)
  "_fetchedRel2": "대분류A, 대분류B"  (연결 레코드 2개, fetch_separator=",")

FE accessor: _fetchedRel2
```

**category_depth 의미표**

| depth 값 | 반환값 |
|---|---|
| 1 | 최상위(대분류) 이름 또는 레코드 |
| 2 | 두 번째 단계(중분류) 이름 또는 레코드 |
| N | N번째 단계 이름 또는 레코드 (경로 길이 초과 시 null) |

**fetch_fields 없음 — category_depth에 해당하는 카테고리 레코드 전체를 Map으로 반환**

이름 외에 카테고리 레코드의 여러 필드가 필요할 때 사용한다.  
첫 번째 연결 레코드 기준으로 처리한다.

```
-- 설정 예시 (id=3 — 중분류 레코드 전체 반환)
fetch_fields:    (비워둠)
category_depth:  2
```

```
조회 결과:
  "_fetchedRel3": { "form1": { "title": "중분류명", "content1": "설명", "parentId": "695" } }

FE accessor 예시:
  중분류명  컬럼 → accessor: _fetchedRel3.form1.title
  중분류설명 컬럼 → accessor: _fetchedRel3.form1.content1
```

**설정 비교 (id=2 vs id=3)**

```
-- id=2: 대분류 이름만 (fetch_fields 있음 → 문자열)
fetch_fields:    form1.title
category_depth:  1
→ "_fetchedRel2": "대분류명"
   FE accessor: _fetchedRel2

-- id=3: 중분류 레코드 전체 (fetch_fields 없음 → Map)
fetch_fields:    (없음)
category_depth:  2
→ "_fetchedRel3": { "form1": { "title": "중분류명", "content1": "..." } }
   FE accessor: _fetchedRel3.form1.title / _fetchedRel3.form1.content1
```

> **slave_type 선택 기준**
> - slave 레코드 자체에 원하는 값이 있으면 → `TABLE`
> - slave에 연결 레코드(링크)만 있고 계층 부모로 올라가야 값이 있으면 → `CATEGORY`

> **fetch_fields 설정 기준**
> - 특정 필드 1개를 문자열로 표시, 다중 결과를 separator로 합쳐야 하면 → 있음
> - 같은 slug_relation으로 여러 컬럼을 커버하고 싶으면 → 없음 (FE accessor에서 직접 경로 지정)

> **slug_relation vs 직접 저장**: 데이터가 다른 slug에 있고 런타임에 연결이 필요한 경우 사용.  
> 같은 slug에 저장되는 관계는 groupId(§0-4)로 충분하다.

#### 0-5-1. 자기참조(master_slug == slave_slug) FILTER relation (신규)

`master_slug`와 `slave_slug`가 **같은 slug**를 가리키는 FILTER relation도 등록 가능하다 — "이 slug 자신을 계층으로 검색"해야 하는데 relation 방향이 반대(예: 원래는 A가 master였는데 이제 검색 대상이 A 자기 자신인 경우)일 때 쓴다. BE `resolveFilterRelationIds`(`PageDataService.java`)는 계층 순회(하위 id 수집) 로직은 자기참조 여부와 무관하게 동일하게 동작시키되, **마스터 id 추출만** 분기한다:
- 일반(`master_slug != slave_slug`): `extractField(dataJson, slave_key)`로 마스터 id 추출
- 자기참조(`master_slug == slave_slug`): 조회된 행 자신의 **page_data 네이티브 `id` 컬럼**을 마스터 id로 사용

실제 사례·정확한 relation 등록값은 `00-3.builder_field_reference.md` §17-1 참고.

#### 0-5-2. 조인 검색(join search)용 `resolveJoinFilterIds` — `resolveFilterRelationIds`와 별개 (신규)

검색 필드(Select/Input)의 `joinRelationSlugId`가 쓰는 BE 메서드는 `resolveFilterRelationIds`(카테고리 계층 전용, 위 0-5-1 포함)가 **아니라** 신규 메서드 `resolveJoinFilterIds`다. 계층 순회 없이 "연동 slug에서 조건에 맞는 레코드를 찾아 slave_key로 값 추출 → master_key 경로가 그 값과 일치하는 메인 slug 행 id 수집" 하는 평면 조인이다. 상세는 `00-3.builder_field_reference.md` §1-1 참고.

---

#### 조인 검색 — 연동 slug의 필드 값으로 현재 목록 필터링 (`joinr_`/`joink_`/`joinv_`)

검색필드 타입 `select` 또는 `input`에 `joinRelationSlugId` + `joinSlaveKey`를 설정하면,
**다른 slug에 저장된 필드 값**으로 현재 목록을 필터링할 수 있다. FILTER/FETCH와는 별개의
파라미터 규약을 쓰며, **slug_relation의 `relation_dir` 값은 검사하지 않는다** — 이미
FETCH용으로 등록된 관계 행을 조인 검색에 그대로 재사용해도 된다(단, 아래 "필수 조건"은
반드시 맞아야 한다).

```
-- FE가 검색필드 값 선택 시 전송하는 파라미터 (그룹키 n = fieldKey)
joinr_{n} = relationSlugId
joink_{n} = 연동 slug에서 검색할 필드 경로 (input 타입이면 앞에 '~' 붙여 ILIKE, select면 EQ)
joinv_{n} = 사용자가 선택/입력한 검색값
```

```
조회 흐름:
  ① 연동 slug(slug_relation.slave_slug)에서 joink_/joinv_ 조건(+ slave_filter)에 맞는 레코드 조회
  ② 매칭된 레코드에서 slug_relation.slave_key 경로 값을 추출 (연결값)
  ③ 현재 목록(slug_relation.master_slug)에서 master_key 경로 값이 연결값과 일치하는 행의 id 수집
  ④ 그 id 목록으로 현재 목록을 필터링 (여러 조인 필드 동시 검색 시 AND 교집합)
```

**필수 조건 — ⚠️ 가장 흔한 실수**

`slug_relation.master_slug`는 반드시 **현재 검색 대상 목록 자신의 data slug**(Table 위젯의
`connectedSlug`)와 같아야 한다. FETCH용으로 만든 관계를 재사용할 때 이 조건을 놓치기 쉬운데,
`master_slug`가 어긋나면 ③에서 엉뚱한 슬러그의 id를 걸러내 **항상 0건**이 되거나(관계는
존재하니 `continue`로 스킵되지 않아 필터 자체는 "적용된 것처럼" 동작하지만 대상이 틀려서
결과가 항상 비게 된다), 우연히 id 네임스페이스가 겹치면 **엉뚱한 행이 걸린다**. 에러 없이
조용히 틀린 결과가 나오므로 관계를 재사용하기 전에 반드시 `master_slug`를 확인할 것.

**지원하지 않는 조합**

- `join_type = ARRAY_CONTAINS`인 관계는 조인 검색에서 지원하지 않는다(로그 경고 후 해당
  조건 스킵). 배열 포함 조건이 필요하면 FETCH 방향 관계를 별도로 구성할 것.

### 0-6. flattenPageDataItem = page_data row 처리 표준

page_data API 응답 item을 row로 다루는 모든 곳에서 반드시 `flattenPageDataItem`을 사용한다.

```typescript
// utils.ts — 공통 함수
flattenPageDataItem(item)
// item: { id, groupId?, dataJson, createdAt?, ... }
// 반환: { _id, _groupId, ...flat key, ...원본 중첩 구조, createdAt, ... }
```

**적용 대상 (예외 없음)**

| 사용처 | 적용 방법 |
|---|---|
| 테이블 조회 | `.map(flattenPageDataItem)` |
| 카테고리 위젯 항목 조회 | `flattenPageDataItem(item)` 후 flat key로 접근 |
| SelectField SLUG 모드 옵션 로드 | `flattenPageDataItem(item)` 후 `optionValueKey`, `optionTextKey` 접근 |
| MultiSelect 옵션 로드 | `flattenPageDataItem(item)` 후 flat key로 접근 |
| `parseActionParams` row 인자 | `flattenPageDataItem(item)` 결과를 넘길 것 |

> **절대 금지**: `item.dataJson`에서 직접 값을 꺼내거나 별도 평탄화 로직 작성 금지.  
> 새 위젯·렌더러에서 page_data item을 다룰 때도 반드시 동일하게 적용.

## 1. 데이터 모델 (dataJson 구조)

### 1-1. 단순 구조 (Form 단일, contentKey 없음)

```json
{
    "id": 1,
    "name": "홍길동",
    "email": "hong@test.com",
    "status": "active",
    "profileImage": [101, 102],
    "createdAt": "2026-05-27T10:00:00"
}
```

> - `id`는 BE에서 INSERT 후 자동 주입 (FE에서 보내지 않아도 됨)
> - 파일 필드(`file` / `image`)는 파일 ID 숫자 배열로 저장
> - `createdAt` 등 감사 컬럼은 테이블 조회 시 FE에서 flat하게 병합

### 1-2. 중첩 구조 (Form + SubList, contentKey 있음)

```json
{
    "id": 1,
    "userInfo": {
        "name": "홍길동",
        "email": "hong@test.com",
        "profileImage": [101]
    },
    "orderHistory": {
        "rows": [
            { "product": "노트북", "qty": 1, "attachFile": [201] },
            { "product": "마우스", "qty": 2, "attachFile": [] }
        ]
    },
    "relatedTags": [3, 7, 12]
}
```

> - `userInfo` : Form 위젯의 contentKey
> - `orderHistory` : SubList 위젯의 contentKey
> - `relatedTags` : MultiSelect 위젯의 contentKey (ID 배열)

### 1-3. BE INSERT 후 id 자동 주입

```java
// PageDataService.create() — INSERT 완료 후 dataJson에 id 즉시 추가
Map<String, Object> dataJsonWithId = new LinkedHashMap<>(request.getDataJson());
dataJsonWithId.put("id", newId);
UPDATE page_data SET data_json = CAST(:dataJson AS jsonb) WHERE id = :id;
```

> **이유**: 카테고리 위젯처럼 자신의 id를 참조해야 하는 케이스 지원.  
> FE에서 id를 보내지 않아도 저장 후 항상 `data_json` 안에 id가 포함된다.

### 1-4. 데이터 형식 — 저장 구조와 accessor 사용법

#### 저장 구조

**1단계 — flat (contentKey 없음)**
```json
{ "name": "홍길동", "status": "active" }
```

**2단계 — contentKey 중첩**
```json
{ "userInfo": { "name": "홍길동", "status": "active" } }
```

---

#### 테이블 컬럼 accessor 사용법

| 단계 | 저장 구조 | accessor | 비고 |
|---|---|---|---|
| 1단계 | `{ "name": "홍길동" }` | `name` | flat 직접 접근 |
| 2단계 (contentKey 1개) | `{ "userInfo": { "name": "홍길동" } }` | `name` 또는 `userInfo.name` | 중복 없으면 `fieldKey`만으로 가능 |
| 2단계 (contentKey 2개 — 같은 fieldKey 중복) | `{ "userInfo": { "name": "A" }, "orgInfo": { "name": "B" } }` | `userInfo.name`, `orgInfo.name` | **중복 시 dot notation 필수** |

#### 핵심 규칙

```
1단계, 2단계 → fieldKey 단독 접근 가능 (flattenPageDataItem이 자동으로 root에 flat 병합)
              단, 동일 fieldKey가 여러 contentKey에 중복이면 → contentKey.fieldKey 필수
```

#### 단일 수정 모드 (`?id=123`)

#### 신규 모드 초기화

`widgetSub/[slug]/page.tsx`에서 `?id` 파라미터가 없는 신규 입력 모드로 진입하면  
Form 위젯의 모든 필드에 대해 기본값과 URL 파라미터를 아래 우선순위로 초기화한다.

```
우선순위 (높을수록 우선)
  1. URL 파라미터 (urlVal = searchParams.get(fieldKey))
  2. defaultOptionValue (select·radio·checkbox)
  3. defaultValueMsgKey (t()로 번역된 다국어 기본값)
  4. defaultValue (직접 텍스트 기본값)
```

```typescript
fw.fields.forEach(f => {
    const key = f.fieldKey || f.label || '';
    const urlVal = key ? searchParams.get(key) : null;

    if (f.type === 'date' && ...) {
        // date 처리
    } else if (f.type === 'dateRange') {
        // dateRange 처리
    } else if ((f.type === 'select' || f.type === 'radio' || f.type === 'checkbox') && f.defaultOptionValue) {
        vals[f.id] = urlVal ?? f.defaultOptionValue;   // URL 파라미터 우선
    } else if (f.defaultValueMsgKey) {
        vals[f.id] = urlVal ?? t(f.defaultValueMsgKey);
    } else if (f.defaultValue) {
        vals[f.id] = urlVal ?? f.defaultValue;
    }
});
```

> **활용 예시**: 카테고리 등록 팝업에서 `?depth=2&parentId=5`를 전달하면 fieldKey가 `depth`, `parentId`인 필드에 자동 세팅.  
> `select` 필드의 `defaultOptionValue`가 `"active"`로 설정되어 있으면 신규 입력 시 `"활성"`이 기본 선택된다.

**Search 위젯 기본값 초기화** (`widget/[slug]/page.tsx`, `widgetSub/[slug]/page.tsx` 공통)  
페이지 로딩 시 useEffect 1회 실행:

```typescript
// select·radio·checkbox
} else if ((f.type === 'select' || f.type === 'radio' || f.type === 'checkbox') && f.defaultOptionValue) {
    initSearchVals[f.id] = f.defaultOptionValue;
// input·hidden 등 직접 텍스트 기본값
```

### 2-3. 검색 흐름

```
[검색 버튼 클릭]
  handleSearch(searchWidgetId)
       │
       ├── searchValuesRef.current 참조 (stale closure 방지)
       └── Table 위젯 중 connectedSearchIds에 searchWidgetId 포함된 것 찾기
            └── fetchTableData({ ..., sv: 현재검색값, page: 0 })
                 └── GET /page-data/{slug}?필드키=값&page=0&size=10
```

#### 검색 파라미터 → BE 동적 쿼리 매핑

```
FE 파라미터 형식                        BE 쿼리 조건
──────────────────────────────────────────────────────────────────────────────────
name=홍길동                             data_json->>'name' ILIKE '%홍길동%'
eq_status=active                        data_json->>'status' = 'active'
fieldKey_from / fieldKey_to             data_json->>'fieldKey' 범위 비교 (dateRange 2컬럼)
fieldKey_gte / fieldKey_lte             data_json->>'fieldKey' 범위 비교 (singleDateRange)
createdAt_gte / createdAt_lte           created_at 실컬럼 범위 비교 — 상세는 00-3 DateRangeField 참고
```

> - 일반 파라미터 (`name=값`): ILIKE 부분 일치 + contentKey 1단계 중첩도 동시 검색  
> - `eq_` 접두사: 정확 일치 (카테고리 parentId 필터 등에 사용)  
> - `~` 구분자: 값 하나에 담긴 범위 검색 (숫자 범위 등, dateRange 필드는 `_from`/`_to`/`_gte`/`_lte` 사용)

#### 2-3-1. 조인 검색(join search) — 결과 테이블 슬러그에 없는 값으로 검색 (신규)

결과 테이블의 `connectedSlug`가 A인데, 검색값은 A에 없고 연결된(FETCH relation) B에만 있는 경우(예: 결과 테이블은 `category-data`인데 "제품명"은 `product-data`에만 있음) — 위 §2-3의 일반 파라미터/`eq_`로는 안 된다. `_fetchedRel{id}`는 **조회 이후에만** 채워지는 값이라 WHERE절에 못 쓰기 때문.

이 경우 Select/Input 필드에 `joinRelationSlugId`(FETCH relation) + `joinSlaveKey`(B 쪽 비교 필드 경로)를 설정한다. 상세 스펙은 `00-3.builder_field_reference.md` §1-1 참고. 파라미터는 `joinr_/joink_/joinv_` 3개, BE는 `resolveJoinFilterIds`(신규, `resolveFilterRelationIds`와 별개)가 처리 — B에서 조건에 맞는 레코드를 찾아 relation `slave_key`로 연결값을 뽑고, `master_key` 경로가 그 값과 일치하는 A의 행 id를 모아 **기존 `id IN (...)` 패턴 그대로 재사용**한다(§0-5 FILTER relation과 동일한 "축약 후 IN" 골격).

**제약**: relation의 `master_slug`가 검색 위젯이 속한 결과 테이블의 `connectedSlug`와 같아야 한다. 다르면 조용히 0건.

#### `parseActionParams` — Actions 컬럼 파라미터 → initialValues 변환 (WidgetRenderer.tsx)

테이블 Actions 컬럼의 수정 버튼에 **파라미터 문자열**을 설정하면,  
버튼 클릭 시 `parseActionParams`가 해당 row 데이터를 참조해 `initialValues`로 변환한다.

```
파라미터 형식: "title,temp1=1,temp2=abc"  (쉼표 구분)
  - =없음 (title)    → { title: row['title'] }   row의 해당 key 값
  - =있음 (temp1=1)  → { temp1: '1' }            고정값
```

**작은따옴표 리터럴 & 쉼표 보호**

```
파라미터 형식: "title='타이틀',content=title,type='Seoul, Korea'"
  - 'text' (따옴표로 감쌈) → 무조건 고정 텍스트(리터럴). row 필드명과 우연히 같아도 치환되지 않음
  - 따옴표로 감싼 값 안의 쉼표는 구분자로 취급되지 않음 (예: 'Seoul, Korea'는 하나의 값)
  - 따옴표 짝이 안 맞으면(예: O'Brien처럼 홀수 개) 자동으로 단순 쉼표 분리로 폴백 — 크래시 없이 안전 처리
```

**빌더 설정 위치**:  
`ActionsField.tsx` → 수정 버튼 `editPageRules`에서 각 규칙의 전달 파라미터(passParam) 입력  
`EditPageRule.passParam` 필드에 저장됨

#### `copyFixedParams` — Table row 복사 시 고정값 세팅 (WidgetRenderer.tsx `onCopy`)

Table Actions 컬럼에 복사(copy) 액션이 켜져 있으면, 행 복사 시 특정 필드를 원본 값 대신
지정한 값으로 덮어써서 저장할 수 있다. `parseActionParams`를 재사용하되, 신규 레코드
생성이 목적이라 팝업 initialValues가 아니라 dataJson에 직접 오버레이한다.

```

    → copyFixedParams 파싱 결과를 dataJson에 오버레이 (file 처리보다 나중 실행 → 최종 우선순위)
    → POST /page-data/{slug}  (기존 등록 API 재사용, 신규 레코드 생성)
```

> ⚠️ 동적 참조(예: `content=title`)의 대상 row는 반드시 `flattenPageDataItem`(§0-6)으로
> 평탄화한 것을 사용해야 한다 — 그래야 contentKey 중첩 필드(예: `form1.title`)도
> "title"이라는 단순 이름으로 참조 가능하다. 원본 dataJson(중첩 구조)을 그대로 넘기면 안 된다.

**빌더 설정 위치**:  
`ActionsField.tsx` → 복사 체크박스 선택 시 노출되는 입력창 → `TableColumnConfig.copyFixedParams` 필드에 저장됨

### 3-2. 파일 처리 2단계 구조


### 3-3. 유효성 검사 (`validateFormFields`)

**위치**: `utils.ts`  
**사용처**: `handleContentAction` (page 모드), `handlePopupContentAction` (popup 모드)

```typescript
// utils.ts — 각 조건 실패 시 toast.warning + false 반환 (첫 실패에서 즉시 중단)
validateFormFields(
    fields:           FormFieldItem[],
    values:           Record<string, string>,    // fieldId → 값
    fileValues:       Record<string, File[]>,    // fieldId → 신규 파일
    existingFileMeta: Record<string, any[]>,     // fieldId → 기존 파일 메타
    allValues?:       Record<string, string>,    // 페이지 내 전체 Form 값 (cross-form 조건 평가용)
    allKeyToId?:      Record<string, string>,    // 페이지 내 전체 fieldKey→id 역매핑
    t?:               (key: string) => string,   // 다국어 변환 함수 — labelMsgKey 해석용
)
```

| 검사 순서 | 검사 항목 | 조건 |
|---|---|---|
| 0 | hidden 타입 | 항상 건너뜀 |
| 1 | hideCondition | 조건 충족 시 해당 필드 건너뜀 (HIDE 상태이므로) |
| 2 | 필수 | `required=true` + 값이 비어있음 |
| 3 | 최소 길이 | `minLength` 미충족 |
| 4 | 최대 길이 | `maxLength` 초과 |
| 5 | 패턴 | `pattern` 정규식 불일치 |
| 6 | 파일 최대 개수 | `maxFileCount` 초과 (기존 + 신규 합산) |
| 7 | 파일 단건 용량 | `maxFileSizeMB` 초과 |
| 8 | 파일 전체 용량 | `maxTotalSizeMB` 초과 |
| 9 | 날짜 범위 | `dateRange` / `yearMonthRange` 종료일 < 시작일 |

#### hideCondition / disableCondition 평가 — `FormRenderer` 내 `evalCondition`

> **공용화 안내**: `FormRenderer.evalCondition`은 자체 로직이 아니라 `make/_shared/utils.ts`의 **`evalConditionExpr`(저수준 공용 함수)**에 위임한다. `SearchRenderer`의 hide/disable, `evalFieldCondition`(폼 검증 시 hidden 필드 스킵), `evalColumnDataExpr`(테이블/서브리스트 data 표현식, `02.input_builder_outputpanel.md` §4-11 참고)까지 전부 이 한 함수를 공유한다. **조건식 문법을 확장할 때는 `utils.ts`의 `evalConditionExpr`/`evalSingleCondition` 한 곳만 고치면 이 4곳 전부에 반영된다.**

```typescript
// FormRenderer.tsx evalCondition → utils.ts evalConditionExpr 위임
// 지원 연산자: =(문자열 동등) / !=(문자열 비동등) / < > <= >=(숫자 또는 날짜 비교)
// 지원 문법: 복수 AND 조건("k1=v1,k2=v2,..."), today() 함수토큰(날짜 포맷 자동추론)
// 참조 우선순위: 현재 Form 필드 > cross-form 필드 > URL 파라미터 > cross-tab 필드

// 예시
"status=1"                 → status 필드 값이 "1"이면 HIDE/DISABLE
"type!=Y"                  → type 필드 값이 "Y"가 아니면 HIDE/DISABLE
"status=1,type=Y"          → 두 조건 모두 충족 시 HIDE/DISABLE (AND)
"form1.status=1"           → 다른 Form(contentKey=form1)의 status 필드 값이 "1"이면 HIDE/DISABLE
"test=1"                   → URL 파라미터 ?test=1로 접근 시 HIDE/DISABLE (필드 없을 때 urlParams 폴백)
"date_from>today()"        → date_from 필드값이 오늘보다 미래면 HIDE/DISABLE (today()는 좌/우 어느 쪽에도 가능)
```

> ⚠️ **리터럴 vs 필드 참조**: `val` 자리는 따옴표가 없으면 먼저 필드/URL파라미터/cross-tab 순으로 조회하고, 어디에도 없으면 그 문자열 자체를 리터럴로 비교한다. 확실한 텍스트 비교가 필요하면 따옴표(`'`)로 감쌀 것(예: `name='John'`).
> ⚠️ **미로드 데이터 주의**: cross-tab 참조 조건(특히 `!=`)에서 상대 탭 데이터가 아직 로드되지 않은 순간에는, 그 key가 리터럴로 취급되어 일시적으로 다른 결과가 나올 수 있다. 실사용 중 이상 동작 시 우선 데이터 로딩 타이밍을 의심할 것.

**URL 파라미터 참조 동작 원리**

`key`에 해당하는 Form 필드(fieldId)가 없을 경우 `urlParams` 에서 직접 조회한다.

```
condition: "test=1"
  → resolvedKeyToId["test"] → undefined (해당 fieldKey 없음)
  → urlParams["test"] → "1" (URL ?test=1로 접근한 경우)
  → 조건 충족 → HIDE
```

`urlParams`는 `widgetSub/[slug]/page.tsx` → `PageGridRenderer` → `WidgetRenderer` → `FormRenderer` 순으로 전달된다.  
`id`, `group_id` 키는 제외되고 나머지 URL 파라미터만 포함된다.

> **cross-form hideCondition**: 동일 페이지 내 다른 Form 필드 참조 시  
> `contentKey.fieldKey` 형식 사용. (예: `personal.status=Y`)  
> 단, contentKey에 `.`(점) 포함 불가 (FormBuilder contentKey 입력 시 자동 차단됨)

### 3-6. `buildDataJson` 공통 함수

**위치**: `utils.ts`  
**사용처**: `handleContentAction` (page 모드), `handlePopupContentAction` (popup 모드)

#### 함수 시그니처

```typescript
buildDataJson(
    widgets:        Array<{ type, widgetId?, fields?, contentKey? }>,
    formValuesMap:  Record<widgetId, Record<fieldId, 값>>,
    formFileIdsMap: Record<widgetId, Record<fieldId, number[]>>,  // 기존+신규 합산 완료
    subListRowsMap: Record<widgetId, 처리된 행[]>,               // _rowId 제거, 파일ID 완성
    multiSelectMap: Record<widgetId, number[]>,
): { dataJson: Record<string, unknown>, pkKeys: string[] }
```

#### contentKey 처리 규칙

```typescript
// Form 위젯
if (fw.contentKey) dataJson[fw.contentKey] = section;  // 중첩 저장
else Object.assign(dataJson, section);                   // flat 저장

// SubList 위젯
if (sw.contentKey) dataJson[sw.contentKey] = { rows };
else dataJson.rows = rows;

// MultiSelect 위젯
if (mw.contentKey) dataJson[mw.contentKey] = selectedIds;
```

#### 호출 전 사전 처리 순서 (필수)

```
1. Form 파일 필드 업로드
   fileValuesMap[widgetId][fieldId] 의 신규 File[] 를 업로드
   → formFileIdsMap[widgetId][fieldId] = [...기존MetaId[], ...신규업로드Id[]]

2. SubList 파일 컬럼 업로드 + rows 처리
   각 row의 파일 컬럼 신규 File[] 를 업로드
   _rowId 제거 + 파일 컬럼 ID 배열 완성
   → processedSubListRowsMap[widgetId] = processedRows[]

3. buildDataJson 호출
   → { dataJson, pkKeys }
```

> **왜 순서가 중요한가**: `buildDataJson`은 순수 동기 함수다.  
> 비동기 파일 업로드가 완료된 후에 호출해야 파일 ID가 dataJson에 포함된다.  
> SubList 파일 처리 전에 호출하면 해당 파일 컬럼이 빈 배열로 저장된다.

### 3-8. 검증 규칙(ValidationRule) — 저장 시점 unique/maxCount 체크

action-button(connType `datasave`/`content`)에 다중선택된 검증 규칙이 실제 저장(create/update) 시점에 BE에서 수행된다.

```
[빌더] "규칙생성" 버튼 → RuleCreateModal → slug(slugRegistryId) 선택 → 규칙 CRUD
   (/api/v1/validation-rules, §6-3)
        │
[빌더] action-button 필드 편집 → ValidationRuleMultiSelect에서 규칙 다중선택
   datasave  → FieldEditValues.validationRuleIds:        number[]
   content   → FieldEditValues.contentValidationRuleIds:  Record<위젯ID, number[]>
        │
[live] 저장 실행 → PageGridRenderer → SpaceRenderer → useWidgetPageState
   → POST/PUT /page-data/{slug} body에 validationRuleIds 포함
        │
[BE] PageDataService.create()/update() → checkValidationRules()
   → 규칙 type별로 checkUniqueRule() / checkMaxCountRule() 수행
```

**규칙 타입**

| type | 필수 값 | 동작 | 위반 시 |
|---|---|---|---|
| `unique` | `fields` (콤마구분 복합키) | `fields` 값 조합이 같은 data_slug 내 유일해야 함 | `VALIDATION_RULE_UNIQUE_VIOLATION` (409) |
| `maxCount` | `maxCount` | `condition`을 만족하는 데이터가 maxCount를 넘으면 거부 | `VALIDATION_RULE_MAX_COUNT_EXCEEDED` (409) |

둘 다 `condition`(evalConditionExpr 문법 재사용 — 콤마구분 AND, `=`/`!=`/`<`/`>`/`<=`/`>=` 연산자 + `today()` 함수토큰)으로 대상 범위를 한정할 수 있다 (선택).
예: `bannerPosition='INFORMATION',postDate_from<today(),postDate_to>today()` — "오늘 기준 노출중"으로 범위를 좁힘.

**`fields`/`condition` key 표기법** — 실제 저장 구조(§1-1, §1-2)와 반드시 일치해야 한다.
- `title` → 최상위(flat) 저장 필드 (`dataJson.title`)
- `form1.title` → 컨텐츠 위젯의 `contentKey` 내부 필드 (`dataJson.form1.title`, dot 1개까지만 지원)
- dot 없는 키는 항상 최상위만 조회한다. Form 위젯에 `contentKey`가 설정되어 있으면(컨텐츠 위젯은 대부분 필수) 그 안의 필드는 반드시 `contentKey.fieldKey`로 지정해야 조회된다.

**⚠️ maxCount + condition 조합 시 주의** — 지금 저장하려는 데이터 자신이 `condition`을 만족할 때만 건수 제한이 적용된다. 조건과 무관한 값을 저장할 때는 기존 건수와 무관하게 항상 허용해야 한다 (무조건 `기존건수 + 1`로 판정하면 안 됨).

**site_id 스코프** — 검증 규칙 체크는 검색(§2-3)과 동일하게 `(site_id = 현재사이트 OR site_id IS NULL)` 범위에서만 카운트한다. 다른 사이트에 속한 데이터는 검증 대상에서 제외된다. (create는 항상 siteId 사용 가능, update는 컨트롤러가 `X-Site-Id` 헤더를 받아 전달해야 함)

**⚠️ unique 규칙의 범위(scope) 설계 주의** — `fields`에 넣는 필드 조합이 "무엇을 기준으로 유일해야 하는지"를 정확히 결정한다. 예를 들어 "같은 부모 카테고리 안에서만 제품이 유일해야 한다"(다른 카테고리에는 같은 제품이 있어도 됨)는 요구사항을 `fields=product.id`(단일 필드)로 만들면 **전체 범위에서 유일**을 강제해버려서 의도보다 훨씬 넓게 막힌다. 이 경우 `fields=product.id,product.parentId`(복합키)로 "제품id + 부모카테고리id 조합"이 유일하도록 좁혀야 의도한 정책이 된다. 규칙을 설계할 때 "무엇의 조합이 중복되면 안 되는가"를 먼저 명확히 하고 `fields`를 정할 것.

### 3-9. inlineEdit 셀 즉시 저장 — `patchField` 단일 필드 부분수정 (신규)

Table 위젯의 `cellType: "inlineEdit"` 컬럼(toggle/radio/checkbox)은 셀 클릭 시 전체 폼 저장 없이 **필드 하나만** 즉시 저장한다.

```
[live] 셀 클릭 → handlers.onInlineEdit(col, row, newValue)
   │
   ├─ col.inlineEditRelationSlugId 없음 → 기존 동작: PATCH /page-data/{connectedSlug}/{row._id}/field
   │                                        body: { fieldKey: col.inlineEditFieldKey, value: newValue }
   │
   └─ col.inlineEditRelationSlugId 있음(저장 대상 리다이렉트) →
        1. relation.masterKey 경로로 현재 행(row)에서 값을 읽어 slave 레코드 id로 사용
           (relation은 FETCH 타입, resolveAccessor(row, relation.masterKey) — 이 relation의 slave가
            실제 저장 대상. row 자신의 connectedSlug가 아니라 relation의 slaveSlug로 PATCH가 나간다)
        2. PATCH /page-data/{relation.slaveSlug}/{slaveId}/field
        3. 성공 시에만 로컬 테이블 상태(col.accessor)를 갱신 — 실패 시 상태 변경 자체가 없어
           별도 롤백 로직이 필요 없음(낙관적 업데이트를 안 하는 구조)
```

**값 매핑**: `inlineEditTrueValue`/`inlineEditFalseValue`(toggle 전용)를 설정하면 `value===trueValue`로 on/off를 판정하고 저장값도 이 매핑값을 사용한다. 미설정 시 `Boolean(value)` 폴백(하위호환).

**`patchField`의 dot notation(중첩 경로) 지원** — `PageDataService.patchField()`는 `fieldKey`에 점(`.`)이 있으면 그 경로를 그대로 중첩 Map으로 타고 들어가 값을 쓴다(중간 경로가 없으면 새로 생성). 점이 없으면 기존과 동일하게 최상위 키에 쓴다.
```java
// fieldKey = "productDataForm.trainingYn" → dataJson.productDataForm.trainingYn 에 저장
// fieldKey = "trainingYn"                 → dataJson.trainingYn (최상위) 에 저장 — 기존 동작 그대로
```
⚠️ 이 지원이 없던 시절에는 점 포함 fieldKey를 보내면 `"productDataForm.trainingYn"`이라는 **문자 그대로의 이상한 최상위 키**가 새로 생기기만 하고 실제 중첩값은 안 바뀌는 버그가 있었다(새로고침하면 원래대로 복구되는 것처럼 보임). `inlineEditFieldKey`에 점 표기 경로를 쓸 때는 이 지원이 반영된 BE인지 반드시 확인할 것.

### 4-4. 테이블 row 변환 — `flattenPageDataItem` 공통 함수 (⭐ 필독)

**위치**: `utils.ts` → `export function flattenPageDataItem(item)`  
**사용처**: `useWidgetPageState.ts`, `/admin/widget/[slug]/page.tsx`, `/admin/widgetSub/[slug]/page.tsx`, SelectField/MultiSelectRenderer의 SLUG 옵션 로드  
공통 사용처 모두 `.map(flattenPageDataItem)` 한 줄로 처리. **절대 복붙 금지.**

#### accessor 처리 규칙

| 단계 | dataJson 저장 구조 | accessor 사용법 |
|---|---|---|
| **1단계 flat** | `{ title: "A", use: "1" }` | `title`, `use` |
| **2단계 contentKey** | `{ form1: { title: "A", use: "1" } }` | `title`, `use` (중복 없으면 자동 root 병합) |
| **2단계 contentKey 중복** | `{ form1: { title: "A" }, form2: { title: "B" } }` | `form1.title`, `form2.title` (중복 → dot notation 필수) |

#### 핵심 로직

```typescript
// utils.ts — flattenPageDataItem
export function flattenPageDataItem(item) {
    /* dataJson.id를 제거하지 않고 유지 — params에서 id로 접근 가능하도록 */
    const restDataJson = item.dataJson ?? {};

    // 최상위 object 값(contentKey 섹션) 감지
    const sectionEntries = Object.entries(restDataJson)
        .filter(([, v]) => v !== null && typeof v === 'object' && !Array.isArray(v));

    // 중복 없는 fieldKey만 root에 flat 병합 (2단계 대응) + 정렬용 경로 맵 생성
    const flatExtra: Record<string, unknown> = {};
    const _pathMap: Record<string, string> = {};   // fieldKey → "sectionKey.fieldKey" (정렬 파라미터 변환용)
    if (sectionEntries.length > 0) {
        const keyCount: Record<string, number> = {};
        sectionEntries.forEach(([, s]) =>
            Object.keys(s).forEach(k => { keyCount[k] = (keyCount[k] ?? 0) + 1; })
        );
        sectionEntries.forEach(([sectionKey, s]) => {
            // dot notation accessor 키를 재귀적으로 root에 추가 (예: "form1.title", "tab1.form1.title")
            addDotNotationKeys(flatExtra, s, sectionKey);
            Object.entries(s).forEach(([k, v]) => {
                if (keyCount[k] === 1) { flatExtra[k] = v; _pathMap[k] = `${sectionKey}.${k}`; }  // 중복 없는 것만
            });
        });

        // _fetchedRel{id} 섹션 전용 단축키: "_fetchedRel8.currMgmtForm.currMgmtTitle" → "_fetchedRel8.currMgmtTitle"
        // (접두사 _fetchedRel{id}는 연동 slug 출처 식별자라 생략하지 않음, 동일 섹션 내 leaf명 중복 시 생성 안 함)
    }

    return {
        _id: item.id,
        _groupId: item.groupId ?? null,
        _pathMap,           // fieldKey → sectionKey.fieldKey 경로 맵 (정렬용)
        ...flatExtra,       // 중복 없는 fieldKey flat (단순 accessor용) + dot notation 키
        ...restDataJson,    // 원본 중첩 구조 보존 (dot notation accessor용)
        createdAt: item.createdAt ?? null,
        createdBy: item.createdBy ?? null,
        updatedAt: item.updatedAt ?? null,
        updatedBy: item.updatedBy ?? null,
    };
}
```

> 실제 구현은 위 흐름에 `addDotNotationKeys`(재귀적 dot notation 키 생성)와 `_fetchedRel{id}` 섹션 단축키 생성 로직이 추가되어 있다 (utils.ts L581~659 참고). 핵심 flat 병합 원칙(1단계/2단계 accessor 규칙)은 동일하다.

#### 동작 예시

```
// 케이스 1: flat (contentKey 없음)
dataJson = { title: "A" }
→ row = { _id:1, title:"A", createdAt:... }
→ accessor "title" ✅

// 케이스 2: contentKey 1개 (중복 없음)
dataJson = { form1: { title: "A", use: "1" } }
→ row = { _id:1, title:"A", use:"1", form1:{ title:"A", use:"1" }, ... }
→ accessor "title" ✅ / "form1.title" ✅

// 케이스 2: contentKey 2개 (fieldKey 중복)
dataJson = { form1: { title: "A" }, form2: { title: "B" } }
→ row = { _id:1, form1:{title:"A"}, form2:{title:"B"}, ... }  // title은 root 미병합
→ accessor "form1.title" ✅ / "form2.title" ✅ / "title" ❌
```

#### TableCellRenderer dot notation 지원 (TableCellRenderer.tsx)

```typescript
const value = col.accessor.includes('.')
    ? col.accessor.split('.').reduce((acc, key) => {
        if (acc && typeof acc === 'object' && !Array.isArray(acc))
            return acc[key];
        return undefined;
    }, row)

```

> `.` 구분자로 중첩 경로 탐색 — `contentKey.fieldKey` dot notation 자동 지원.

### 4-5. 카테고리 위젯 조회 방식 — dot notation 필드 읽기

테이블과 카테고리는 contentKey 중첩 구조를 처리하는 방식이 다르다.

```
테이블 (fetchTableData):
  dataJson의 모든 object wrapper를 자동으로 펼침 (flat-map)
  → { lv1pop: { title: 'A' } } → flat에 title 직접 추가

카테고리 (fetchItems):
  readField() 유틸로 dot notation 경로를 명시적으로 지정
  → widget.fieldTitle = 'lv1pop.title' 이면 dataJson['lv1pop']['title'] 접근
```

**`readField()` 동작 원리** (`CategoryRenderer.tsx` 내부):

```typescript
const readField = (dataJson: Record<string, unknown>, fieldKey: string): unknown => {
    const dotIdx = fieldKey.indexOf('.');
    if (dotIdx !== -1) {
        const parentKey = fieldKey.slice(0, dotIdx);   // 'lv1pop'
        const childKey  = fieldKey.slice(dotIdx + 1);  // 'title'
        const section   = dataJson[parentKey];
        if (section && typeof section === 'object' && !Array.isArray(section)) {
            return (section as Record<string, unknown>)[childKey];
        }
        return undefined;
    }
    return dataJson[fieldKey];  // dot 없으면 root 직접 접근
};
```

**동작 예시**:

```
dataJson = { "lv1pop": { "title": "대분류A", "code": "A-001" } }

readField(dataJson, 'lv1pop.title')  → '대분류A'   ✓
readField(dataJson, 'lv1pop.code')   → 'A-001'     ✓
readField(dataJson, 'title')         → undefined    (root에 없음)
```

> 카테고리 위젯 빌더에서 카드 필드 키를 `폼key.컬럼명` 형식으로 설정하는 이유가 이것이다.  
> 설정값이 연결 폼의 contentKey와 다르면 해당 필드는 빈값으로 표시된다.

### 5-2. ref 경유 패턴 — 언제 사용하는가


| ref | 경유 이유 |
|---|---|
| `searchValuesRef` | `handleSearch` / `handleReset` 의존성에서 `searchValues` 제외하기 위함 |
| `tableDataMapRef` | `handleLoadMore` 의존성에서 `tableDataMap` 제외하기 위함 |

```typescript
// ✅ ref 경유 — 항상 최신 검색값 참조
const handleSearch = useCallback((searchWidgetId: string) => {
    const sv = searchValuesRef.current; // 클로저 시점 고정 없음

}, [widgetItems, ...]);

    const sv = searchValues; // widgetItems 변경 시 stale
    ...
}, [widgetItems, searchValues]); // searchValues를 의존성에 추가하면 무한 재생성 위험
```

### 금지 9: keep-alive 탭에서 radio name 미고유화

탭 위젯은 keep-alive 패턴(탭 전환 시 `display: none` 처리, 언마운트 안 함)을 사용한다.  
이 때 모든 탭의 DOM이 동시에 존재하므로 같은 `name` 속성의 라디오 버튼이 중복되어 클릭이 정상 동작하지 않는다.


### 9-1-1. relation JOIN + 다운로드 방식(데이터테이블/사용자정의)

엑셀 버튼(`connType='excel'`) 설정에 다음 2가지가 추가되어 있다.

**① relation JOIN 다중 선택** (`excelRelationIds: number[]`)
- 선택한 테이블의 연결 Slug(masterSlug)를 기준으로 하는 FETCH 타입 relation만 체크박스로 노출 (TABLE/ARRAY_CONTAINS 유형만 대상, **CATEGORY 유형은 미지원**)
- export 시 `relationIds`(콤마구분) 파라미터로 전달 → BE `PageDataService.applyFetchBatch()`가 relation당 **배치 IN 조회 1건**으로 slave 데이터를 조회해 `_fetchedRel{id}` 키로 병합 (기존 search/getById용 `applyFetch`의 행당 쿼리 N+1 방식과 별개 — export 전용으로 신설)
- `ExcelService.getNestedValue()`는 화면 표시용 축약 accessor(`_fetchedRel8.currMgmtTitle`)와 전체 경로(`_fetchedRel8.currMgmtForm.currMgmtTitle`) **둘 다** 해석 가능 — 전체 경로 탐색 실패 시 해당 `_fetchedRel{id}` 섹션 내부를 재귀 스캔해 leaf 키가 유일하면 그 값을 반환하는 폴백이 있음

**② 다운로드 방식 라디오** (`excelDownloadMode: 'current' | 'custom'`, 기본값 `'current'`)

| 값 | 동작 |
|---|---|
| `'current'`(데이터테이블, 기본값) | 기존과 동일 — 테이블 컬럼(`validCols`) 그대로 export, codeMaps도 컬럼 기준 정상 생성 |
| `'custom'`(사용자정의) | **테이블 컬럼을 완전히 무시**하고 `excelExtraColumns`(컬럼 목록)에 정의한 것만 export 전체를 구성(추가/append 아님, 대체/replace). codeMaps는 항상 빈 객체(사용자정의 컬럼에는 codeGroupCode 개념이 없음). **컬럼 목록이 비어있으면 컬럼 0개짜리 빈 파일이 그대로 나온다**(테이블 컬럼 폴백 없음 — 의도된 동작) |

`WidgetRenderer.doExcelDownload()`는 `extraColumns`가 `undefined`인지(→ 데이터테이블 모드) 아니면 배열(빈 배열 포함, → 사용자정의 모드)인지로 두 모드를 구분한다. `SpaceRenderer.tsx`가 `isExcelCustom ? (field.excelExtraColumns ?? []) : undefined`로 이 구분을 만들어 전달하므로, "사용자정의 모드+컬럼 0개"와 "데이터테이블 모드"가 둘 다 `undefined`로 뭉개지지 않도록 반드시 이 패턴을 유지해야 한다.

### 9-2. tableWidgetsMap 구성 방식

`WidgetRenderer`는 테이블 위젯 정보를 직접 조회하지 않는다.
`PageGridRenderer`가 `widgetItems`에서 수집해 `tableWidgetsMap` prop으로 주입한다.

```typescript
// PageGridRenderer.tsx — useMemo, widgetItems 변경 시 자동 재구성
const tableWidgetsMap = useMemo(() => {
    const map: Record<string, TableWidget> = {};
    widgetItems.flatMap(item => item.contents.map(c => c.widget))
        .filter((w): w is TableWidget => w.type === 'table')
        .forEach(w => { map[w.widgetId] = w; });
    return map;
}, [widgetItems]);
```

> **금지**: WidgetRenderer 내부에서 API를 호출해 테이블 위젯 정보를 직접 조회하는 것은 금지.
> 반드시 `tableWidgetsMap` prop 경유.

### 9-3. ⚠️ 정책 — 반드시 지켜야 할 규칙

#### 정책 1 — 헤더는 `headerMsgKey` 우선

```typescript
// ✅ 올바른 방법
const headers = validCols.map(c => c.headerMsgKey ? t(c.headerMsgKey) : c.header).join(',');

// ❌ 금지
const headers = validCols.map(c => c.header).join(',');
```

**이유**: 빌더에서 다국어 키(`headerMsgKey`)만 지정하고 `header`를 빈 문자열로 두는 패턴이 일반적이다.
이런 컬럼이 목록 뒤쪽에 있으면 join 결과에 trailing 쉼표가 생기고,
BE `split(",")` 기본 동작이 trailing 빈 문자열을 제거해 헤더 수가 줄어든다.
예: `"이름,이메일,,"` → Java `split(",")` → `["이름", "이메일"]` (2개만 남음)

---

#### 정책 2 — BE `split(",", -1)` 필수

```java
// ✅ 올바른 방법
List<String> headerList = Arrays.asList(headers.split(",", -1));

// ❌ 금지
List<String> headerList = Arrays.asList(headers.split(","));
```

**이유**: Java `split(regex)` 기본 동작은 trailing 빈 문자열을 제거한다.
`limit=-1`로 호출해야 빈 문자열도 보존된다.
FE 정책 1을 지켜도 안전장치로 반드시 적용한다.

---

#### 정책 3 — 셀 값은 `getNestedValue()` 경유 필수

```java
// ✅ 올바른 방법
Object value = getNestedValue(rowData, keys.get(colIdx));

// ❌ 금지
Object value = rowData.get(keys.get(colIdx));
```

**이유**: `page_data.data_json`은 `contentKey.fieldKey` 같은 중첩 구조로 저장된다.
테이블 컬럼 accessor가 `"form1.제목"` 형태일 때 `rowData.get("form1.제목")`은
항상 null이다 — Map의 키는 dot을 포함한 문자열로 저장되지 않는다.

`getNestedValue()` 동작:
```
getNestedValue(row, "제목")              → row.get("제목")                    [1단계, flat]
getNestedValue(row, "form1.제목")        → row → "form1" → "제목"             [2단계]
```

---

#### 정책 5 — 날짜 컬럼 포맷은 `dateFormats` 파라미터로 전달

테이블 컬럼에 `dateFormat`(예: `YYYY-MM-DD`, `YYYY/MM/DD HH:mm`)이 설정된 경우 화면 표시와 동일한 포맷이 엑셀에도 적용된다.

FE는 `keys`와 동일한 순서로 각 컬럼의 `dateFormat` 값을 `dateFormats` 파라미터에 포함해 전달한다. 포맷이 없는 컬럼은 빈 문자열로 채워 **순서를 반드시 유지**해야 한다. 순서가 어긋나면 엉뚱한 컬럼에 날짜 포맷이 적용된다.

BE `ExcelService`는 ISO 8601 형식 값을 파싱하여 `YYYY`, `MM`, `DD`, `HH`, `mm`, `ss` 토큰을 치환한다. 파싱에 실패하면 원본값을 그대로 출력한다.

`createdAt`, `updatedAt` 등 메타 필드도 accessor로 사용 가능하므로, 날짜 포맷이 필요하면 테이블 컬럼 빌더에서 해당 컬럼의 `dateFormat`을 설정하면 된다.

---

#### 정책 6 — 공통코드 컬럼은 `codeMaps` 파라미터로 라벨 전달

```typescript
// ✅ 올바른 방법 — FE가 화면과 동일한 규칙(패턴 C)으로 라벨을 만들어 전달
const codeMaps: Record<string, Record<string, string>> = {};
validCols.forEach(c => {
    if (!c.codeGroupCode || c.displayAs === 'value') return;
    const details = codeGroups.find(g => g.groupCode === c.codeGroupCode)?.details ?? [];
    codeMaps[c.accessor] = Object.fromEntries(details.map(d => [d.code, codeDetailToLabel(d, t)]));
});
```

**이유**: BE는 `page_data.data_json`의 raw 코드값만 갖고 있어 사람이 읽는 라벨을 모른다. 화면(`TableCellRenderer`)이 이미 갖고 있는 `codeGroups` + 다국어 규칙(`03.builder_i18n_guide.md` 패턴 C)을 그대로 재사용해 FE가 라벨을 만들고, BE(`ExcelService.applyCodeMap()`)는 콤마 다중값 분할 후 기계적으로 치환만 한다 — 헤더의 `headerMsgKey` 처리와 동일한 구조다.

---

#### 정책 7 — export 전용 신규 파라미터는 반드시 `reservedForExport`에 등록

`PageDataService.exportAll()`은 요청 파라미터 중 `RESERVED_PARAMS`(`page`/`size`/`sort`)와 `reservedForExport`(`format`/`headers`/`keys`/`dateFormats`/`codeMaps`/`reason`)에 없는 나머지 전부를 **검색조건으로 취급**해 JSONB `WHERE`절을 동적 생성한다.

```java
// ✅ export 전용 파라미터를 새로 추가할 때 반드시 여기에도 등록
reservedForExport.addAll(Set.of("format", "headers", "keys", "dateFormats", "codeMaps", "reason"));
```

**이유**: 등록을 누락하면 그 파라미터가 존재하지도 않는 `data_json` 필드에 대한 검색조건으로 오인되어, 실제 데이터와 무관하게 결과가 0건이 될 수 있다. export API에 새 파라미터를 추가할 때마다 이 목록을 함께 갱신해야 한다.

#### 정책 4 — 컬럼 너비는 `measureLen()` 기반, `autoSizeColumn()` 사용 금지

```java
// ✅ 올바른 방법
sheet.setColumnWidth(i, Math.min(Math.max(colMaxLen[i] + 2, 12), 50) * 256);

// ❌ 금지
sheet.autoSizeColumn(i);
```

**이유**: Apache POI `autoSizeColumn()`은 한글 폰트 너비를 영문 기준으로 과소 측정해
한글이 포함된 컬럼이 절반 이하 너비로 잘려 보인다.

`measureLen()` 규칙:
- 한글 완성형(AC00~D7A3), 자모(1100~11FF, 3130~318F): 2자
- 영문·숫자·기타: 1자

`colMaxLen[]` 추적:
- 헤더 행 작성 시 각 컬럼 초기화
- 데이터 행 작성 시 최대값으로 업데이트 (행 작성과 동시에 추적)

너비 범위: 최소 12자 ~ 최대 50자 (여유 2자 포함)

### 9-5. API 연동(`connType='api'`) 파일다운로드

Space 액션버튼의 API 연동(mode2 — `apiInfoId` 선택 시)에 **"파일다운로드"** 체크박스(`apiDownloadFile: boolean`)가 있다. mode1(apiInfoId 미선택, entity 자동 CRUD)에는 파일 개념이 없어 노출되지 않는다.

```
[Space 버튼 클릭 — connType='api', apiDownloadFile=true]
     │
     ▼
SpaceRenderer.handleButtonClick → onApiCall(apiInfoId, params, connectedContentWidgetIds, downloadFile=true)
     │
     ▼
useWidgetPageState.handleApiCall()
  downloadFile === true
    → responseType:'blob'로 요청 (GET/DELETE는 params, POST/PUT/PATCH는 body 양쪽 다 적용)
    → 성공 시 토스트 대신 다운로드 트리거(doExcelDownload와 동일한 URL.createObjectURL + <a>.click() 패턴)
    → 파일명: 응답 Content-Disposition 헤더 파싱(parseContentDispositionFilename) 우선,
      없으면 `${apiInfo.name || 'download'}.xlsx` 폴백
  downloadFile !== true → 기존과 동일 JSON 요청 + 성공 토스트
```

**⚠️ 알려진 제한사항 (비차단, 후속 개선 후보)**
- blob 응답으로 실패(4xx/5xx)한 경우 서버가 body에 담아 보낸 실제 에러 메시지를 읽지 못하고 항상 고정 문구("요청을 실행할 수 없습니다")만 표시된다. 원인 파악이 필요하면 `err.response.data`(Blob)를 `.text()` + JSON 파싱하는 보강이 필요하다.
- Content-Disposition 헤더가 없는 API의 폴백 확장자가 `.xlsx`로 고정되어 있다. 이 체크박스는 엑셀 전용이 아닌 범용 "파일다운로드" 기능이므로, pdf/csv/zip 등을 반환하는 API에 연결하면 실제 내용과 무관하게 파일명만 `.xlsx`로 저장된다.
