## 0. ⭐ 핵심 아키텍처 원칙 — 반드시 숙지

### 0-1. slug = 데이터 네임스페이스

`page_data` 테이블의 모든 데이터는 `template_slug` 컬럼으로 격리된다.  
하나의 운영 페이지에서 여러 위젯이 각자 다른 `connectedSlug`를 가질 수 있다.

```
/admin/widgetSub/user-form
  └── Form 위젯 (connectedSlug: 'user-info')     → page_data WHERE template_slug = 'user-info'
  └── SubList 위젯 (connectedSlug: 'user-roles') → page_data WHERE template_slug = 'user-roles'
  └── Table 위젯 (connectedSlug: 'user-list')    → page_data WHERE template_slug = 'user-list'
```

> **slug 없이는 어떤 저장도 불가능하다.** `connectedSlug`가 비어있으면 해당 위젯의 저장 로직은 전부 건너뜀.

### 0-2. widgetId = FE 상태 키

동일 타입의 위젯이 한 페이지에 여러 개 있어도 `widgetId`로 구분한다.  
FE의 모든 상태 맵(formValuesMap, tableDataMap, …)은 widgetId를 키로 사용한다.

```typescript
formValuesMap[widgetId]        // 해당 Form 위젯의 입력값
tableDataMap[widgetId]         // 해당 Table 위젯의 데이터
subListRowsMap[widgetId]       // 해당 SubList 위젯의 행 목록
```

### 0-3. contentKey = dataJson 내부 섹션 구분자

하나의 `page_data` 레코드에 여러 위젯 데이터를 통합 저장할 때 `contentKey`로 섹션을 분리한다.

```json
{
    "id": 1,
    "userInfo": { "name": "홍길동", "email": "hong@test.com" },
    "orderHistory": { "rows": [{ "product": "노트북", "qty": 1 }] },
    "relatedTags": [3, 7, 12]
}
```

| contentKey 유무 | dataJson 저장 방식 |
|---|---|
| `contentKey` 없음 | 필드값을 최상위에 flat하게 저장 |
| `contentKey` 있음 | `{ [contentKey]: { 필드값들 } }` 중첩 구조로 저장 |

### 0-4. groupId = 다중 slug 묶음 키

한 페이지에서 서로 다른 `connectedSlug`에 동시 저장할 때 FE에서 UUID를 생성하여 모든 레코드에 동일한 `groupId`를 심는다.  
수정/삭제 시 `groupId`만 알면 연관 레코드 전체를 한 번에 처리할 수 있다.

```
신규 저장 (다중 slug):
  crypto.randomUUID() → groupId 생성
  POST /page-data/user-info    { dataJson, groupId: "uuid-abc" }
  POST /page-data/user-roles   { dataJson, groupId: "uuid-abc" }
  POST /page-data/user-address { dataJson, groupId: "uuid-abc" }

수정 모드 진입:
  URL: ?group_id=uuid-abc
  GET /page-data/user-info/group/uuid-abc
  GET /page-data/user-roles/group/uuid-abc
  GET /page-data/user-address/group/uuid-abc

삭제:
  DELETE /page-data/user-info/group/uuid-abc  (← 첫 번째 slug만 호출, BE에서 전체 삭제)
```

## 1. 데이터 모델 (dataJson 구조)

### 1-1. 단순 구조 (Form 단일, contentKey 없음)

```json
{
    "id": 1,
    "name": "홍길동",
    "email": "hong@test.com",
    "status": "active",
    "profileImage": [101, 102],
    "createdAt": "2026-05-27T10:00:00"
}
```

> - `id`는 BE에서 INSERT 후 자동 주입 (FE에서 보내지 않아도 됨)
> - 파일 필드(`file` / `image`)는 파일 ID 숫자 배열로 저장
> - `createdAt` 등 감사 컬럼은 테이블 조회 시 FE에서 flat하게 병합

### 1-2. 중첩 구조 (Form + SubList, contentKey 있음)

```json
{
    "id": 1,
    "userInfo": {
        "name": "홍길동",
        "email": "hong@test.com",
        "profileImage": [101]
    },
    "orderHistory": {
        "rows": [
            { "product": "노트북", "qty": 1, "attachFile": [201] },
            { "product": "마우스", "qty": 2, "attachFile": [] }
        ]
    },
    "relatedTags": [3, 7, 12]
}
```

> - `userInfo` : Form 위젯의 contentKey
> - `orderHistory` : SubList 위젯의 contentKey
> - `relatedTags` : MultiSelect 위젯의 contentKey (ID 배열)

### 1-4. 탭 위젯 중첩 구조 (2단계 contentKey)

탭(Tab) 위젯은 `TabItem.contentKey` → `FormWidget.contentKey` 2단계 중첩으로 `dataJson`을 저장한다.

```json
{
    "tab1": {
        "form1": {
            "title": "홍길동",
            "email": "hong@test.com",
            "profileImage": [101]
        }
    },
    "tab2": {
        "form2": {
            "content": "상세 내용"
        }
    }
}
```

| 레벨 | 키 출처 | 예시 |
|---|---|---|
| 1단계 | `TabItem.contentKey` | `"tab1"`, `"tab2"` |
| 2단계 | `FormWidget.contentKey` | `"form1"`, `"form2"` |
| 필드값 | `FormFieldItem.fieldKey` | `"title"`, `"email"` |

#### 저장 시 구성 순서

```
buildDataJson 호출 시 각 폼의 contentKey 처리:
  tabSection[form.contentKey] = { fieldKey: 값 }  ← Form 데이터
  dataJson[tab.contentKey]    = tabSection         ← 탭 데이터
```

#### 수정 시 복원 순서

```typescript
// dataJson 예시: { "tab1": { "form1": { "title": "홍길동" } } }

// STEP 1: 탭 섹션 추출
const tabSection = dataJson[tab.contentKey] as Record<string, unknown>;
// tabSection = { "form1": { "title": "홍길동" } }

// STEP 2: 폼 섹션 추출
const section = tabSection[form.contentKey] as Record<string, unknown>;
// section = { "title": "홍길동" }

// STEP 3: 필드값 매핑
vals[field.id] = String(section[field.fieldKey] ?? '');
```

> **tabContentKey 없는 탭**: `TabItem.contentKey`가 없으면 dataJson 최상위에서 읽는다.  
> **formContentKey 없는 폼**: `FormWidget.contentKey`가 없으면 tabSection 최상위에서 읽는다.  
> 두 값 모두 없으면 일반 단일 폼 저장과 동일하게 dataJson 최상위에서 읽는다.

### 1-3. BE INSERT 후 id 자동 주입

```java
// PageDataService.create() — INSERT 완료 후 dataJson에 id 즉시 추가
Map<String, Object> dataJsonWithId = new LinkedHashMap<>(request.getDataJson());
dataJsonWithId.put("id", newId);
UPDATE page_data SET data_json = CAST(:dataJson AS jsonb) WHERE id = :id;
```

> **이유**: 카테고리 위젯처럼 자신의 id를 참조해야 하는 케이스 지원.  
> FE에서 id를 보내지 않아도 저장 후 항상 `data_json` 안에 id가 포함된다.

### 1-5. 데이터 형식 3단계 — 저장 구조와 accessor 사용법

#### 저장 구조

**1단계 — flat (contentKey 없음)**
```json
{ "name": "홍길동", "status": "active" }
```

**2단계 — contentKey 중첩**
```json
{ "userInfo": { "name": "홍길동", "status": "active" } }
```

**3단계 — tabKey + contentKey 중첩**
```json
{ "tab1": { "userInfo": { "name": "홍길동", "status": "active" } } }
```

---

#### 테이블 컬럼 accessor 사용법

| 단계 | 저장 구조 | accessor | 비고 |
|---|---|---|---|
| 1단계 | `{ "name": "홍길동" }` | `name` | flat 직접 접근 |
| 2단계 (contentKey 1개) | `{ "userInfo": { "name": "홍길동" } }` | `name` 또는 `userInfo.name` | 중복 없으면 `fieldKey`만으로 가능 |
| 2단계 (contentKey 2개 — 같은 fieldKey 중복) | `{ "userInfo": { "name": "A" }, "orgInfo": { "name": "B" } }` | `userInfo.name`, `orgInfo.name` | **중복 시 dot notation 필수** |
| 3단계 | `{ "tab1": { "userInfo": { "name": "홍길동" } } }` | `tab1.userInfo.name` | **항상 full dot notation 필수** |

#### 핵심 규칙

```
1단계, 2단계 → fieldKey 단독 접근 가능 (buildTableRow가 자동으로 root에 flat 병합)
              단, 동일 fieldKey가 여러 contentKey에 중복이면 → contentKey.fieldKey 필수

3단계 (탭)   → tabKey.contentKey.fieldKey 항상 필수 (flat 병합 없음)
```

### 2-2. 수정 모드 데이터 복원

URL 파라미터에 따라 3가지 모드로 동작한다.

```
URL 파라미터       진입 모드        동작
─────────────────────────────────────────────────
?id=123           단일 수정 모드   단일 Form connectedSlug로 1건 조회
?group_id=uuid    다중 수정 모드   slug별로 개별 API 조회 후 각 위젯에 복원
(없음)            신규 입력 모드   hidden defaultValue + URL params 적용
```

#### 단일 수정 모드 (`?id=123`)

```typescript
// 첫 번째 Form 위젯의 connectedSlug 사용
GET /api/v1/page-data/{connectedSlug}/123
└── restoreFromDataJson(dataJson, formWidgets, sublistWidgets, multiSelectWidgets)
```

#### 다중 수정 모드 (`?group_id=uuid`)

```typescript
// 페이지의 모든 Form+SubList+MultiSelect 위젯에서 connectedSlug 수집
const slugSet = new Set([
    ...formWidgets.map(fw => fw.connectedSlug),
    ...sublistWidgets.map(sw => sw.connectedSlug),
    ...multiSelectWidgets.map(mw => mw.connectedSlug),
].filter(Boolean));

// slug별 개별 조회 후 복원
slugSet.forEach(s => {
    GET /api/v1/page-data/{s}/group/{groupId}
    └── restoreFromDataJson(dataJson, slugForms, slugSublists, slugMultiSelects)
});
```

#### `restoreFromDataJson` 내부 동작

```
dataJson 수신
  │
  ├── Form 위젯별
  │    contentKey 있으면 dataJson[contentKey] 섹션에서 필드 매핑
  │    contentKey 없으면 dataJson 최상위에서 fieldKey 기준 매핑
  │    → setFormValuesMap(widgetId, { fieldId: 값 })
  │
  ├── SubList 위젯별
  │    dataJson[contentKey].rows → _rowId 부여 → setSubListRowsMap()
  │
  ├── MultiSelect 위젯별
  │    dataJson[contentKey] (number[]) → setMultiSelectValuesMap()
  │
  └── 파일 메타 로드
       └── dataJson 재귀 순회 → 숫자 배열 → fileIds 수집
            GET /api/v1/page-files/meta?ids=1,2,3
            └── setExistingFileMetaMap(widgetId, { fieldId: [{id, origName, fileSize}] })
                 + image 타입 필드: GET /api/v1/page-files/{id} (blob) → setImgBlobUrls()
```

#### 탭 포함 페이지 수정 모드 (`?id=` + TabRenderer)

일반 폼 페이지와 달리, 탭 위젯이 포함된 페이지는 최상위 페이지(`widgetSub/[slug]`)에서 Form을 직접 찾을 수 없다.  
폼이 `TabRenderer` 내부에 중첩되어 있기 때문에 아래 2단계 흐름으로 데이터를 복원한다.

```
URL: /admin/widgetSub/{slug}?id=588
     │
     ▼ [1단계] TabRenderer 마운트
     │   useSearchParams()로 ?id 읽기
     │   tabs 배열 순회 → tab.pageSlug 있는 탭마다
     │   sharedDataIdMap[tab.pageSlug] = urlId 초기화
     │
     ▼ [2단계] useWidgetPageState (각 탭 패널 내부)
         sharedDataId = sharedDataIdMap[connectedSlug] 수신
         → GET /api/v1/page-data/{connectedSlug}/{sharedDataId}
         → dataJson[tab.contentKey][form.contentKey] 2단계 추출
         → setFormValuesMap, setExistingFileMetaMap 복원
```

**TabRenderer 초기화 코드 패턴**:

```typescript
const searchParams = useSearchParams();
const [sharedDataIdMap, setSharedDataIdMap] = useState<Record<string, number>>(() => {
    const urlId = searchParams.get('id') ? Number(searchParams.get('id')) : null;
    if (!urlId) return {};
    const map: Record<string, number> = {};
    tabs.forEach(tab => {
        if (tab.pageSlug) map[tab.pageSlug] = urlId; // slug별로 동일 id 세팅
    });
    return map;
});
```

> **주의**: `widgetSub/[slug]/page.tsx`에서 `restoreFromDataJson`으로 복원하려 하면  
> `flatWidgets(widgetItems)`에 TabRenderer 내부 폼이 포함되지 않아 **빈 배열**이 반환된다.  
> 탭 포함 페이지는 반드시 TabRenderer → useWidgetPageState 경유 복원 방식을 사용해야 한다.

#### 신규 모드 초기화

`widgetSub/[slug]/page.tsx`에서 `?id` 파라미터가 없는 신규 입력 모드로 진입하면  
Form 위젯의 모든 필드에 대해 기본값과 URL 파라미터를 아래 우선순위로 초기화한다.

```
우선순위 (높을수록 우선)
  1. URL 파라미터 (urlVal = searchParams.get(fieldKey))
  2. defaultOptionValue (select·radio·checkbox)
  3. defaultValueMsgKey (t()로 번역된 다국어 기본값)
  4. defaultValue (직접 텍스트 기본값)
```

```typescript
fw.fields.forEach(f => {
    const key = f.fieldKey || f.label || '';
    const urlVal = key ? searchParams.get(key) : null;

    if (f.type === 'date' && ...) {
        // date 처리
    } else if (f.type === 'dateRange') {
        // dateRange 처리
    } else if ((f.type === 'select' || f.type === 'radio' || f.type === 'checkbox') && f.defaultOptionValue) {
        vals[f.id] = urlVal ?? f.defaultOptionValue;   // URL 파라미터 우선
    } else if (f.defaultValueMsgKey) {
        vals[f.id] = urlVal ?? t(f.defaultValueMsgKey);
    } else if (f.defaultValue) {
        vals[f.id] = urlVal ?? f.defaultValue;
    }
});
```

> **활용 예시**: 카테고리 등록 팝업에서 `?depth=2&parentId=5`를 전달하면 fieldKey가 `depth`, `parentId`인 필드에 자동 세팅.  
> `select` 필드의 `defaultOptionValue`가 `"active"`로 설정되어 있으면 신규 입력 시 `"활성"`이 기본 선택된다.

**Search 위젯 기본값 초기화** (`widget/[slug]/page.tsx`, `widgetSub/[slug]/page.tsx` 공통)  
페이지 로딩 시 useEffect 1회 실행:

```typescript
// select·radio·checkbox
} else if ((f.type === 'select' || f.type === 'radio' || f.type === 'checkbox') && f.defaultOptionValue) {
    initSearchVals[f.id] = f.defaultOptionValue;
// input·hidden 등 직접 텍스트 기본값
} else if (f.defaultValue) {
    initSearchVals[f.id] = f.defaultValue;
}
```

### 2-3. 검색 흐름

```
[검색 버튼 클릭]
  handleSearch(searchWidgetId)
       │
       ├── searchValuesRef.current 참조 (stale closure 방지)
       └── Table 위젯 중 connectedSearchIds에 searchWidgetId 포함된 것 찾기
            └── fetchTableData({ ..., sv: 현재검색값, page: 0 })
                 └── GET /page-data/{slug}?필드키=값&page=0&size=10
```

#### 검색 파라미터 → BE 동적 쿼리 매핑

```
FE 파라미터 형식                        BE 쿼리 조건
──────────────────────────────────────────────────────────────────────────────────
name=홍길동                             data_json->>'name' ILIKE '%홍길동%'
eq_status=active                        data_json->>'status' = 'active'
createdAt=2026-01-01~2026-12-31
  → createdAt_start/end                 data_json->>'createdAt' >= '2026-01-01'
                                         AND data_json->>'createdAt' <= '2026-12-31'
tab1.form1.title=홍                     data_json->'tab1'->'form1'->>'title' ILIKE '%홍%'
eq_tab1.form1.title=홍길동              data_json->'tab1'->'form1'->>'title' = '홍길동'
```

> - 일반 파라미터 (`name=값`): ILIKE 부분 일치 + contentKey 1단계 중첩도 동시 검색  
> - `eq_` 접두사: 정확 일치 (카테고리 parentId 필터 등에 사용)  
> - `~` 구분자: 범위 검색 (날짜 범위, 숫자 범위)  
> - **dot notation** (`tab1.form1.title=값`): 탭·SubList 등 contentKey 하위 중첩 경로를 `.`으로 직접 지정. `eq_` 조합도 가능

#### 검색 파라미터 SQL Injection 방지

```java
// dot notation 키: 세그먼트 분리 후 각각 영문자/숫자/언더스코어 검증
if (key.contains(".")) {
    String[] segments = key.split("\\.");
    if (!isValidSegments(segments)) return;  // 세그먼트별 [a-zA-Z0-9_]+ 검사
}
// 일반 키: 단일 패턴 검증
```

### 2-5. popup 모드 데이터 복원 (`handleInternalPopupOpen`)

팝업 모드는 page 모드와 동일한 QUICK_DETAIL 템플릿을 레이어로 표시한다.  
표시 방식만 다를 뿐 데이터 읽기·저장 구조는 page 모드와 **완전히 동일**하다.

#### `parseActionParams` — Actions 컬럼 파라미터 → initialValues 변환 (WidgetRenderer.tsx)

테이블 Actions 컬럼의 수정 버튼에 **파라미터 문자열**을 설정하면,  
버튼 클릭 시 `parseActionParams`가 해당 row 데이터를 참조해 `initialValues`로 변환한다.

```
파라미터 형식: "title,temp1=1,temp2=abc"  (쉼표 구분)
  - =없음 (title)    → { title: row['title'] }   row의 해당 key 값
  - =있음 (temp1=1)  → { temp1: '1' }            고정값
```

**빌더 설정 위치**:  
`ActionsField.tsx` → 수정 버튼 `editPageRules`에서 각 규칙의 전달 파라미터(passParam) 입력  
`EditPageRule.passParam` 필드에 저장됨

#### editPageRules 매칭 흐름

수정 버튼 클릭 시 `editPageRules` 배열에서 이동할 규칙을 2단계로 탐색한다.

```
흐름:
onEdit(row)
  └── editPageRules 2단계 매칭
       1단계: conditionParam 있는 규칙 탐색
              conditionParam 형식: "key=value" (예: status=1)
              → row[key] === value 이면 매칭
       2단계: 1단계 매칭 없으면 conditionParam 빈 규칙(기본 경로)으로 폴백

  └── 매칭된 rule의 connType 분기 (connType 없으면 'popup' 기본값)
       connType='popup' → parseActionParams(rule.passParam, row) → initialValues
                          └── handleInternalPopupOpen(slug, editId, dataSlug, initialValues, groupId)
                               └── outputMode 자동 분기 (page이면 router.push, layerpopup이면 레이어 표시)
```

### 3-3. 유효성 검사 (`validateFormFields`)

**위치**: `utils.ts`  
**사용처**: `handleContentAction` (page 모드), `handlePopupContentAction` (popup 모드)

```typescript
// utils.ts — 각 조건 실패 시 toast.warning + false 반환 (첫 실패에서 즉시 중단)
validateFormFields(
    fields:           FormFieldItem[],
    values:           Record<string, string>,    // fieldId → 값
    fileValues:       Record<string, File[]>,    // fieldId → 신규 파일
    existingFileMeta: Record<string, any[]>,     // fieldId → 기존 파일 메타
    allValues?:       Record<string, string>,    // 페이지 내 전체 Form 값 (cross-form 조건 평가용)
    allKeyToId?:      Record<string, string>,    // 페이지 내 전체 fieldKey→id 역매핑
)
```

| 검사 순서 | 검사 항목 | 조건 |
|---|---|---|
| 0 | hidden 타입 | 항상 건너뜀 |
| 1 | hideCondition | 조건 충족 시 해당 필드 건너뜀 (HIDE 상태이므로) |
| 2 | 필수 | `required=true` + 값이 비어있음 |
| 3 | 최소 길이 | `minLength` 미충족 |
| 4 | 최대 길이 | `maxLength` 초과 |
| 5 | 패턴 | `pattern` 정규식 불일치 |
| 6 | 파일 최대 개수 | `maxFileCount` 초과 (기존 + 신규 합산) |
| 7 | 파일 단건 용량 | `maxFileSizeMB` 초과 |
| 8 | 파일 전체 용량 | `maxTotalSizeMB` 초과 |
| 9 | 날짜 범위 | `dateRange` / `yearMonthRange` 종료일 < 시작일 |

#### hideCondition / disableCondition 평가 — `FormRenderer` 내 `evalCondition`

```typescript
// FormRenderer.tsx 내부 — "key=val" / "key!=val" / 복수 AND 조건 ("k1=v1,k2=v2") 지원
// 참조 우선순위: 현재 Form 필드 > cross-form 필드 > URL 파라미터

// 예시
"status=1"           → status 필드 값이 "1"이면 HIDE/DISABLE
"type!=Y"            → type 필드 값이 "Y"가 아니면 HIDE/DISABLE
"status=1,type=Y"    → 두 조건 모두 충족 시 HIDE/DISABLE (AND)
"form1.status=1"     → 다른 Form(contentKey=form1)의 status 필드 값이 "1"이면 HIDE/DISABLE
"test=1"             → URL 파라미터 ?test=1로 접근 시 HIDE/DISABLE (필드 없을 때 urlParams 폴백)
```

**URL 파라미터 참조 동작 원리**

`key`에 해당하는 Form 필드(fieldId)가 없을 경우 `urlParams` 에서 직접 조회한다.

```
condition: "test=1"
  → resolvedKeyToId["test"] → undefined (해당 fieldKey 없음)
  → urlParams["test"] → "1" (URL ?test=1로 접근한 경우)
  → 조건 충족 → HIDE
```

`urlParams`는 `widgetSub/[slug]/page.tsx` → `PageGridRenderer` → `WidgetRenderer` → `FormRenderer` 순으로 전달된다.  
`id`, `group_id` 키는 제외되고 나머지 URL 파라미터만 포함된다.

> **cross-form hideCondition**: 동일 페이지 내 다른 Form 필드 참조 시  
> `contentKey.fieldKey` 형식 사용. (예: `personal.status=Y`)  
> 단, contentKey에 `.`(점) 포함 불가 (FormBuilder contentKey 입력 시 자동 차단됨)

```typescript
// Form 위젯
if (fw.contentKey) dataJson[fw.contentKey] = section;  // 중첩 저장
else Object.assign(dataJson, section);                   // flat 저장

// SubList 위젯
if (sw.contentKey) dataJson[sw.contentKey] = { rows };
else dataJson.rows = rows;

// MultiSelect 위젯
if (mw.contentKey) dataJson[mw.contentKey] = selectedIds;
```

#### 호출 전 사전 처리 순서 (필수)

```
1. Form 파일 필드 업로드
   fileValuesMap[widgetId][fieldId] 의 신규 File[] 를 업로드
   → formFileIdsMap[widgetId][fieldId] = [...기존MetaId[], ...신규업로드Id[]]

2. SubList 파일 컬럼 업로드 + rows 처리
   각 row의 파일 컬럼 신규 File[] 를 업로드
   _rowId 제거 + 파일 컬럼 ID 배열 완성
   → processedSubListRowsMap[widgetId] = processedRows[]

3. buildDataJson 호출
   → { dataJson, pkKeys }
```

> **왜 순서가 중요한가**: `buildDataJson`은 순수 동기 함수다.  
> 비동기 파일 업로드가 완료된 후에 호출해야 파일 ID가 dataJson에 포함된다.  
> SubList 파일 처리 전에 호출하면 해당 파일 컬럼이 빈 배열로 저장된다.

### 3-7. popup 모드 저장 흐름 (`handlePopupContentAction`)

page 모드(`handleContentAction`)와 저장 로직이 동일하다. `buildDataJson` 공통 함수를 공유한다.

```
[팝업 내 Space 버튼 → 저장]
     │
① 유효성 검사 — form 위젯별 (page 모드와 동일)
   validateFormFields(fwFields, popupFormValuesMap[fwId], popupFileValuesMap[fwId], popupExistingMetaMap[fwId])

② Form 파일 업로드 → formFileIdsMap[widgetId][fieldId]
   popupExistingMetaMap[fwId][f.id].map(m => m.id) + 신규 업로드

③ SubList rows 처리 → processedSubListRowsMap[widgetId]
   파일 컬럼 업로드 + _rowId 제거

④ buildDataJson(allWidgets, popupFormValuesMap, formFileIdsMap, processedSubListRowsMap, {})
   → { dataJson }

⑤ PUT  /page-data/{popupListSlug}/{popupEditId}  (수정)
   POST /page-data/{popupListSlug}                (신규)

⑥ PATCH /page-files/link  (신규 파일 있을 때만)

⑦ onRefresh() → 목록 재조회
   setCategoryRefreshTick() → 카테고리 위젯 재조회
   handlePopupClose()
```

> **page 모드와의 차이**: popup은 단일 slug만 처리 (다중 slug·groupId 없음).  
> 저장 후 `router.back()` 없이 팝업만 닫힌다.

### 4-4. 테이블 row 변환 — `buildTableRow` 공통 함수 (⭐ 필독)

**위치**: `utils.ts` → `export function buildTableRow(item)`  
**사용처**: `useWidgetPageState.ts`, `/admin/widget/[slug]/page.tsx`, `/admin/widgetSub/[slug]/page.tsx`  
세 곳 모두 `.map(buildTableRow)` 한 줄로 공통 처리. **절대 복붙 금지.**

#### 3단계 accessor 처리 규칙

| 단계 | dataJson 저장 구조 | accessor 사용법 |
|---|---|---|
| **1단계 flat** | `{ title: "A", use: "1" }` | `title`, `use` |
| **2단계 contentKey** | `{ form1: { title: "A", use: "1" } }` | `title`, `use` (중복 없으면 자동 root 병합) |
| **2단계 contentKey 중복** | `{ form1: { title: "A" }, form2: { title: "B" } }` | `form1.title`, `form2.title` (중복 → dot notation 필수) |
| **3단계 tab+contentKey** | `{ tab1: { form1: { title: "A" } } }` | `tab1.form1.title` (dot notation 필수) |

#### 핵심 로직

```typescript
// utils.ts — buildTableRow
export function buildTableRow(item) {
    const { id: _omit, ...restDataJson } = item.dataJson ?? {};

    // 최상위 object 값(contentKey 섹션) 감지
    const sectionEntries = Object.entries(restDataJson)
        .filter(([, v]) => v !== null && typeof v === 'object' && !Array.isArray(v));

    // 중복 없는 fieldKey만 root에 flat 병합 (2단계 대응)
    const flatExtra: Record<string, unknown> = {};
    if (sectionEntries.length > 0) {
        const keyCount: Record<string, number> = {};
        sectionEntries.forEach(([, s]) =>
            Object.keys(s).forEach(k => { keyCount[k] = (keyCount[k] ?? 0) + 1; })
        );
        sectionEntries.forEach(([, s]) =>
            Object.entries(s).forEach(([k, v]) => {
                if (keyCount[k] === 1) flatExtra[k] = v;  // 중복 없는 것만
            })
        );
    }

    return {
        _id: item.id,
        _groupId: item.groupId ?? null,
        ...flatExtra,       // 중복 없는 fieldKey flat (단순 accessor용)
        ...restDataJson,    // 원본 중첩 구조 보존 (dot notation accessor용)
        createdAt: item.createdAt ?? null,
        createdBy: item.createdBy ?? null,
        updatedAt: item.updatedAt ?? null,
        updatedBy: item.updatedBy ?? null,
    };
}
```

#### 동작 예시

```
// 케이스 1: flat (contentKey 없음)
dataJson = { title: "A" }
→ row = { _id:1, title:"A", createdAt:... }
→ accessor "title" ✅

// 케이스 2: contentKey 1개 (중복 없음)
dataJson = { form1: { title: "A", use: "1" } }
→ row = { _id:1, title:"A", use:"1", form1:{ title:"A", use:"1" }, ... }
→ accessor "title" ✅ / "form1.title" ✅

// 케이스 2: contentKey 2개 (fieldKey 중복)
dataJson = { form1: { title: "A" }, form2: { title: "B" } }
→ row = { _id:1, form1:{title:"A"}, form2:{title:"B"}, ... }  // title은 root 미병합
→ accessor "form1.title" ✅ / "form2.title" ✅ / "title" ❌

// 케이스 3: tab+contentKey
dataJson = { tab1: { form1: { title: "A" } } }
→ row = { _id:1, tab1:{ form1:{ title:"A" } }, ... }
→ accessor "tab1.form1.title" ✅ (TableCellRenderer dot notation 지원)
```

#### TableCellRenderer dot notation 지원 (TableCellRenderer.tsx)

```typescript
const value = col.accessor.includes('.')
    ? col.accessor.split('.').reduce((acc, key) => {
        if (acc && typeof acc === 'object' && !Array.isArray(acc))
            return acc[key];
        return undefined;
    }, row)
    : row[col.accessor];
```

> `.` 개수 제한 없음 — `tab1.form1.fieldKey` 3단계까지 자동 지원.

### 4-5. 카테고리 위젯 조회 방식 — dot notation 필드 읽기

테이블과 카테고리는 contentKey 중첩 구조를 처리하는 방식이 다르다.

```
테이블 (fetchTableData):
  dataJson의 모든 object wrapper를 자동으로 펼침 (flat-map)
  → { lv1pop: { title: 'A' } } → flat에 title 직접 추가

카테고리 (fetchItems):
  readField() 유틸로 dot notation 경로를 명시적으로 지정
  → widget.fieldTitle = 'lv1pop.title' 이면 dataJson['lv1pop']['title'] 접근
```

**`readField()` 동작 원리** (`CategoryRenderer.tsx` 내부):

```typescript
const readField = (dataJson: Record<string, unknown>, fieldKey: string): unknown => {
    const dotIdx = fieldKey.indexOf('.');
    if (dotIdx !== -1) {
        const parentKey = fieldKey.slice(0, dotIdx);   // 'lv1pop'
        const childKey  = fieldKey.slice(dotIdx + 1);  // 'title'
        const section   = dataJson[parentKey];
        if (section && typeof section === 'object' && !Array.isArray(section)) {
            return (section as Record<string, unknown>)[childKey];
        }
        return undefined;
    }
    return dataJson[fieldKey];  // dot 없으면 root 직접 접근
};
```

**동작 예시**:

```
dataJson = { "lv1pop": { "title": "대분류A", "code": "A-001" } }

readField(dataJson, 'lv1pop.title')  → '대분류A'   ✓
readField(dataJson, 'lv1pop.code')   → 'A-001'     ✓
readField(dataJson, 'title')         → undefined    (root에 없음)
```

> 카테고리 위젯 빌더에서 카드 필드 키를 `폼key.컬럼명` 형식으로 설정하는 이유가 이것이다.  
> 설정값이 연결 폼의 contentKey와 다르면 해당 필드는 빈값으로 표시된다.

```typescript
    [widgetId]: { ...(prev[widgetId] ?? {}), [fieldId]: value },
}));
```

> **이유 1**: flat 구조에서는 `contentKey`가 있어도 `buildDataJson`이 섹션을 찾지 못해 중첩 저장이 불가능하다.  
> **이유 2**: form 위젯이 2개 이상일 때 fieldId가 겹치면 나중 폼의 값이 이전 폼의 값을 덮어쓰는 충돌이 발생한다.

### 9-2. tableWidgetsMap 구성 방식

`WidgetRenderer`는 테이블 위젯 정보를 직접 조회하지 않는다.
`PageGridRenderer`가 `widgetItems`에서 수집해 `tableWidgetsMap` prop으로 주입한다.

```typescript
// PageGridRenderer.tsx — useMemo, widgetItems 변경 시 자동 재구성
const tableWidgetsMap = useMemo(() => {
    const map: Record<string, TableWidget> = {};
    widgetItems.flatMap(item => item.contents.map(c => c.widget))
        .filter((w): w is TableWidget => w.type === 'table')
        .forEach(w => { map[w.widgetId] = w; });
    return map;
}, [widgetItems]);
```

> **금지**: WidgetRenderer 내부에서 API를 호출해 테이블 위젯 정보를 직접 조회하는 것은 금지.
> 반드시 `tableWidgetsMap` prop 경유.

### 9-3. ⚠️ 정책 — 반드시 지켜야 할 규칙

#### 정책 1 — 헤더는 `headerMsgKey` 우선

```typescript
// ✅ 올바른 방법
const headers = validCols.map(c => c.headerMsgKey ? t(c.headerMsgKey) : c.header).join(',');

// ❌ 금지
const headers = validCols.map(c => c.header).join(',');
```

**이유**: 빌더에서 다국어 키(`headerMsgKey`)만 지정하고 `header`를 빈 문자열로 두는 패턴이 일반적이다.
이런 컬럼이 목록 뒤쪽에 있으면 join 결과에 trailing 쉼표가 생기고,
BE `split(",")` 기본 동작이 trailing 빈 문자열을 제거해 헤더 수가 줄어든다.
예: `"이름,이메일,,"` → Java `split(",")` → `["이름", "이메일"]` (2개만 남음)

---

#### 정책 2 — BE `split(",", -1)` 필수

```java
// ✅ 올바른 방법
List<String> headerList = Arrays.asList(headers.split(",", -1));

// ❌ 금지
List<String> headerList = Arrays.asList(headers.split(","));
```

**이유**: Java `split(regex)` 기본 동작은 trailing 빈 문자열을 제거한다.
`limit=-1`로 호출해야 빈 문자열도 보존된다.
FE 정책 1을 지켜도 안전장치로 반드시 적용한다.

---

#### 정책 3 — 셀 값은 `getNestedValue()` 경유 필수

```java
// ✅ 올바른 방법
Object value = getNestedValue(rowData, keys.get(colIdx));

// ❌ 금지
Object value = rowData.get(keys.get(colIdx));
```

**이유**: `page_data.data_json`은 `tab1.form1.키` 같은 중첩 구조로 저장된다.
테이블 컬럼 accessor가 `"tab1.form1.제목"` 형태일 때 `rowData.get("tab1.form1.제목")`은
항상 null이다 — Map의 키는 dot을 포함한 문자열로 저장되지 않는다.

`getNestedValue()` 동작:
```
getNestedValue(row, "제목")              → row.get("제목")                    [1단계, flat]
getNestedValue(row, "form1.제목")        → row → "form1" → "제목"             [2단계]
getNestedValue(row, "tab1.form1.제목")   → row → "tab1" → "form1" → "제목"   [3단계]
```

---

#### 정책 5 — 날짜 컬럼 포맷은 `dateFormats` 파라미터로 전달

테이블 컬럼에 `dateFormat`(예: `YYYY-MM-DD`, `YYYY/MM/DD HH:mm`)이 설정된 경우 화면 표시와 동일한 포맷이 엑셀에도 적용된다.

FE는 `keys`와 동일한 순서로 각 컬럼의 `dateFormat` 값을 `dateFormats` 파라미터에 포함해 전달한다. 포맷이 없는 컬럼은 빈 문자열로 채워 **순서를 반드시 유지**해야 한다. 순서가 어긋나면 엉뚱한 컬럼에 날짜 포맷이 적용된다.

BE `ExcelService`는 ISO 8601 형식 값을 파싱하여 `YYYY`, `MM`, `DD`, `HH`, `mm`, `ss` 토큰을 치환한다. 파싱에 실패하면 원본값을 그대로 출력한다.

`createdAt`, `updatedAt` 등 메타 필드도 accessor로 사용 가능하므로, 날짜 포맷이 필요하면 테이블 컬럼 빌더에서 해당 컬럼의 `dateFormat`을 설정하면 된다.

---

#### 정책 4 — 컬럼 너비는 `measureLen()` 기반, `autoSizeColumn()` 사용 금지

```java
// ✅ 올바른 방법
sheet.setColumnWidth(i, Math.min(Math.max(colMaxLen[i] + 2, 12), 50) * 256);

// ❌ 금지
sheet.autoSizeColumn(i);
```

**이유**: Apache POI `autoSizeColumn()`은 한글 폰트 너비를 영문 기준으로 과소 측정해
한글이 포함된 컬럼이 절반 이하 너비로 잘려 보인다.

`measureLen()` 규칙:
- 한글 완성형(AC00~D7A3), 자모(1100~11FF, 3130~318F): 2자
- 영문·숫자·기타: 1자

`colMaxLen[]` 추적:
- 헤더 행 작성 시 각 컬럼 초기화
- 데이터 행 작성 시 최대값으로 업데이트 (행 작성과 동시에 추적)

너비 범위: 최소 12자 ~ 최대 50자 (여유 2자 포함)
