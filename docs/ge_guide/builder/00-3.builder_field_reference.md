### 0-1. Preview vs Live 모드

| 구분 | preview 모드 (빌더 미리보기) | live 모드 (운영 페이지) |
|---|---|---|
| 입력 | `pointer-events-none` — 클릭·입력 모두 차단 | 실제 입력·클릭 가능 |
| 옵션 | 처음 3~4개만 샘플 표시 | 전체 옵션 로드 |
| 공통코드 | `field.options` 슬라이스로 대체 | `codeGroupCode` → API로 전체 로드 |
| 파일 업로드 | `canAdd = false`, 버튼 비활성 | 실제 업로드 가능 |
| onChange | 없음 (또는 undefined) | 활성 |
| 읽기전용 | 구분 없음 (전체 비활성) | `field.readonly=true`면 `bg-slate-50` 스타일 적용 |

```typescript

```

### 0-3. 검색 API 파라미터 규칙

SearchRenderer에서 수집한 `values` 맵을 기반으로 API 쿼리를 구성한다.

| 필드 타입 | values 키 | API 파라미터 | 예시 |
|---|---|---|---|
| `input`, `select`, `radio` | `fieldId` | `eq_fieldKey` | `eq_status=active` |
| `checkbox`, `button`(다중) | `fieldId` | `in_fieldKey` | `in_tags=tag1,tag2` |
| `date` | `fieldId` | `fieldKey_gte` / `fieldKey_lte` | `createdAt_gte=2024-01-01` |
| `dateRange` | `fieldId_from`, `fieldId_to` | `fieldKey_gte` / `fieldKey_lte` | |
| `category` | `fieldId` | `eq_fieldKey` | `eq_categoryId=5` |
| `dateRangeStatus` | `fieldId` | `fieldKey_status` | `period_status=in_range` |
| `input`·`select` 등 (OR) | `fieldId` | `fieldKey1\|fieldKey2` | `title\|content=홍길동` |
| `select`(조건식, `data` 설정 시) | `fieldId` | `condexpr_fieldKey` + `condval_fieldKey` | 파생값(계산된 표시값) 조건 검색 |

> ⚠️ 파라미터 키는 `fieldId`(렌더러 내부 상태 키)가 아닌 `fieldKey`(데이터 컬럼명) 기준으로 API에 전달된다. 혼동 주의.

#### OR 검색 — fieldKey에 `|` 사용

하나의 검색어로 **여러 컬럼을 동시에(OR) 부분 일치 검색**하려면 검색 필드의 `fieldKey`에 `|`로 컬럼을 나열한다. FE 변경 없이 **fieldKey 설정만으로** 동작한다.

```
검색 필드 설정:  fieldKey = "title|content"
FE → API 전달:   ?title|content=홍길동
```

**BE 처리** (`bo-api … service/PageDataService.java` — `appendWhereConditions()` + `bindSearchParams()`):

파라미터 키에 `|`가 포함되면 OR 다중 필드 ILIKE 블록으로 분기한다. **최상위 컬럼과 1단계 중첩(`form1.title` 등) 객체를 동시에 탐색**한다.

```sql
-- ?title|content=홍길동
AND (
    data_json->>'title'   ILIKE '%홍길동%'
    OR data_json->>'content' ILIKE '%홍길동%'
    OR EXISTS (
        SELECT 1 FROM jsonb_each(data_json) kv
        WHERE jsonb_typeof(kv.value) = 'object'
        AND ( kv.value->>'title'   ILIKE '%홍길동%'
           OR kv.value->>'content' ILIKE '%홍길동%' )
    )
)
```

| 항목 | 규칙 |
|---|---|
| 매칭 방식 | 각 필드 `ILIKE '%값%'` 부분 일치, 필드 간 **OR** 결합 |
| 탐색 범위 | 최상위 키 + **1단계 중첩** 객체 (`jsonb_each`로 form 컨테이너 내부까지) |
| 필드명 제약 | 각 컬럼명은 `[a-zA-Z0-9_]+` 만 허용 — 하나라도 위반 시 조건 생략 (SQL Injection 방지) |
| 파라미터 바인딩 | `p_or_{fields}` (`\|` → `__` 치환) 단일 파라미터로 모든 필드에 `%값%` 공통 바인딩 |

> 단순 단일 컬럼(`|` 없음) 검색도 동일하게 최상위 + 1단계 중첩을 ILIKE로 탐색한다. `|`는 거기에 OR 다중 컬럼만 추가하는 확장이다.

### 0-4. 공통코드(codeGroupCode) 처리 방식

```typescript
// FieldRenderer.tsx — resolveOptions()
const resolveOptions = (field, codeGroups) => {
    if (field.codeGroupCode) {
        // live: codeGroups 배열에서 groupCode 일치하는 그룹 찾아 details 반환
        return codeGroups
            .find(g => g.groupCode === field.codeGroupCode)
            ?.details.filter(d => d.active)
            .map(d => `${d.nameMsgKey ?? d.name}:${d.code}`) ?? [];
    }
    return field.options ?? [];
};
```

- **preview 모드**: `field.options?.slice(0, 4)` — 빌더에서 수동 입력한 샘플 4개만 표시
- **live 모드**: `codeGroupCode`로 실시간 조회 — `nameMsgKey` 있으면 `t(msgKey)`로 번역

#### 필드별 공통코드 사용 방식

공통코드는 두 가지 방식으로 활용된다.

**① 옵션 소스형** — 사용자가 직접 선택, 선택 결과(코드값)가 저장됨

| 필드 | 사용 위젯 | 동작 |
|---|---|---|
| `SelectField` | Search, Form, SubList | 코드그룹 → 드롭다운 옵션 자동 생성 |
| `RadioField` | Search, Form, SubList | 코드그룹 → 라디오 버튼 옵션 자동 생성 |
| `CheckboxField` | Search, Form, SubList | 코드그룹 → 체크박스 옵션 자동 생성 |
| `ButtonField` | Search, Form | 코드그룹 → 버튼 옵션 자동 생성 |

**② 표시 변환형** — 이미 저장된 코드값을 이름으로 변환해서 보여줌 (readOnly)

| 필드 | 사용 위젯 | 동작 |
|---|---|---|
| `InputField` | Form, SubList | 코드값 → 이름/코드값 변환 표시. `displayAs: 'text'`=이름, `displayAs: 'value'`=코드값 |
| TableColumn `text` | Table | 동일 (`TextCodeGroupField`로 설정) |

> ⚠️ 표시 변환형은 저장값을 변경하지 않는다. 화면 표시만 변환되며 해당 필드는 자동으로 readOnly 처리된다.
- **옵션 포맷**: `"텍스트:값"` 문자열 → `parseOpt()` 함수로 `{ text, value }` 분리

### 0-5. hideCondition / disableCondition 문법

```
형식:  "fieldKey=값"               단일 조건 (일치 시 숨김/비활성)
       "fieldKey!=값"              단일 조건 (불일치 시 숨김/비활성)
       "key1=v1,key2=v2"          AND 복수 조건 (모두 일치 시 적용)
예:    "status=active,type=form"
```

**적용 흐름** (SearchRenderer / FormRenderer):
1. `evalFieldCondition(hideCondition, keyToId, values)` 호출
2. 조건 충족 → 해당 필드 `return null` (렌더링 스킵)
3. `disableCondition` 충족 → `forceDisabled=true`를 FieldRenderer에 전달
4. **preview 모드에서는 조건 평가 안 함** (항상 표시)
5. **hideCondition 충족된 필드는 유효성 검사 건너뜀** (required여도 오류 없음)

**FormRenderer cross-form 참조**:
- 현재 폼 필드 외에 다른 폼·탭의 fieldKey도 참조 가능
- URL 파라미터도 조건값으로 사용 가능 (`urlParams`)

### 0-6. 유효성 검사 흐름 (validateFormFields)

저장 버튼 클릭 시 `utils.ts`의 `validateFormFields()`가 순서대로 검사한다.

```
1. hideCondition 충족 필드 → 건너뜀
2. required → 빈 값이면 toast.warning 후 false 반환
3. minLength / maxLength → 글자 수 초과/미달
4. pattern → 정규식 불일치 + patternDesc 안내
5. 파일(file/image/video/media):
   - maxFileCount: 개수 초과
   - maxFileSizeMB: 개당 용량 초과
   - maxTotalSizeMB: 전체 합산 용량 초과
6. dateRange: 종료일 < 시작일
7. date/dateRange 필드간 대소비교(§0-7 참고): compareExpr 설정된 경우 콤마로 구분된 각 조건을 순서대로 검사, 첫 위반에서 중단
8. checkbox: minSelect / maxSelect 위반
```

> 검증 실패 시 즉시 `false` 반환 + toast 노출. 통과 시 `true` 반환.

### 0-7. 필드간 대소비교 검증 (compareExpr)

date/dateRange 필드에 "다른 날짜 필드보다 이전/이후여야 한다"는 조건을 콤마구분 단일 문자열로 설정할 수 있다.
hideCondition/optionFilter와 동일한 `$fieldKey` 참조 컨벤션을 그대로 따른다.

**문법**: `[part]연산자$대상fieldKey[_from|_to]` 를 콤마(,)로 나열 — 콤마=AND(전부 만족해야 저장 가능)
- 연산자: `< <= > >=`
- `part`(연산자 앞, 옵션): 자기 자신이 dateRange일 때 비교에 쓸 값(from/to). date 필드는 해당 없음
- `_from`/`_to`(대상fieldKey 뒤, 옵션): 대상이 dateRange일 때 쓸 값. 생략 시 date는 그대로/dateRange는 from 기본

| 예시 | 의미 |
|---|---|
| `<$endDate` | 자신(date) < endDate(date) |
| `<$endDate,<$dueDate` | 자신 < endDate 이고 자신 < dueDate (AND) |
| `to<$endDate_from` | 자신(dateRange)의 종료일 < endDate(dateRange)의 시작일 |

**크로스위젯 참조**: SubList 컬럼의 compareExpr은 같은 SubList 컬럼을 먼저 찾고, 없으면 같은 페이지의 다른 Form 위젯 필드로 폴백한다(hideCondition의 cross-form 참조와 동일한 방향).

**검증 위치**: `utils.ts`의 `validateFormFields()`/`validateSubListRows()` 내 `checkCompareExpr()`/`parseCompareToken()` — 위반 시 공통 메시지(`common.validation.date_compare_lt/lte/gt/gte`, `{label}`/`{compareLabel}` 필드명 치환) 표시 후 저장 차단.

> 대상 fieldKey를 못 찾거나 문법이 어긋난 토큰은 에러 없이 조용히 무시된다(hideCondition/optionFilter와 동일한 관대한 컨벤션).

| `joinRelationSlugId` | number | - | **조인 검색 연동** (Search 위젯 전용, 신규) — 이 필드의 검색값이 자기 slug가 아니라 연결된(FETCH) slug에만 있을 때 사용. §1-1 참고 |
| `joinSlaveKey` | string | - | 조인 검색 시 연결 slug 쪽에서 비교할 필드 경로(예: `productDataForm.productNm`) |

### 세부 설정 UI

1. **_FieldBase** — 라벨 행 오른쪽에 "연결 Slug" 셀렉터(FETCH 타입만 표시), Key 행 오른쪽에 "Data" 표현식 입력
2. **Placeholder** — 텍스트 직접 입력 또는 MessageKeySelector (i18n 모드)
3. **기본값** — 텍스트 직접 입력 또는 MessageKeySelector
4. **유효성 검사** — 최소/최대 글자(2열), 글자수 표시 토글, 정규식 + 설명
5. **데이터 자동생성** — [DataGenerationSection](#a-공통--datagenerationsection) 참조

> **빌더 패널 레이아웃 (연결 Slug 설정 시)**
> ```
> [ 라벨 입력          | 연결 Slug 드롭다운 ]   ← FETCH 타입만 표시
> [ Key (자동: _fetchedRel{id}) | Data 표현식 ]
> ```

### 렌더러 동작

```tsx
// FieldRenderer.tsx — input 렌더링

// 1. 연결 Slug 설정 시 — fetchRelData에서 값 참조 (Form 위젯 전용)
if (field.relationSlugId) {
    // 값이 문자열(단건 매칭)이면: data 표현식 있으면 evalColumnDataExpr로 계산,
    //   없으면 rowData[field.fieldKey](_fetchedRel{id}) 직접 참조
    // 값이 배열(다건 매칭)이면: 관리자가 설정한 구분자(fetchSeparator)로 한 줄로 합쳐 표시
    //   — input은 여러 줄 표시가 불가능한 단일 라인 요소라 항상 한 줄로 합침 (formatFetchedRelValue 공통함수)
    return <input readOnly value={displayVal} />;
}

// 2. 일반 input
<input
    type="text"
    value={value ?? ''}
    placeholder={field.placeholderMsgKey ? t(field.placeholderMsgKey) : field.placeholder}
    maxLength={field.showCharCount && field.maxLength ? field.maxLength : undefined}
    disabled={isDisabled}
    readOnly={isReadOnly}
    onChange={e => onChange?.(e.target.value)}
/>
{/* 글자수 표시 ON + maxLength 있으면 */}
{field.showCharCount && field.maxLength && (
    <span>{(value ?? '').length}/{field.maxLength}</span>
)}
```

**연결 Slug(FETCH) 데이터 흐름 (Form 위젯)**:
1. BE `getById` → `applyFetch` 호출 → `dataJson` top-level에 `_fetchedRel{id}` 포함
   (매칭 1건이면 문자열, 2건 이상이면 배열 — 배열일 때는 형제 키 `_fetchedRel{id}_sep`에 구분자 함께 전달)
2. `useWidgetPageState` → `extractFetchRelData` → `formFetchRelMap` 상태에 저장
3. `FormRenderer` → `rowData` 생성 시 `fetchRelData` 값 주입
4. `FieldRenderer` → `formatFetchedRelValue` 공통함수로 표시 — 문자열이면 그대로, 배열이면 구분자로 한 줄 합침
   (다건 매칭 응답 규격 전체는 `docs/pages/slug-relation/be_slug-relation.md`, `fe_slug-relation.md` 참조)

**데이터 자동생성 적용 (FormRenderer)**:
- 이 필드 onChange 시 `dataGenerations` 배열 순회
- 각 세트의 변환 로직 적용 후 `generationKey` 경로의 대상 필드에 자동 입력
- 변환 순서: `dataReplacement(하이픈)` → `caseChange(대/소문자)` → `appendText` → `truncateLength`
- `onlyIfEmpty=true`면 대상 필드가 빈값인 경우에만 적용

### 특수 기능

- **레거시 마이그레이션**: `generationKey` 단일 속성으로 저장된 구버전 데이터는 첫 로드 시 `dataGenerations[0]`으로 자동 변환

### 1-1. 조인 검색 연동 (`joinRelationSlugId`/`joinSlaveKey`) — Select/Input 공통, Search 위젯 전용

**언제 필요한가**: 검색 위젯의 결과 테이블(`connectedSlug`)이 A slug 기준인데, 검색 조건으로 쓰려는 값은 A에 없고 **연결된(FETCH relation) B slug에만** 있는 경우. 예: 결과 테이블은 `category-data`인데, "제품명"으로 검색하고 싶고 제품명은 `product-data`에만 있는 경우.

**왜 필요한가 (배경)**: 검색은 BE가 `WHERE data_slug = :slug ...`로 **단일 테이블만** 필터링한다. FETCH relation(`_fetchedRel{id}`)은 **조회 이후(2단계)**에만 값을 채워 넣으므로, "조회할 행을 결정하는 1단계"에는 관여하지 못한다 — 즉 표시는 되는데 그 값으로 검색은 원래 안 된다. `joinRelationSlugId`는 이 문제를 "축약 후 IN 조건" 방식(기존 `rel_`/FILTER 계층검색과 동일한 패턴)으로 해결한다.

**설정**:
- `joinRelationSlugId`: 대상 slug를 가리키는 **FETCH** 타입 relation ID (연결 Slug selectbox에서 FETCH만 후보로 노출됨)
- `joinSlaveKey`: 그 relation의 slave slug 쪽에서 비교할 필드 경로 (예: `productDataForm.productType`)
- 연산자는 **필드 타입으로 자동 결정**된다 — select는 정확일치(EQ), input은 부분일치(ILIKE). 관리자가 별도로 고르는 UI는 없음(YAGNI로 기각됨).

**FE 파라미터**: `joinr_{fieldKey}`=relationId, `joink_{fieldKey}`=slave 필드 경로(ILIKE면 앞에 `~` 마커 붙음), `joinv_{fieldKey}`=검색값. (`useWidgetPageState.ts`의 검색 파라미터 조립 시 select/input 필드가 `joinRelationSlugId`를 갖고 있으면 자동 생성)

**BE 처리**: `resolveJoinFilterIds`(신규, `PageDataService.java`) — `resolveFilterRelationIds`(카테고리 계층 전용, 절대 혼용 금지)와는 **완전히 별개 메서드**다. 연동 slug에서 조건에 맞는 레코드를 찾아 relation의 `slave_key`로 연결값을 뽑고, `master_key` 경로가 그 값과 일치하는 현재 검색 대상(메인 slug) 행 id를 모아 기존 `id IN (...)` 패턴을 그대로 재사용한다. 여러 조인 필드를 동시에 검색하면 AND 교집합.

**제약**: 조인 검색은 **relation의 master_slug == 검색 위젯이 속한 결과 테이블의 connectedSlug**일 때만 정상 동작한다. 다르면 조용히 0건이 나온다(빌더에 이 불일치를 막는 검증은 아직 없음 — 설정 시 relation의 master_slug를 반드시 확인할 것).

**relationSlugId(Form 표시용)와 절대 혼동 금지**: InputField의 기존 `relationSlugId`(위 §1 참고)는 FETCH 값을 **Form에서 읽기전용으로 표시**하는 완전히 다른 기능이다. 이름이 비슷하니 설정 시 어느 필드에 값을 넣는지 주의할 것.

1. **타입 선택** — `selectbox`(기본) / `autocomplete` (빌더 패널 최상단)

### 렌더러 동작

```tsx
// ── selectType 분기 (live 모드) ──────────────────────────────────────────────
// autocomplete + 수동/공통코드: AutocompleteInput
//   - 텍스트 입력 시 text 기준 부분 일치 필터링 (대소문자 무관)
//   - × 버튼으로 선택값 초기화
//   - value(키값) 저장, 화면에는 text 표시 — selectbox와 동일 패턴
// autocomplete + optionSlug: SlugAutocompleteInput
//   - SLUG API fetch 완료 후 AutocompleteInput에 옵션 전달 → 동일 UX
// autocomplete (preview): input + 드롭다운 항상 펼침 (UI 확인용)

// ── selectbox (기존 동작) ────────────────────────────────────────────────────
// SLUG 모드 (live 전용): /page-data/{optionSlug}?size=9999 fetch
// flattenPageDataItem()으로 중첩 flat 후 row[optionValueKey] / row[optionTextKey] 추출
// optionOrderKey 설정 시 FE에서 flat key 기준으로 정렬 (Value/Text 키와 동일 기준)
// → Value 키, Text 키, 정렬 Key 모두 같은 flat key 방식으로 입력 (예: 'prdNm')
// optionFilter 설정 시 evalConditionExpr(utils.ts)로 flat row를 조건 평가해 필터링 (테이블 data 표현식과 동일 문법)
//   - `$fieldKey` 토큰은 같은 Form/Search 위젯 내 다른 필드의 현재 선택값을 참조 (Form/Search 전용, SubList 등에서는 무시됨)
//   - 참조 필드 값이 바뀌면 SLUG 재조회 없이 옵션만 다시 계산됨
// 수동/공통코드 모드: resolveOptions() → codeGroupCode 있으면 codeGroups에서, 없으면 field.options
```

- **preview**: `field.options?.slice(0, 4)` — 수동 입력 샘플 최대 4개 / SLUG 모드는 빈 options
- **live (수동/공통코드)**: `codeGroupCode` 있으면 `resolveOptions()`로 전체 옵션 로드
- **live (SLUG)**: `flattenPageDataItem()` 결과에서 flat key로 직접 접근 → `row[optionValueKey]`, `row[optionTextKey]`

- `selectType` 기본값 `'selectbox'` — 기존 데이터 하위 호환 (속성 미설정 시 기존 동작 유지)
- autocomplete 타입: value(키값)를 저장하고 화면에는 text를 표시 — selectbox와 동일 저장 방식
- 수동/공통코드/SLUG **택일** — 탭 전환 시 나머지 값 전부 초기화
- SLUG key는 `flattenPageDataItem` 기준 flat key 사용 — contentKey 접두어 없이 필드명만 입력 (예: `form.prdNm` 아닌 `prdNm`)
- `optionFilter` 미설정 시 전체 옵션 노출, 설정 시 조건에 맞는 행만 옵션으로 추출
- Value 키, Text 키, 정렬 Key 세 가지 모두 동일한 flat key 방식 — 입력 규칙이 통일됨
- `optionFilter`에 `$fieldKey`를 쓰면 다른 필드의 현재 선택값에 따라 옵션이 달라지는 연동(cascading) select를 구성할 수 있음 — 참조 필드 미선택 시 옵션은 빈 목록

```tsx
const selected = (value ?? '').split(',').filter(Boolean);

};

            <input
                type="checkbox"
                value={val}
                checked={selected.includes(val)}
                disabled={isDisabled}
                onChange={e => handleChange(val, e.target.checked)}
            />
            <span>{t(text)}</span>
        </label>

})}
```

   - `date/yearMonth/datetime`: `±` 버튼 + 숫자 input (offset 제어) + 날짜 input + 오늘 날짜·이전 날짜 토글
   - `time/timeSec`: 직접 시간 입력 (`<input type="time">`) — offset·오늘날짜 토글 없음, timeSec은 step=1(초 단위)
4. **필드간 검증** — 비교식 텍스트 input 1개 (Form/SubList 내 비교 가능한 date/dateRange 필드가 있을 때만 노출, §0-7 참고)

### dateSubType 분기

| dateSubType | input type | 저장 포맷 | offset 지원 | 비고 |
|---|---|---|---|---|
| `date` | `date` | `YYYY-MM-DD` | ✅ | 기본값 |
| `yearMonth` | `month` | `YYYY-MM` | ✅ | 년/월 선택만 |
| `datetime` | `datetime-local` | `YYYY-MM-DDTHH:mm` | ✅ | 날짜+시간 |
| `time` | `time` | `HH:mm` | ❌ | 시분 직접 입력 |
| `timeSec` | `time` (step=1) | `HH:mm:ss` | ❌ | 시분초 직접 입력 |

### 렌더러 동작

```tsx
// yearMonth 레거시 타입도 dateSubType='yearMonth'로 통합 처리
const dateSubType = field.type === 'yearMonth'
    ? 'yearMonth'
    : (field.dateSubType ?? 'date');

const isTimeBased = dateSubType === 'time' || dateSubType === 'timeSec';
const inputType = dateSubType === 'yearMonth' ? 'month'
    : dateSubType === 'datetime' ? 'datetime-local'
    : isTimeBased ? 'time'
    : 'date';

// disablePast: min 속성으로 오늘 날짜 설정
const minDate = field.disablePast ? getTodayString(dateSubType) : undefined;

<input type={inputType} value={value ?? ''} min={minDate}
    disabled={isDisabled} readOnly={isReadOnly}
    onChange={e => onChange?.(e.target.value)}
    onClick={e => e.currentTarget.showPicker?.()} />
```

**defaultToday 적용**: `defaultToday=true`면 렌더러에서 매번 오늘 날짜를 기본값으로 적용 (저장값이 비어있을 때)

**클릭 오픈 범위**: 브라우저 기본 아이콘뿐 아니라 input 전체(텍스트 세그먼트·여백 포함) 클릭 시 `showPicker()`로 달력이 열린다. 운영메뉴 생성 코드(`listGenerator.ts`)의 date/dateRange 검색조건에도 동일하게 반영되어 있다.

### 전용 속성

| 속성 | 타입 | 기본값 | 설명 |
|---|---|---|---|
| `rangeSubType` | `'date' \| 'yearMonth' \| 'datetime' \| 'time' \| 'timeSec'` | `'date'` | 범위 선택 단위 |
| `label2` | string | - | 종료일 라벨 |
| `label2MsgKey` | string | - | 종료일 다국어 키 |

### 세부 설정 UI

1. **범위 타입 선택** — select (date/yearMonth/datetime/time/timeSec)
2. **단일 날짜 범위 검색 토글**
3. **_FieldBase** — 라벨(시작), label2(종료), Key(시작일), 종료일 Key(선택, `fieldKey2`), ColSpan (minSpan=2 고정)
4. **시작일/종료일 기본값** — 각각 `±` 버튼 + 숫자 + 날짜 input + disablePast 토글
5. **최대 조회 기간** — 숫자 input + 단위 selectbox
6. **필드간 검증** — 비교식 텍스트 input 1개 (Form/SubList 내 비교 가능한 date/dateRange 필드가 있을 때만 노출, §0-7 참고)

### rangeSubType 분기

| rangeSubType | 저장 포맷 | offset 지원 | 비고 |
|---|---|---|---|
| `date` | `YYYY-MM-DD` | ✅ | |
| `yearMonth` | `YYYY-MM` | ✅ | |
| `datetime` | `YYYY-MM-DDTHH:mm` | ✅ | |
| `time` | `HH:mm` | ❌ | 시분 직접 입력 |
| `timeSec` | `HH:mm:ss` | ❌ | 시분초 직접 입력, step=1 |

### 렌더러 동작

**저장 키**: `fieldId_from`(시작), `fieldId_to`(종료) — 2개로 분리 저장. `fieldKey2` 지정 시 종료 키는 `fieldId_to`가 아니라 `fieldKey2` 그대로 사용됨  
**API 파라미터**: `fieldKey_gte`(시작), `fieldKey_lte`(종료) 변환 후 전달  
**entity 연결 저장**: SubList/Form이 entity에 연결된 경우, 이 시작/종료 키는 `entityApi.ts`의 `buildEntityDateFieldMeta`/`buildSubListEntityDateFieldMeta`가 요청 바디의 최종 entity 필드명을 만들 때 그대로 사용된다 — `fieldKey2` 미지정 시 `${fieldKey}_from`/`_to`를 만들어 존재하지 않는 필드명이 될 수 있으므로, entity 컬럼이 독립된 이름(`dispFrom`/`dispTo` 등)이면 `fieldKey2`를 반드시 지정해야 한다.

```tsx
// 2개 input 나란히
<div className="flex items-center gap-2">
    <input type={inputType} value={fromValue}
        onChange={e => onFromChange?.(e.target.value)}
        onClick={e => e.currentTarget.showPicker?.()} />
    <span className="text-slate-400">~</span>
    <input type={inputType} value={toValue}
        onChange={e => onToChange?.(e.target.value)}
        onClick={e => e.currentTarget.showPicker?.()} />
</div>
```

**BE 날짜 범위 처리**:  
FE에서 날짜값(`YYYY-MM-DD` 또는 `YYYYMMDD`)만 전달하면 BE가 자동으로 시간을 보정한다.  
- `_from` / `_gte` : 시작 시간(`000000` 또는 `T00:00:00`) 자동 추가  
- `_to` / `_lte`  : 종료 시간(`235959` 또는 `T23:59:59`) 자동 추가  
- fieldKey가 `createdAt`/`updatedAt`이면 실제 테이블 컬럼(`created_at`/`updated_at`)과 `timestamptz`로 비교하고, 그 외 fieldKey는 `data_json` 경로로 비교한다.

이 처리 덕분에 DB에 `YYYYMMDDHHMMSS` / `YYYY-MM-DDTHH:mm` 등 시간이 포함된 값이 저장된 경우에도  
날짜 단위 검색 시 경계 당일 데이터가 누락 없이 조회된다.

**singleDateRange**: `fieldKey_gte`/`_lte` 대신 단일 컬럼(예: `date`)에 `date_gte`/`date_lte`를 붙여 범위 필터링  
**maxRangeValue**: 렌더러에서 종료일이 시작일 + N단위 초과 시 오류 표시 및 조정

### TipTap 툴바 구성

| 그룹 | 버튼 |
|---|---|
| 텍스트 스타일 | Heading(H1~H6), Bold, Italic, Strike, Underline |
| 삽입 | 수평선 |
| 목록 | 글머리, 번호, 들여쓰기, 내어쓰기 |
| 미디어 | 테이블, 이미지, 링크 |
| 꾸미기 | 형광펜, 텍스트 정렬(4종), 글자색, 위첨자, 아래첨자 |
| 동영상 | YouTube 임베드 |

### 렌더러 동작

```typescript
// FieldRenderer.tsx — 두 에디터 모두 SSR 불가, dynamic import 필수
const TiptapEditor = dynamic(() => import('@/components/common/tiptap-editor'), { ssr: false });
const WysiwygEditor = dynamic(() => import('@/components/common/wysiwyg-editor'), { ssr: false });

// 높이 계산: rowSpan × ROW_HEIGHT - (라벨 있으면 44, 없으면 24)
// 이 값이 에디터 컴포넌트의 height prop = 툴바 포함 전체 높이
const editorHeight = `${rowSpan * ROW_HEIGHT - (field.label ? 44 : 24)}px`;

const editorType = field.editorType ?? 'tiptap';
if (editorType === 'toast') {
    return <WysiwygEditor initialValue={value} onChange={...} height={editorHeight} />;
}
return <TiptapEditor initialValue={value} onChange={...} height={editorHeight} />;
```

> ⚠️ **SSR 절대 금지**: `dynamic import + ssr: false` 필수. 서버 컴포넌트에서 직접 import 시 빌드 오류 발생  
> ⚠️ **height prop = 툴바 포함 전체 높이**: TipTap은 내부적으로 `flex-col`로 툴바/콘텐츠 영역을 분리하므로 별도 보정 불필요  
> ⚠️ **이미지 업로드 API**: `/page-files/upload` 사용 (`api` 인스턴스 baseURL이 `/api/v1`이므로 `/api/v1/...` 형태로 중복 입력 금지)

### TipTap 확장 구성 참고

TipTap 에디터는 `StarterKit` + 개별 확장(Image·Youtube·Highlight·TextAlign·TextStyle·Color·
Superscript·Subscript·Table 계열)으로 구성된다. `StarterKit`은 기본적으로 Bold·Blockquote·
BulletList·Code·CodeBlock·Document·Dropcursor·Gapcursor·HardBreak·Heading·UndoRedo·
HorizontalRule·Italic·ListItem·ListKeymap·**Link**·OrderedList·Paragraph·Strike·Text·
**Underline**·TrailingNode를 포함한다.

> 새 확장을 추가하기 전에 위 목록을 먼저 확인한다. 이미 StarterKit에 포함된 확장(특히 Link·Underline)은
> 별도로 다시 추가하지 않고, 옵션 조정이 필요하면 `StarterKit.configure({ link: {...} })`처럼 StarterKit의
> 내장 옵션 경로로 전달한다.

```typescript
// activeDepths=[2,3]이면 index0=depth2, index1=depth3로 매핑된다(1부터 시작하는 고정 인덱스 아님)
// 1. 마운트 시 activeDepths[0] 옵션 즉시 로드
loadDepthOptions(0, null);   // eq_depth = activeDepths[0]

// 2. index idx 선택 시
const handleSelect = (idx, selectedValue) => {
    // 하위 index 선택값 + 옵션 전부 초기화
    onChange?.(selectedValue);
    if (selectedValue && idx < activeDepths.length - 1) {
        loadDepthOptions(idx + 1, selectedValue);   // eq_depth = activeDepths[idx+1]
    }
};
```

**API 호출**: `GET /page-data/{dbSlug}?eq_depth={activeDepths[idx]}&eq_parentId={상위선택값}`

### 17-1. 자기참조(self-reference) FILTER relation — "이 카테고리 자기 자신을 계층으로 검색"

`relationSlugId`(연동 Slug)로 등록하는 FILTER relation은 원래 "다른 slug(A)의 검색값으로 이 카테고리(B)를 필터링"하는 용도(예: product-data 검색 화면에서 카테고리를 고르면 그 카테고리 하위 제품만 보이게)로 설계됐다. 그런데 **카테고리 필드 자신이 속한 검색 위젯의 결과 테이블이 카테고리 데이터(B) 자기 자신을 기준으로 조회하는 경우**(예: 결과 테이블의 `connectedSlug`가 카테고리 데이터 slug 그 자체인 경우), 기존 relation의 `master_slug`(A)와 실제 조회 대상(B)이 달라서 검색이 깨진다.

**해결책**: `master_slug`와 `slave_slug`를 **동일하게**(자기 자신) 등록한 신규 FILTER relation을 만들면 된다. BE `resolveFilterRelationIds`(`PageDataService.java`)는:
- 계층 펼치기(하위 카테고리 id 수집) 로직은 slaveKey 경로에서 parentId를 자동 인식하므로 자기참조 여부와 무관하게 동일하게 동작한다.
- **마스터 id 추출**만 자기참조 여부로 분기한다 — `master_slug != slave_slug`(기존)면 `extractField(dataJson, slaveKey)`(예: `product.id`)로 마스터 id를 뽑고, `master_slug == slave_slug`(자기참조)면 **그 행 자신의 page_data id(네이티브 `id` 컬럼)**를 마스터 id로 쓴다. 이 분기가 없으면 자기참조 relation에서 엉뚱한 값(예: 제품 자신의 id)이 마스터 id로 잘못 쓰인다.

등록 예시(제품 카테고리 계층 검색을 category-data 자기 자신 기준으로 복구한 실제 사례):
```
master_slug = category-data
slave_slug  = category-data   ← 자기 자신
master_key  = id              ← 참고용, resolveFilterRelationIds가 실제로 읽지는 않음
slave_key   = product.id      ← 리프 판정(depth3 제외) + parentId 경로 유도 + 계층 순회 기준
relation_dir = FILTER
slave_type   = TABLE
category_depth = 1
slave_filter = depth=3        ← 리프(제품) 레코드만 대상
```

### 17-2. relationSlugId(FILTER) vs optionFilterRelationSlugId(FETCH) — 절대 혼동 금지

| | `relationSlugId`(연동 Slug) | `optionFilterRelationSlugId`(옵션 사전필터) |
|---|---|---|
| relation 방향 | **FILTER** | **FETCH** |
| 용도 | 카테고리 목록 자체를 계층으로 펼쳐서 필터링 | 화면에 안 보이는 depth의 연결 데이터로 옵션 집합을 사전에 좁힘 |
| 검색 시 나가는 파라미터 | `rel_{relationSlugId}=선택값` (`useWidgetPageState.ts`가 `type==='category' && relationSlugId`일 때만 생성) | 검색 파라미터에 안 실림 — 옵션 목록 계산에만 쓰임(마운트 시 1회) |
| 잘못 섞으면 | 옵션 사전필터용 FETCH relation id를 `relationSlugId`에 넣으면, 검색 시 `rel_{id}=선택값`이 엉뚱하게 나가서 **결과 테이블 검색 자체가 깨진다** |

```typescript
// 버튼 색상: 정적 맵 사용 (동적 Tailwind 클래스 금지)
const BG_COLOR_MAP = {
    black: 'bg-slate-900', green: 'bg-green-600', blue: 'bg-blue-600',
    yellow: 'bg-yellow-500', red: 'bg-red-600', gray: 'bg-slate-500', pink: 'bg-pink-500'
};

const handleButtonClick = async (field) => {
    // saveConfirm: 먼저 confirm 창 표시
    if (field.saveConfirm && !confirm('저장하시겠습니까?')) return;

    switch (field.connType) {
        case 'content':
            // onContentAction(widgetIds, 'save' | 'delete') 호출
            onContentAction?.(field.connectedContentWidgetIds, field.contentAction);
            if (field.goBackAfterAction) router.back();
            break;
        case 'close':
            onClose?.() ?? router.back();
            break;
        case 'popup':
            onPopupOpen?.(field.popupSlug, parseParams(field.params));
            break;
        case 'excel':
            // excelPrivacyPopup=true면 사유 입력 팝업 → 확인 후 다운로드
            // excelPrivacyPopup=false/undefined면 바로 다운로드
            // 입력된 reason은 /export?reason= 파라미터로 전달 → download_log 테이블에 비동기 저장
            onExcelDownload?.(field.excelTableWidgetId, field.excelPrivacyPopup);
            break;
        case 'datasave':
            await onDataSave?.(field.connectedContentWidgetIds, field.dataSaveSlug, ...);
            if (field.goBackAfterAction) router.back();
            break;
        case 'path':
            router.push(field.fileLayerSlug);
            break;
        case 'api':
            // 실제 HTTP 호출 로직은 이 스위치문 안에 없다 — onApiCall(콜백)만 호출한다.
            // 호출 로직 본체는 useWidgetPageState.ts의 handleApiCall(apiInfoId, params)에 있다
            // (GET /api-infos/active 캐시 매칭 → path 변수 치환 → method별 분기 → toast).
            // 상세는 06.builder_entity_to_api_guide.md §8 참고.
            onApiCall?.(field.apiInfoId, field.params);
            break;
    }
};
```

**연결 타입 변경 시 전체 초기화**: `connType` 변경 → 연결 속성 전부 `undefined`로 리셋 (`apiInfoId` 포함)

## 20. DateRangeStatusColumnField

**파일:** `builder/fields/DateRangeStatusColumnField.tsx`  
**용도:** 테이블 셀에서 날짜 범위 데이터를 "예정 / 진행중 / 종료" 텍스트로 자동 표시  
**사용 위젯:** Table

---

### 동작 개요

API가 반환한 `{accessor}_from` / `{accessor}_to` 두 값을 읽어, **현재 시각**과 비교하여 상태를 판정한다.  
비교 기준 포맷은 `linkedRangeSubType`이 결정한다.

```
API 응답 예시
  period_from: "2024-01-01"
  period_to:   "2024-12-31"

  → TableCellRenderer가 today(=YYYY-MM-DD)와 비교
  → "2024-01-01 ≤ today ≤ 2024-12-31" 이면 "진행중" 표시
```

---

### 빌더 설정 순서

#### STEP 1 — 같은 테이블에 dateRange 컬럼이 있어야 한다
데이터 accessor가 `period`라면 API는 `period_from` / `period_to`를 반환해야 한다.  
해당 dateRange 컬럼의 `rangeSubType`을 확인해둔다.

#### STEP 2 — dateRangeStatus 컬럼 추가
테이블 컬럼 추가 시 셀 타입 `dateRangeStatus` 선택

#### STEP 3 — 연결 설정 (필수)

| 항목 | 설명 |
|---|---|
| **연결 dateRange 컬럼 Key** | dateRange 컬럼의 accessor 입력 (예: `period`) |
| **날짜 범위 서브타입** | 연결된 dateRange 컬럼의 `rangeSubType`과 **동일하게** 선택 |
| **이전/포함/이후 텍스트** | 각 상태에 표시할 텍스트 입력 (기본값: 예정 / 진행중 / 종료) |

> ⚠️ `날짜 범위 서브타입`이 dateRange 컬럼의 `rangeSubType`과 다르면 비교 포맷이 맞지 않아 **항상 오판정**된다.

---

### 상태 판정 기준

**Table 컬럼 live 표시**: FE(`TableCellRenderer.tsx`)가 `_from`/`_to` 값을
`formatNowBySubType(linkedRangeSubType)`(`make/_shared/utils.ts`)로 만든 "지금" 문자열과
비교해 판정한다 (시:분 단위).

**Search 위젯 drs_ 필터**: BE(`PageDataService`)가 `_from`/`_to`의 앞 8자리(YYYYMMDD)만
오늘 날짜와 비교한다 (일 단위, 별도 로직 — 정밀도가 다르다).

"지금" 값은 반드시 `formatNowBySubType()`을 재사용한다 (직접 조립 금지).

| rangeSubType | 비교 기준 | from/to 값 형태 |
|---|---|---|
| `date` (기본) | 오늘 날짜 `YYYY-MM-DD` | `2024-07-01` |
| `yearMonth` | 이번 년월 `YYYY-MM` | `2024-07` |
| `datetime` | 현재 날짜+시각 `YYYY-MM-DDTHH:mm` | `2024-07-01T14:30` |
| `time` | 현재 시각 `HH:mm` | `14:30` |
| `timeSec` | 현재 시각 `HH:mm:ss` | `14:30:00` |

**판정 결과**:
- `현재 < from` → **이전 텍스트** (예: 예정)
- `from ≤ 현재 ≤ to` → **포함 텍스트** (예: 진행중)
- `현재 > to` → **이후 텍스트** (예: 종료)
- from / to 모두 없음 → `-`

---

| `linkedDateRangeKey` | string | - | 연결된 dateRange 컬럼의 accessor (필수) |
| `linkedRangeSubType` | `'date' \| 'yearMonth' \| 'datetime' \| 'time' \| 'timeSec'` | `'date'` | 상태 판정 포맷 — dateRange 컬럼의 rangeSubType과 일치시킬 것 |
| `beforeText` | string | `'예정'` | 날짜 이전 표시 텍스트 |

**Google 연동 — 저수준 API만 사용, UI는 자체 렌더링**:
- Google이 기본 제공하는 자동완성 위젯(구글 자체 스타일)은 사용하지 않는다. `AutocompleteService.getPlacePredictions`(후보 검색)와 `PlacesService.getDetails`(선택 시 상세정보+좌표 조회) 저수준 API만 호출하고, 드롭다운 UI는 `select` 타입의 autocomplete가 쓰는 `PortalDropdown` 골격을 그대로 재사용해 그린다 — preview 모드(목데이터 드롭다운)와 live 모드가 동일한 시각 스타일을 갖는다 (Rule 10-3).
- 검색어 입력마다 300ms 디바운스(`debounce`, `utils.ts`) 후 후보 조회. 후보 선택 시 상세정보(좌표 포함) 비동기 조회 후 `onAddressSelect(address, lat, lng)` 콜백 1회 호출로 3값을 동시에 확정한다(단일 `onChange`로는 3값을 원자적으로 전달할 수 없어 `onFromChange`/`onToChange`처럼 전용 콜백을 신설).
- 관련 유틸: `make/_shared/utils/googlePlaces.ts` (`searchAddressPredictions`, `getAddressDetail` — 둘 다 `language: 'ko'|'en'` 파라미터로 `field.addressLanguage`를 그대로 전달받음)

**필수 설정**:
- 신규 의존성 `@googlemaps/js-api-loader` (공식 구글 npm 패키지, `@types/google.maps`를 자체 dependencies로 포함)
- 환경변수 `NEXT_PUBLIC_GOOGLE_MAPS_KEY` — bo `.env.local`/`.env.development`에 필요. bo-api `application-local.yml`의 `ls.lse.outApi.googleMapKey`와 동일 값(브라우저에서 직접 로드하는 클라이언트 키이므로 Google Cloud Console에서 HTTP 리퍼러(도메인) 제한 설정 필요 — IP 제한 방식은 브라우저 호출이 막힐 수 있음)

**preview 모드**: 실제 API 호출 없이 `field.options`(목데이터 주소 목록)를 드롭다운에 항상 펼쳐서 보여준다. 위경도는 preview에서도 노출하지 않는다.

## A. 공통 — DataGenerationSection

**파일:** `builder/fields/DataGenerationSection.tsx`  
**포함 필드:** InputField, FormTextareaField, EditorField

### 동작 원리

이 필드 값을 변환하여 **다른 필드에 자동 입력**하는 기능이다.  
예: 상품명 입력 → slug 필드에 URL-safe 값 자동 생성

### 변환 순서

```
원본 값
  → (1) stripHtml      — HTML 태그 제거 (editor 전용)
  → (2) dataReplacement — 공백·특수문자 → '-' 치환 (hyphen 선택 시)
  → (3) caseChange      — 대문자(upper) / 소문자(lower) 변환
  → (4) appendText      — 끝에 고정 문자열 추가
  → (5) truncateLength  — N자 미만으로 자르기
  → generationKey 경로의 대상 필드에 자동 입력
```

**예시**: `applyDataGeneration('Hello World!', 'hyphen', 'lower', '-doc', 20)`  
→ `'hello-world--doc'`

### DataGenerationEntry 속성

| 속성 | 타입 | 설명 |
|---|---|---|
| `generationKey` | string | 결과를 입력할 대상 필드 accessor (dot notation) |
| `onlyIfEmpty` | boolean | 대상 필드가 빈값인 경우에만 적용 |
| `dataReplacement` | `'none' \| 'hyphen'` | 공백·특수문자 → `-` 치환 |
| `caseChange` | `'none' \| 'upper' \| 'lower'` | 대/소문자 변환 |
| `appendText` | string | 변환 후 끝에 붙이는 텍스트 |
| `truncateLength` | number | N자 미만으로 자르기 |
| `stripHtml` | boolean | HTML 태그 제거 (editor만) |

### generationKey 경로 형식 (dot notation)

| 형식 | 예 | 의미 |
|---|---|---|
| `fieldKey` | `slug` | 동일 폼의 `slug` 필드 |
| `contentKey.fieldKey` | `form1.slug` | `contentKey=form1`인 폼의 `slug` |
| `tabKey.contentKey.fieldKey` | `tab1.form1.slug` | 탭 내부 폼의 `slug` |

### 세부 설정 UI

1. **섹션 헤더** — "데이터생성" + "추가" 버튼
2. **세트 목록** (GenerationEntryRow):
   - 생성 KEY input | 빈값일 때만 체크박스 (`onlyIfEmpty`)
   - `fieldType='input'`: 데이터변경 + 문자변경 + 텍스트추가 + 글자자르기
   - `fieldType='editor'`: HTML제거 + 글자자르기만

### 특수 기능

- **레거시 마이그레이션**: 구버전 `generationKey` 단일 속성 → 첫 로드 시 `dataGenerations[0]`으로 자동 변환
- **다중 세트**: "추가" 버튼으로 무제한 생성, 각 세트 독립적

## C. 공통 — SlugSelectField

**파일:** `builder/fields/SlugSelectField.tsx`  
**사용처:** CategoryField, ActionButtonField 내부

> 현재 SlugSelectField는 검색 필터 + Portal 드롭다운 동작을 직접 구현하지 않고, 제네릭 코어 [AutocompleteSelectField](#d-공통--autocompleteselectfield)(§D)를 감싸는 얇은 래퍼로 구현되어 있다. slug 도메인 전용 타입/포맷(값 = slug 문자열, 필터 = name/slug, 표시 = `"name (slug)"`)만 지정하고 나머지 UI/동작은 코어가 담당한다. 아래 Props는 래퍼가 노출하는 인터페이스다.

### Props

| prop | 타입 | 기본값 | 설명 |
|---|---|---|---|
| `value` | string \| null | - | 현재 선택된 slug |
| `slugOptions` | `{ id: string; slug: string; name: string }[]` | - | 선택 가능한 slug 목록 |
| `label` | string | `'연결 Slug'` | 라벨 텍스트 |
| `required` | boolean | false | 필수 표시 |
| `emptyLabel` | string | `'선택 안 함'` | 빈값 옵션 텍스트 |
| `formatDisplay` | `(opt) => string` | - | 선택 후 표시 텍스트 커스터마이징 |

### 렌더러 동작

```
검색어 타이핑 → name/slug 실시간 필터링 → 드롭다운 표시 (name (slug) 형식)
선택 → formatDisplay? 커스텀 텍스트 : 기본 "name (slug)"
선택 해제 → ✕ 버튼 클릭 → value = null
Escape 키 → 입력창 원상복구 (선택값 유지)
외부 클릭 → 드롭다운 닫기
```

## D. 공통 — AutocompleteSelectField

**파일:** `builder/fields/AutocompleteSelectField.tsx`  
**사용처:** SlugSelectField(§C) · ApiInfoSelectField의 공통 코어 (도메인 컴포넌트는 이 코어를 감싸 사용, 직접 사용은 드묾)

"입력창 + Portal 드롭다운 + 검색 필터"라는 동일한 UI/동작을 제공하는 제네릭 코어다. 옵션 타입·선택 값 타입·옵션 필드명·표시 포맷만 제네릭 파라미터와 prop으로 주입받고, 나머지 렌더링·검색·위치 계산·외부클릭 처리는 코어가 공통으로 담당한다.

### 제네릭 파라미터

| 파라미터 | 의미 | 예 |
|---|---|---|
| `T` | 옵션 항목 타입 | `SlugOption`, `ApiInfoOption` |
| `V` | 선택 값 타입 (primitive) | `string`(slug), `number`(id) |

> 값 비교는 `getOptionValue(opt) === value`로 수행하므로 `V`는 `string`/`number` 같은 primitive여야 한다. "빈 값"은 `emptyValue` prop으로 지정한다(slug=`''`, apiInfo=`undefined`).

### Props (주요)

| prop | 타입 | 설명 |
|---|---|---|
| `value` | `V` | 현재 선택된 값 |
| `onChange` | `(value: V) => void` | 선택/해제 시 호출 — 해제 시 `emptyValue` 전달 |
| `options` | `T[]` | 옵션 목록 |
| `getOptionValue` | `(opt: T) => V` | 옵션 → 비교/저장 값 추출 |
| `getOptionKey` | `(opt: T) => React.Key` | 옵션 → React key |
| `getDisplayText` | `(opt: T) => string` | 옵션 → 입력창/목록 기본 표시 텍스트 |
| `filterOption` | `(opt: T, query: string) => boolean` | 검색 필터 (query는 소문자) |
| `emptyValue` | `V` | 해제 시 `onChange`에 전달할 빈 값 |
| `isEmpty` | `(value: V) => boolean` | 값이 비었는지 판정 — 미지정 시 `value === emptyValue` |
| `renderOption` | `(opt, selected) => ReactNode` | 목록 항목 커스텀 렌더링 — 미지정 시 `getDisplayText` |
| `label` / `required` / `emptyLabel` / `hideLabel` | - | 라벨/필수표시(`*`)/빈값 placeholder/라벨 생략 |

### 렌더러 동작

```
입력창 타이핑 → filterOption으로 실시간 필터 (타이핑 시작 시 기존 선택 해제)
포커스/타이핑 → 드롭다운 열림 (useDropdownPosition으로 top/left/width 계산 — input 폭 사용)
드롭다운은 createPortal로 body에 렌더링 → 모달·overflow 컨테이너에 가리지 않음
선택 → getOptionValue(opt) 저장, 입력창에 getDisplayText 표시
✕ 버튼 / 빈값 옵션 클릭 → emptyValue로 해제
Escape → 입력창 원복(선택값 유지)
외부 클릭 → useOutsideClick([container, dropdown]) 판정 후 닫기 + 원복
```

### 현재 구조 — 래퍼 2종

기존 두 도메인 컴포넌트는 이 코어를 감싸는 얇은 래퍼로 구현되어 있다.

| 래퍼 | 파일 | 값 타입 `V` | `emptyValue` | 필터 기준 | 표시 포맷 |
|---|---|---|---|---|---|
| `SlugSelectField`(§C) | `builder/fields/SlugSelectField.tsx` | `string`(slug) | `''` | name / slug | `"name (slug)"` |
| `ApiInfoSelectField` | `builder/fields/ApiInfoSelectField.tsx` | `number`(api_info.id) | `undefined` | name / method / urlPattern | `"METHOD name (urlPattern)"` |

> `ApiInfoSelectField`는 §18 ActionButtonField의 `connType='api'` 연결 UI에서 사용된다(사용중인 API만 검색·선택 → `apiInfoId` 저장). 두 래퍼 모두 도메인 전용 타입/포맷만 지정하고, 검색·Portal 드롭다운 UI/동작은 이 코어가 공통 제공한다.
