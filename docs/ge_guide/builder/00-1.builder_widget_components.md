## 0. 문서 목적

> **위젯 컴포넌트 = 컨텐츠 컴포넌트**

이 두 용어는 완전히 동일한 개념을 가리킨다. 코드에서는 `widget`, 문서·UI에서는 종종 `컨텐츠 컴포넌트`로 불리지만 같은 대상이다.

이 문서는 빌더 시스템에서 사용하는 **8종 위젯의 전체 구조**, 파일 매핑, 연결 관계, 새 위젯 추가 절차를 에이전트가 소스 탐색 없이 즉시 파악할 수 있도록 정리한다.

---

## 1. 왜 필요한가 (Why)

빌더 시스템에서 위젯은 총 8종이며, 각 위젯은 **Builder 파일**(설정 UI)과 **Renderer 파일**(출력 UI)로 분리된다. 이 두 쌍은 항상 함께 존재해야 하고, 두 개의 Dispatcher(`CommonBuilderDispatcher`, `WidgetRenderer`)에 동시에 등록되어야 작동한다.

이 구조를 모르면:
- Builder 파일만 만들고 Renderer는 만들지 않는 실수 발생
- Dispatcher 한 곳만 수정하여 한쪽에서만 동작하는 버그 발생
- 새 위젯을 추가하면서 `renderer/index.ts` export 누락으로 컴파일 에러 발생
- 위젯 간 연결 속성(connectedSearchIds, parentWidgetId 등)을 어디서 설정하는지 몰라 잘못된 위치에 코드 작성

## 2. 핵심 개념 (What)

### 2-1. 용어 정의

| 용어 | 정의 |
|---|---|
| 위젯 컴포넌트 | 빌더에서 배치 가능한 컨텐츠 단위. `type` 키로 식별 |
| 컨텐츠 컴포넌트 | 위젯 컴포넌트와 동의어 — 같은 개념의 다른 표현 |
| 필드 컴포넌트 | 위젯 내부 입력 요소 단위 (input, select, date 등). 위젯 컴포넌트와 다른 계층 |
| Builder 파일 | 위젯 설정 UI — 빌더 좌측 패널에 표시 |
| Renderer 파일 | 위젯 출력 UI — 빌더 미리보기 + 운영 페이지에 표시 |
| Dispatcher | widget.type에 따라 적절한 Builder/Renderer로 분기하는 switch/if 컴포넌트 |
| AnyWidget | 8종 위젯 타입의 TypeScript union — `renderer/types.ts`에 정의 |

### 2-2. 계층 구분

```
위젯 컴포넌트 (컨텐츠 컴포넌트)  ← 이 문서의 대상
└── 필드 컴포넌트                 ← 위젯 내부 입력 요소 (다른 계층)
    └── input, select, date, ...
```

❌ 위젯 컴포넌트와 필드 컴포넌트를 혼동하지 말 것. `FieldRenderer.tsx`, `SearchFieldRenderer.tsx`는 필드 컴포넌트 렌더러이며, 위젯 렌더러가 아니다.

## 5. 두 개의 Dispatcher (How)

위젯 컴포넌트를 사용하는 진입점은 반드시 2개다. 새 위젯 추가 시 두 곳을 **동시에** 수정해야 한다.

### 5-1. CommonBuilderDispatcher (빌더 진입점)

```typescript
// bo/src/app/admin/templates/make/_shared/components/builder/CommonBuilderDispatcher.tsx
export function CommonBuilderDispatcher({ widget, onChange, context }: CommonBuilderDispatcherProps) {
    switch (widget.type) {
        case 'search':    return <SearchWidgetBuilder widget={widget} onChange={w => onChange(w)} />;
        case 'table':     return <TableBuilder widget={widget} onChange={w => onChange(w)} searchWidgets={searchWidgets} slugOptions={slugOptions} />;
        case 'form':      return <FormBuilder widget={widget} onChange={w => onChange(w)} slugOptions={slugOptions} maxColSpan={maxColSpan} slugEntityFields={slugEntityFields} />;
        case 'space':     return <SpaceBuilder widget={widget} onChange={w => onChange(w)} pageTemplates={pageTemplates} contentWidgets={contentWidgets} formWidgets={formWidgets} actionButtonOnly={actionButtonOnly} maxColSpan={maxColSpan} slugOptions={slugOptions} />;
        case 'category':  return <CategoryBuilder widget={widget} onChange={w => onChange(w)} slugOptions={slugOptions} categoryWidgets={categoryWidgets} pageTemplates={pageTemplates} />;
        case 'sublist':   return <SubListBuilder widget={widget} onChange={w => onChange(w)} slugOptions={slugOptions} />;
        case 'multiselect': return <MultiSelectBuilder widget={widget} onChange={w => onChange(w)} slugOptions={slugOptions} />;
        case 'tab':       return <TabBuilder widget={widget} onChange={w => onChange(w)} pageTemplates={pageTemplates} />;
    }
}
```

`context` prop 구조에서 각 위젯 타입별 필수 항목:

| 위젯 type | 필수 context 항목 |
|---|---|
| `search` | 없음 |
| `table` | `slugOptions`, `searchWidgets` |
| `form` | `slugOptions` (우측 드로어 시 `maxColSpan: 2`) |
| `space` | `pageTemplates`, `contentWidgets` |
| `category` | `slugOptions`, `categoryWidgets`, `pageTemplates` |
| `sublist` | `slugOptions` |
| `multiselect` | `slugOptions` |
| `tab` | `pageTemplates` |

### 5-2. WidgetRenderer (렌더러 진입점)

```typescript
// bo/src/app/admin/templates/make/_shared/components/renderer/WidgetRenderer.tsx (line 1196~)
if (widget.type === 'search')      return <SearchRenderer ... />;
if (widget.type === 'table')       return <TableRenderer ... />;
if (widget.type === 'form')        return <FormRenderer ... />;
if (widget.type === 'space')       return <SpaceRenderer ... />;
if (widget.type === 'category')    return <CategoryRenderer ... />;
if (widget.type === 'sublist')     return <SubListRenderer ... />;
if (widget.type === 'multiselect') return <MultiSelectRenderer ... />;
if (widget.type === 'tab')         return <TabRenderer ... />;
```

`WidgetRenderer`는 preview 모드와 live 모드를 모두 처리한다. 개별 Renderer를 직접 사용하는 것은 절대 금지다.

## 6. 위젯 간 연결 관계 (What)

위젯들은 서로 독립적으로 동작하지 않는다. 특정 속성을 통해 연결된다.

### 6-1. 연결 매트릭스

| 연결 유형 | FROM 위젯 | TO 위젯 | 전달 속성 | 동작 |
|---|---|---|---|---|
| 검색 필터 | Search | Table | `Table.connectedSearchIds[]` | Search.accessor → API params |
| 폼 저장 | Space(content) | Form/SubList | `Space.item.connectedContentWidgetIds[]` | buildDataJson → POST/PUT |
| 팝업 열기 | Space(popup) | 대상 페이지 | `Space.item.popupSlug` | params → Layer 팝업 또는 페이지 이동 |
| 카테고리 계층 | Category(depth N) | Category(depth N-1) | `Category.parentWidgetId` | selectedId → eq_parentId 필터 |
| 데이터 저장 | Space(datasave) | Form/SubList/MultiSelect | `Space.item.dataSaveSlug` | 외부 slug에 통합 저장 |
| 엑셀 다운로드 | Space(excel) | Table | `Space.item.excelTableWidgetId` | connectedSlug + columns → export API |

### 6-2. Space 위젯 connType 목록

Space 위젯의 `action-button` 아이템은 `connType`으로 동작을 결정한다.

| connType | 동작 | 주요 연결 속성 |
|---|---|---|
| `content` | Form/SubList 저장·삭제 | `connectedContentWidgetIds[]`, `contentAction('save'/'delete')` |
| `popup` | Layer 팝업 열기 (outputMode에 따라 페이지 이동 가능) | `popupSlug`, `params` |
| `datasave` | 외부 slug에 데이터 저장 | `dataSaveSlug`, `connectedContentWidgetIds[]` |
| `excel` | 테이블 엑셀 다운로드 | `excelTableWidgetId` |
| `path` | 경로 이동 | `params` |
| `close` | 팝업 닫기 | - |

## 7. 실제 화면 패턴 (How)

### 패턴 1. 목록 화면

```
Search (colSpan:12) → 검색 필터 → Table.connectedSearchIds[]
Table  (colSpan:12) → columns[] + actions(edit/detail/delete)
Space  (colSpan:12) → 등록버튼(connType:'popup', popupSlug:'등록 slug')
```

배치 구조:
```
widgetItems[0]: SearchWidget  (colSpan:12)
widgetItems[1]: TableWidget   (colSpan:12, connectedSearchIds: [Search.widgetId])
widgetItems[2]: SpaceWidget   (align:'right')
```

### 패턴 2. 등록·수정 팝업 (quick-detail)

```
Form  (contentKey:"") → fields[] 입력
SubList (contentKey:"items") → 다건 행 편집
Space (align:right) → 저장버튼(connType:'content', contentAction:'save', connectedContentWidgetIds:[form.widgetId, sublist.widgetId])
                     → 취소버튼(connType:'close')
```

### 패턴 3. 카테고리 관리

```
Category(depth:1, colSpan:3)
Category(depth:2, colSpan:3, parentWidgetId: depth1.widgetId)
Category(depth:3, colSpan:3, parentWidgetId: depth2.widgetId)
Category(depth:4, colSpan:3, parentWidgetId: depth3.widgetId)
```

각 Category 위젯은 하나의 depth를 담당한다. `parentWidgetId`가 있으면 상위 선택 값을 `eq_parentId` 필터로 사용한다.

### 패턴 4. 탭 페이지

```
Tab.tabs[0] (required:true, pageSlug:'detail-slug')
  → items[]: Form + Space (저장 버튼)
Tab.tabs[1] (pageSlug:'detail-slug', contentKey:'tab2')
  → items[]: Form + Space (저장 버튼)
Tab.tabs[2] (pageSlug:'detail-slug', contentKey:'tab3')
  → items[]: SubList + MultiSelect + Space (저장 버튼)
```

같은 `pageSlug`를 사용하는 탭이 여러 개일 때 `contentKey`로 `dataJson` 내 섹션을 분리한다.

## 8-1. MultiSelect 위젯 상세 가이드

> 다음 에이전트가 MultiSelect 기능을 개선하거나 확장할 때 반드시 읽어야 할 내용이다.

### 전체 속성 목록

| 속성 | 타입 | 필수 | 역할 |
|---|---|---|---|
| `sourceSlug` | string | ✅ | 옵션 목록을 조회할 slug |
| `labelFields` | string | ✅ | 옵션 표시 텍스트 구성 필드. 쉼표 구분 |
| `contentKey` | string | ✅ | dataJson 저장 키 — 없으면 저장 안 됨 |
| `connectedSlug` | string | — | 저장 시 연결 slug |
| `title` / `titleMsgKey` | string | — | 위젯 상단 타이틀 |
| `description` / `descriptionMsgKey` | string | — | 설명 텍스트 |
| `placeholder` / `placeholderMsgKey` | string | — | 입력창 플레이스홀더 |
| `showBorder` | boolean | — | 테두리 표시 여부 |
| `bgColor` | string | — | 배경색 |
| `extraFields[]` | MultiSelectExtraField[] | — | 선택된 항목별 추가 입력 필드 |

### sourceSlug 옵션 로드 동작

```typescript
// MultiSelectRenderer.tsx — 옵션 로드 플로우
api.get(`/page-data/${widget.sourceSlug}`, { params: { size: 9999 } })
// 1. 전체 목록을 한 번에 로드 (페이징 없음 — 무조건 size=9999)
// 2. BE 응답: { content: [{ id: PK, dataJson: { 섹션: { 필드: 값 } } }, ...] }
// 3. buildTableRow로 nested dataJson을 flat 병합 (테이블 렌더러와 동일한 공통 패턴)
// 4. _id → id로 매핑 (PK는 BE 응답 최상위 id 필드)
```

**BE 응답 → 옵션 변환 예시** (`product-data` slug 기준)

```json
// BE 응답 원본
{ "id": 1, "dataJson": { "product": { "prdNm": "노트북 A", "price": 1200000 } } }

// buildTableRow 변환 후 (nested dataJson → flat 병합)
{ "_id": 1, "id": 1, "prdNm": "노트북 A", "price": 1200000, "product": { "prdNm": "노트북 A", "price": 1200000 } }

// labelFields = "prdNm"   → 표시: "노트북 A"
// labelFields = "prdNm,price" → 표시: "노트북 A > 1200000"
```

### labelFields 작성 규칙

| 규칙 | 설명 | 예시 |
|---|---|---|
| flat 변환 후 필드명 사용 | buildTableRow가 섹션을 flat 병합하므로 섹션명 불필요 | `prdNm` ✅ / `product.prdNm` ❌ |
| dot notation 지원 | 중첩 접근이 필요한 경우 dot 사용 가능 | `form1.title` |
| 쉼표로 복수 지정 | 여러 필드를 `" > "`로 연결해서 표시 | `prdNm,brandNm` → "노트북 A > 삼성" |

> ⚠️ **핵심**: `sourceSlug`의 dataJson이 `{ 섹션: { 필드 } }` 구조이면 `buildTableRow`가 자동 flat 병합한다.  
> `labelFields`에는 **섹션명 없이 필드명만** 입력한다. (`prdNm` O, `product.prdNm` X)

### dataJson 저장 구조

```json
// contentKey = "relatedProducts" 일 때 저장되는 dataJson
{
  "relatedProducts": [3, 7, 12]
}
```

- `contentKey`는 **필수**다. 없으면 dataJson에 저장되지 않는다.
- SubList와 달리 contentKey 없는 flat 저장은 지원하지 않는다.

### extraFields 구조

선택된 각 항목 태그 옆에 인라인으로 렌더링되는 추가 입력 필드다. `FieldRenderer`를 재사용한다.

```typescript
// renderer/types.ts — MultiSelectExtraField
interface MultiSelectExtraField {
    id: string;              // DnD 내부 식별자 (uid 자동생성, 변경 불가)
    key: string;             // 저장 키
    type: 'input' | 'date' | 'select' | 'radio' | 'checkbox';
    label: string;
    labelMsgKey?: string;
    options?: string[];      // select / radio / checkbox 전용
    codeGroupCode?: string;  // 공통코드 그룹 코드 (select/radio/checkbox에서 사용)
    required?: boolean;
    placeholder?: string;
    placeholderMsgKey?: string;
}
```

### preview vs live 모드 차이

| 항목 | preview 모드 | live 모드 |
|---|---|---|
| 옵션 데이터 | `PREVIEW_OPTIONS` 하드코딩 샘플 5개 | `sourceSlug` API 호출 |
| 선택 상태 | `PREVIEW_SELECTED_IDS [1, 3]` 고정 | `selectedIds` prop으로 외부 관리 |
| 드롭다운 | 닫힌 상태 고정 (disabled) | 클릭 시 열림 / 바깥 클릭 시 닫힘 |
| 검색 필터 | disabled | `labelFields` 기반 실시간 필터링 |

## 8-2. Category 위젯 드래그 순서 변경 가이드

> 다음 에이전트가 CategoryRenderer의 드래그 기능을 수정하거나,
> 유사한 드래그 순서 변경 기능을 구현할 때 반드시 읽어야 할 내용이다.

### 드래그 구현 방식

CategoryRenderer live 모드 드래그는 빌더(`DndWrappers.tsx`)의 `@dnd-kit`이 아닌
**브라우저 네이티브 HTML5 Drag & Drop API**로 구현되어 있다.

| 이벤트 | 핸들러 | 역할 |
|---|---|---|
| `onDragStart` | `handleDragStart(index)` | 시작 인덱스 → `dragIndexRef`에 기록 |
| `onDragOver` | `handleDragOver(e, index)` | 삽입 위치 갱신 + `e.preventDefault()` |
| `onDragLeave` | `handleDragLeave(e)` | 드롭 인디케이터 초기화 |
| `onDrop` | `handleDrop(e, index)` | 배열 재정렬 + BE sortOrder 저장 |

### 핵심 구현 규칙 3가지

#### 규칙 1. 드롭 인디케이터는 반드시 `position: absolute` 오버레이

```tsx
// ❌ 금지 — 형제 div 삽입 시 피드백 루프 발생 (항목 떨림)
// 인디케이터 등장 → 항목 8px 밀림 → dragLeave → 인디케이터 제거 → 복귀 → 무한반복
<div key={item.id}>
    {dropIndex === index && <div className="h-0.5 bg-blue-400 mb-1.5 mx-1" />}
    <div draggable ...>

// ✅ 올바른 방법 — absolute 오버레이: 레이아웃에 영향 없음
<div key={item.id} className="relative">
    {dropIndex === index && (
        <div className="absolute top-0 left-1 right-1 h-0.5 bg-blue-400 rounded z-10 pointer-events-none" />
    )}
    <div draggable ...>
```

#### 규칙 2. onDragLeave는 자식 요소 오탐 방지 체크 필수

```typescript
// ❌ 금지 — 카드 내 자식 요소(아이콘, 텍스트, 버튼) 경계마다 dragleave 발생 → 깜빡임
const handleDragLeave = () => { setDropIndex(null); };

// ✅ 올바른 방법
const handleDragLeave = (e: React.DragEvent) => {
    // 자식 요소 간 이동 시 오탐 방지 — 실제 컨테이너 밖으로 나갈 때만 초기화
    if (e.currentTarget.contains(e.relatedTarget as Node)) return;
    setDropIndex(null);
};
// 바인딩: onDragLeave={e => handleDragLeave(e)}  ← 이벤트 객체 전달 필수
```

#### 규칙 3. 순서 저장 시 반드시 ID 기준 비교

```typescript
// ❌ 금지 — 위치(index) 기준 비교: sortOrder가 1,2,3 이면 항상 같아 API 호출 안 됨
const changed = updated.filter((item, i) => item.sortOrder !== items[i]?.sortOrder);

// ✅ 올바른 방법 — ID 기준 비교
const originalSortMap = new Map(items.map(item => [item.id, item.sortOrder]));
const changed = updated.filter(item => item.sortOrder !== originalSortMap.get(item.id));
```

> ⚠️ 왜 이 버그가 보이지 않다 나타나는가:  
> DB sortOrder가 비순차(10, 20, 30)이면 비교가 달라져 저장됨.  
> 한 번 저장하면 1, 2, 3으로 정규화 → 이후 드래그 시 `changed` 항상 비어 저장 안 됨.

### onDragOver 중복 리렌더 방지

```typescript
const handleDragOver = (e: React.DragEvent, index: number) => {
    e.preventDefault();               // ← drop 허용 위해 반드시 먼저 호출
    if (dropIndex === index) return;  // 같은 인덱스 반복 시 불필요한 setState 차단
    setDropIndex(index);
};
```

> ⚠️ `e.preventDefault()`를 `if (dropIndex === index) return` **이전**에 호출해야 한다.  
> 순서가 바뀌면 `onDrop` 이벤트가 발생하지 않는 케이스가 생긴다.

## 10. 새 위젯 추가 체크리스트 (How)

새 위젯을 추가할 때 수정해야 할 파일은 7개다. 순서대로 진행한다.

| 순서 | 파일 | 작업 내용 |
|---|---|---|
| 1 | `builder/{Name}Builder.tsx` 신규 생성 | Widget 인터페이스 + 설정 UI 구현 |
| 2 | `renderer/{Name}Renderer.tsx` 신규 생성 | RendererContainer 사용 필수 |
| 3 | `renderer/types.ts` | `AnyWidget` union에 새 타입 추가 (FormWidget/TableWidget 아닌 경우 인터페이스도 이 파일에 정의) |
| 4 | `builder/CommonBuilderDispatcher.tsx` | `switch` case 추가 |
| 5 | `renderer/WidgetRenderer.tsx` | `if (widget.type === '...')` 분기 추가 |
| 6 | `builder-contents-layout/page.tsx` | 퍼블리싱 탭 추가 **(Rule 0 STEP 3 필수 — 승인 후 진행)** |
| 7 | `renderer/index.ts` | Renderer export 등록 |

> ⚠️ 6번 builder-contents-layout 추가는 사전 승인 필수 항목이다. Rule 0 참조.

## 11. 절대 금지 패턴

### 금지 1. SearchBuilder.tsx와 SearchWidgetBuilder.tsx 혼동

```typescript
// ❌ 금지 — 잘못된 파일명
import { SearchBuilder } from './SearchBuilder';

// ✅ 올바른 방법
// bo/.../builder/SearchWidgetBuilder.tsx
import { SearchWidgetBuilder } from './SearchWidgetBuilder';
```

`SearchBuilder.tsx`는 위젯 빌더가 아닌 별도 파일이다. 위젯 빌더는 반드시 `SearchWidgetBuilder.tsx`다.

### 금지 2. FieldRenderer를 위젯 렌더러로 혼동

```typescript
// ❌ 금지 — 필드 렌더러를 위젯 렌더러로 사용
import { FieldRenderer } from './FieldRenderer';
import { SearchFieldRenderer } from './SearchFieldRenderer';
// 위 두 파일은 필드 컴포넌트 렌더러 — 위젯 컴포넌트 렌더러가 아님

// ✅ 올바른 방법 — 위젯 렌더링은 WidgetRenderer 단일 진입점 사용
// bo/.../renderer/WidgetRenderer.tsx
import { WidgetRenderer } from '../renderer/WidgetRenderer';
<WidgetRenderer mode="preview" widget={widget} />
```

### 금지 3. FormWidget/TableWidget 타입을 types.ts에서 찾기

```typescript
// ❌ 금지 — types.ts에서 FormWidget 찾으려 하면 없음
import type { FormWidget } from '../renderer/types';

// ✅ 올바른 방법 — 각 Builder 파일에서 import
// bo/.../builder/FormBuilder.tsx
import type { FormWidget } from '../builder/FormBuilder';
// bo/.../builder/TableBuilder.tsx
import type { TableWidget } from '../builder/TableBuilder';
```

### 금지 4. Dispatcher 한 곳만 수정

```typescript
// ❌ 금지 — CommonBuilderDispatcher만 수정하고 WidgetRenderer 누락
// builder/CommonBuilderDispatcher.tsx에만 case 추가

// ✅ 올바른 방법 — 반드시 2곳 동시 수정
// 1. bo/.../builder/CommonBuilderDispatcher.tsx — switch case 추가
// 2. bo/.../renderer/WidgetRenderer.tsx — if 분기 추가
```

### 금지 5. 개별 Renderer 직접 사용

```typescript
// ❌ 금지 — 개별 Renderer 직접 import/사용
import { FormRenderer } from '../renderer/FormRenderer';
<FormRenderer mode="live" fields={...} ... />

// ✅ 올바른 방법 — WidgetRenderer 단일 진입점 경유
// bo/.../renderer/WidgetRenderer.tsx
import { WidgetRenderer } from '../renderer/WidgetRenderer';
<WidgetRenderer mode="live" widget={formWidget} dataSlug="my-slug" onRefresh={...} />
```

## 12. 혼동하기 쉬운 케이스

| 케이스 | 원인 | 해결 |
|---|---|---|
| `SearchWidgetBuilder`를 못 찾음 | `SearchBuilder.tsx`라는 파일이 따로 존재해서 혼동 | 파일명 `SearchWidgetBuilder.tsx` 정확히 확인 |
| `FormWidget` 타입이 `types.ts`에 없음 | FormWidget과 TableWidget은 Builder 파일에 정의된 예외 | `FormBuilder.tsx`에서 import |
| 위젯 추가 후 빌더에서만 동작하고 미리보기에서 안 됨 | `WidgetRenderer.tsx` if 분기 누락 | Dispatcher 2곳 동시 수정 확인 |
| 카테고리 depth 2가 depth 1 선택에 반응 안 함 | `parentWidgetId`를 depth 1의 `widgetId`로 설정하지 않음 | `CategoryWidget.parentWidgetId = depth1.widgetId` 설정 |
| Space 저장 버튼 클릭 시 아무 일도 안 일어남 | `connectedContentWidgetIds`에 Form/SubList widgetId 미설정 | `connType:'content'` 버튼에 `connectedContentWidgetIds` 배열 설정 |
| builder-contents-layout 없이 개발 진행 | Rule 0 STEP 3 위반 | 퍼블리싱 탭 먼저 추가하고 승인 후 진행 |
| renderer/index.ts export 누락 | 새 Renderer 추가 후 export 등록 빠뜨림 | `renderer/index.ts` 마지막에 export 추가 |

## 13. 검증 체크리스트

위젯 관련 작업 완료 후 아래를 전부 확인한다.

### 신규 위젯 추가 시

- [ ] `builder/{Name}Builder.tsx` 생성 완료
- [ ] `renderer/{Name}Renderer.tsx` 생성 완료 (RendererContainer 사용)
- [ ] `renderer/types.ts` AnyWidget union에 추가 완료
- [ ] `builder/CommonBuilderDispatcher.tsx` case 추가 완료
- [ ] `renderer/WidgetRenderer.tsx` if 분기 추가 완료
- [ ] `builder-contents-layout/page.tsx` 퍼블리싱 탭 추가 완료 (승인 후 진행)
- [ ] `renderer/index.ts` export 등록 완료
- [ ] 파일명이 `{Name}Builder.tsx` / `{Name}Renderer.tsx` 형식인가
- [ ] FormWidget/TableWidget이 아니라면 타입 인터페이스가 `renderer/types.ts`에 있는가

### 기존 위젯 수정 시

- [ ] Builder 파일과 Renderer 파일 양쪽 수정 여부 확인
- [ ] 인터페이스 변경 시 Dispatcher props 타입 영향도 확인
- [ ] `builder-contents-layout` 샘플 데이터와 타입 불일치 여부 확인

### 연결 관계 설정 시

- [ ] Search → Table: `Table.connectedSearchIds[]`에 Search.widgetId 포함 확인
- [ ] Space(content) → Form: `connectedContentWidgetIds[]`에 Form.widgetId 포함 확인
- [ ] Category 계층: `parentWidgetId`가 상위 depth Category의 `widgetId`와 일치 확인
