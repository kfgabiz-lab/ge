# Entity 기반 API 자동생성 → 빌더 연동 통합 가이드

> 작성일: 2026-07-11
> 대상: "Slug Entity 정의 → BE 파일생성 → API 등록 → Space 빌더 버튼 연동"까지 전체 흐름을
>       처음부터 끝까지 한 번에 파악해야 하는 모든 에이전트.
>       각 단계의 상세 스펙은 이 문서가 아니라 아래 4개 문서에 있다 — 이 문서는 그것들을

## 1. 개요

> **관련 개념 — "Data Entity" 타입과의 관계**: `OutputModePanel`의 빌더패널 "타입"에서
> `Slug`/`Slug Entity`는 여전히 page_data(SlugRegistry)를 그대로 쓰고, `Data Entity`만
> 이 문서가 다루는 entity 자체 생성 API(`/api/v1/hero-data` 등)를 실제로 연동하는 개념이다.
> 세 타입의 저장소·목적 비교표는 [04번 가이드 §0-1-1](./04.builder_slug_entity_guide.md#0-1-1-저장소는-그대로다--slug-entity--data-entity)에 있다.
> 이 문서(06번)가 다루는 흐름은 아래 두 갈래로 전부 완성되어 있다:
> - **Table/Form 등 컨텐츠 컴포넌트**가 entity API로 데이터를 직접 읽고 쓰는 것 — Table 읽기전용 조회,
>   Form 단건 create/update(날짜 왕복·FILE 필드 포함)까지 전부 구현·검증 완료. 런타임 분기 로직은
>   `bo/src/app/admin/templates/make/_shared/utils/entityApi.ts`(entityApiPath/normalizeEntityRow/
>   toEntityDateString/toEntityFieldName 등)에 모여 있고, `useWidgetPageState.ts`가 `connectedType`
>   기준으로 이 유틸과 기존 page_data 경로를 분기한다.
> - **Space 빌더 버튼**이 등록된 API를 실제로 호출하는 것 — §8 참고, 완료됨.

이 흐름은 4개의 서로 다른 화면·기능에 걸쳐 있다.

1. **Slug Entity 관리** (`/admin/settings/slug-entity`) — Entity/필드 정의 + 파일생성
2. **bo-api 재기동** — 신규 생성된 Java 파일/DB 테이블 반영
3. **API 정보 관리** (`/admin/system/api`) — API 동기화로 신규 컨트롤러 자동 등록
4. **Space 빌더** (위젯 빌더의 공간영역) — 버튼 '연결'에서 등록된 API를 선택해 연동

각 화면은 독립적으로 문서화되어 있지만(위 4개 문서), 실제로 신규 데이터 모델 하나를
"화면에서 쓸 수 있는 API 연동 버튼"까지 만들려면 4단계를 **순서대로** 거쳐야 한다.
이 문서는 그 순서와 각 단계 사이에 놓치기 쉬운 전제조건(선행 조건)을 하나로 정리한다.

## 2. 전체 흐름 다이어그램

```
[Slug Entity 관리] Entity + 필드 정의, table_name 입력
        │
        ▼
[파일생성] 버튼 → 미리보기 → 저장
        │  Entity/Request/Response/Repository/Service/Controller 6개 Java 파일 생성
        │  (Controller에 @ApiLinkedEntity 자동 부여)
        ▼
[bo-api 재기동]                                    ← 필수, 건너뛸 수 없음
        │  신규 클래스 컴파일 반영 + hibernate.ddl-auto:update로 DB 테이블 자동 생성
        ▼
[/admin/system/api] API 동기화 버튼
        │  Spring MVC 핸들러 스캔 → 신규 컨트롤러만 api_info 테이블에 추가 (active=true 기본)
        ▼
[Space 빌더] 버튼(action-button) '연결' → 'API 연동' 선택
        │  GET /api-infos/active 목록(사용중인 API만)에서 autocomplete로 선택
        │  선택은 **optional** — 비워두면(`— API 선택 —`) mode1, 채우면 mode2로 동작이 갈린다
        ▼
[운영화면] 버튼 클릭
        │  apiInfoId 없음(mode1) → 선택된 Form(main)/SubList(sub) entity를
        │  id기반 생성/수정 자동판단으로 순차 직접 호출(api_info 조회 없음)
        │
        │  apiInfoId 있음(mode2) → useWidgetPageState.ts의 handleApiCall(apiInfoId, params)이
        │  GET /api-infos/active 캐시에서 apiInfoId 매칭 → urlPattern의 /api/v1 접두사 제거 →
        │  {key} path 변수를 params(콤마구분 key='값')로 치환 → method별 분기(GET/DELETE=query,
        │  POST/PUT/PATCH=body)로 선택한 API 1건만 호출 → 성공/실패 toast
        ▼
(완료 — 모드별 상세는 §8 참고)
```

## 3. STEP 1 — Slug Entity 정의 + 파일생성

**화면:** `/admin/settings/slug-entity`

1. Entity 등록 (slug, 표시명, 설명) 후 필드 목록 구성
2. **DB 테이블명(`tableName`) 입력 — 이게 없으면 [파일생성] 버튼이 비활성화된다**
3. [파일생성] 클릭 → 6개 파일(+DDL) 미리보기 모달 → [저장]
4. 저장 시 실제 `bo-api` 소스 디렉토리에 파일이 기록된다 (기존 파일 있으면 자동 백업)

> ⚠️ **local 프로파일 전용 기능**이다. dev/운영 환경에서는 동작하지 않는다.

상세 스펙(생성 파일 6종 경로, key→camelCase 정규화 규칙 등)은 [`04.builder_slug_entity_guide.md`](./04.builder_slug_entity_guide.md) §10 참고.

## 4. STEP 2 — bo-api 재기동 (필수, 절대 생략 불가)

파일생성만으로는 **아무것도 실행되지 않는다.**

- 새로 생성된 Java 클래스(Entity/Controller 등)는 재기동해서 컴파일 반영해야 실제 API로 동작한다
- 새로 생성된 Entity의 DB 테이블은 `hibernate.ddl-auto: update`(local 프로파일)에 의해 **재기동 시점에** 자동 생성된다 — DDL을 따로 실행할 필요 없다
- 재기동 전까지는 STEP 3(API 동기화)에서 신규 컨트롤러가 스캔되지 않는다

## 5. STEP 3 — API 동기화

**화면:** `/admin/system/api`

재기동 후 **[API 동기화]** 버튼을 눌러야 신규 컨트롤러가 `api_info` 테이블에 등록된다.

- 이미 등록된 API는 건너뛰고, **신규 컨트롤러만** 추가됨 (added/skipped 카운트로 결과 표시)
- 신규 추가되는 API는 **`active=true`(사용)로 기본 등록**된다
- Controller에 `@ApiLinkedEntity("EntityName")`이 있으면 `connectedEntity` 컬럼에 자동 기록됨 (STEP1의 파일생성이 이 어노테이션을 자동으로 붙여준다)

목록에서 사용여부(active)는 토글로 즉시 켜고 끌 수 있으며, **active=false인 API는 STEP 4의 빌더 드롭다운에 노출되지 않는다.**

상세 스펙은 [`05.builder_api_sync_guide.md`](./05.builder_api_sync_guide.md) 참고.

## 6. STEP 4 — Space 빌더 버튼에서 'API 연동' 선택

**화면:** 위젯 빌더(Space 아이템 → action-button)

1. Space 아이템에 액션 버튼 추가
2. '연결(connType)' select에서 **'API 연동'** 선택
3. autocomplete selectbox(ApiInfoSelectField)에 STEP 3에서 활성화된 API 목록만 표시됨
4. 검색어로 필터링 후 선택 → 선택한 `api_info.id`가 `apiInfoId`로 저장됨(mode2). **선택하지 않고 `— API 선택 —`로 두면 mode1**(main/sub entity 자동 생성/수정)로 동작한다 — 모드별 동작은 §8 참고
5. 템플릿 저장 후 재진입해도 선택값 유지

상세 스펙(connType 유니온, ApiInfoSelectField 재사용 훅 구조 등)은 [`00-1.builder_widget_components.md`](./00-1.builder_widget_components.md) §6-2, [`00-3.builder_field_reference.md`](./00-3.builder_field_reference.md) §18 참고.

## 7. 실제 예시 — hero-data 엔티티 (이번 세션 실제 검증 사례)

1. Slug Entity `hero-data`에 필드(title, postDate, titleText, sub, btnText, btnUrl, orderNo, content, defaultVal 등) 정의, `tableName: hero_banner` 입력
2. [파일생성] → 미리보기에서 `HeroData.java`(Entity, `@Table(name="hero_banner")`), `HeroDataRequest/Response`, `HeroDataRepository`, `HeroDataService`, `HeroDataController`(`@ApiLinkedEntity("HeroData")`) 6개 확인 → 저장
3. `bo-api` 재기동 → `hero_banner` 테이블 자동 생성, `/api/v1/hero-data` 엔드포인트 활성화
4. `/admin/system/api`에서 [API 동기화] → `HeroDataController`가 신규 등록됨 (active=true)
5. Space 빌더 버튼 '연결' → 'API 연동' → 검색창에 'hero' 입력 → `GET /api/v1/hero-data` 선택
6. 템플릿 저장(`POST /page-templates`) → `configJson`에 `"connType":"api","apiInfoId":66` 기록 확인
7. 재진입 시 선택값(`GET /api/v1/hero-data`) 그대로 복원됨
8. 운영화면에서 버튼 클릭 → 실제로 `GET /api/v1/hero-data`가 호출되고 성공 토스트 표시. path 변수가 있는
   API(예: `GET /api/v1/hero-data/{id}`)를 선택하고 버튼 파라미터에 `id='5'`를 넣으면 실제 요청 URL이
   `GET /api/v1/hero-data/5`로 치환되어 나간다. 존재하지 않는 id면 BE의 한글 에러 메시지가 그대로
   에러 토스트로 노출된다("해당 데이터를 찾을 수 없습니다. id=5" 등).

## 8. Live 호출 구현 — 이중 모드(mode1/mode2)

**Space 버튼을 클릭하면 apiInfoId 선택 여부에 따라 서로 다른 두 가지 방식으로 저장/호출된다.**
구현 위치: `bo/src/app/admin/templates/make/_shared/hooks/useWidgetPageState.ts` → `handleApiCall(apiInfoId?: number, params?, connectedContentWidgetIds?)`.

### 8-1. 모드 결정표

| apiInfoId | 모드 | 동작 | 대상 컨텐츠위젯 |
|---|---|---|---|
| 없음(`— API 선택 —`) | **mode1** — 자동 entity CRUD | 선택된 Form(main)/SubList(sub, connectedSlug가 main과 다름) 각각 **id 유무로 생성(POST)/수정(PUT) 자동판단**, api_info 조회 없이 entity 엔드포인트 직접 호출 | Form, SubList만 (MultiSelect는 체크박스 목록 자체에서 제외됨 — `ActionButtonField.tsx`) |
| 있음 | **mode2** — 단일 API 호출 | 선택한 API **1건만** 호출. 자동 체이닝 없음 | Form, SubList, MultiSelect (POST/PUT/PATCH일 때만 체크박스 노출) |

> ⚠️ **하위호환 주의**: 과거에는 apiInfoId를 선택하고 `apiInfo.connectedEntity`가 있으면 SubList까지 자동으로 순회 저장(체이닝)됐다. 이 동작은 제거되어 지금은 mode2가 단일 호출만 한다. 같은 자동 체이닝을 원하면 apiInfoId를 `— API 선택 —`으로 비워 mode1로 전환해야 한다.

### 8-2. mode1 — main/sub entity 자동 CRUD

- **main 식별**: 선택된 컨텐츠위젯 중 `type==='form'`.
- **sub 식별**: 선택된 컨텐츠위젯 중 `type==='sublist'`이면서 `connectedSlug`가 main과 다른 것.
- **main 저장**: 현재 로드된 레코드 id(페이지가 들고 있는 storedId/storedGroupId) 유무로 `PUT entityItemPath(mainSlug, id)` / 없으면 `POST entityApiPath(mainSlug)`. 응답 id를 parentId로 확보.
- **sub 저장**: sub의 각 row를 순회하며 `typeof row.id === 'number'`(기존 로드된 행, 실제 숫자 PK)면 `PUT entityItemPath(subSlug, row.id)`, 문자열 UUID(신규 추가행)면 `POST entityApiPath(subSlug)`. 저장 직전 row 내부의 UUID `id`는 제거하고 `{ [parentIdField]: parentId, ...나머지필드(camelCase) }`로 변환.
- **FK 필드는 빌드타임 지정**: 운영 화면(`widget/[slug]/page.tsx`)은 entity 필드 스키마를 조회하지 않으므로(`slugOptions`/entityId 매핑은 빌더 전용), `SubListWidget.parentIdField`(SubList 빌더의 "부모 연결 필드" selectbox)에 미리 지정해둔 값만 런타임에 읽는다. **미지정 SubList는 저장을 건너뛰고 `toast.warning`으로 알린다(조용히 무시하지 않음).**
- 바디 변환은 mode2와 동일하게 `buildEntityRequestBody`/`buildEntityDateFieldMeta`/`buildSubListEntityDateFieldMeta`(entityApi.ts) 공통함수를 재사용한다.
- **알려진 제약(수정 대상 아님)**: main/sub 저장은 무트랜잭션이다. main 성공 후 sub 일부/전체가 실패해도 main은 롤백되지 않는다(고아 레코드 가능). 실패 시 성공/실패 건수를 집계해 toast로 경고만 한다. 트랜잭션/재시도가 필요하면 별도 안건으로 다룰 것 — 이 문서의 구현 범위가 아니다.

### 8-3. mode2 — 단일 API 호출

- 매칭: `GET /api-infos/active` 캐시에서 `apiInfoId`로 조회. `GET /api-infos/{id}`(단건, 관리자 전용) 사용 금지.
- URL: `urlPattern`의 `/api/v1` 접두사 제거 → `{key}` path 변수를 `parseActionParams(params, {})` 결과로 치환(치환된 키는 나머지 파라미터에서 제거).
- method 분기: `GET`/`DELETE` → 나머지 파라미터를 query string. `POST`/`PUT`/`PATCH` → body.
- body 조립 규칙(POST/PUT/PATCH 한정):

  | 조건 | body |
  |---|---|
  | `connectedContentWidgetIds` 없음 | `restParams`만 |
  | 있음 + `apiInfo.connectedEntity` 없음(일반 외부 API) | `buildDataJson(...)` 결과(`contentKey` nested) + `restParams` |
  | 있음 + `apiInfo.connectedEntity` 있음(entity 연결 API) | `buildDataJson(..., isEntity:true)` → `buildEntityRequestBody(...)`(camelCase 변환) 거친 flat body + `restParams` |

- **`isEntity` 분기 이유**: 코드생성된 entity DTO(예: `TestDataRequest`)는 `SlugEntityCodeGenerator.toCamelCase(field.key())`로 필드명이 생성되므로(`SlugEntityCodeGenerator.java:171-173`), FE도 같은 규칙(`entityApi.ts`의 `toEntityFieldName`)을 거쳐야 필드명이 일치한다. 이 변환을 생략하면 `MALFORMED_JSON`(String 필드에 Object 주입, 또는 `Unrecognized field`) 400이 난다. `buildDataJson`의 `isEntity=true`는 "감싸지 않고 flat으로 쓴다"만 담당하고, 실제 camelCase 변환은 `buildEntityRequestBody`가 별도로 담당한다 — **두 단계 모두 필요**, 하나만 하면 여전히 깨진다.
- `_rel` 오분기 방지: `buildDataJson` 호출 시 `mainConnectedSlug`는 넘기지 않는다(`undefined`) — 이 값은 "현재 페이지 자체의 entity 연관" 개념이라 임의의 API 대상과는 무관하다.
- 자동 체이닝은 발동하지 않는다 — SubList가 선택돼도 그 row들은 위 body 조립 규칙 안에서(nested 또는 flat) 한 번에 실려 나갈 뿐, 별도 엔드포인트로 순회 호출되지 않는다.

### 8-4. 공통 배선

- `PageGridRenderer.tsx` → `WidgetRenderer.tsx` → `SpaceRenderer.tsx` — `onApiCall(apiInfoId?: number, params?, connectedContentWidgetIds?)`를 그대로 전달만 한다(3개 파일 시그니처 동일해야 함). `WidgetRenderer`가 `mode==='live'`일 때만 전달 + `SpaceRenderer`의 preview early-return 가드로 이중 방어.
- `ActionButtonField.tsx` — `connType==='api'` 블록: `ApiInfoSelectField`(선택 안 해도 됨) + (apiInfoId 선택시에만) `params` 자유입력(`datasave`/`popup`/`path`와 동일 컨벤션) + 컨텐츠위젯 체크박스(모드별 대상은 8-1 표 참고, POST/PUT/PATCH 선택 시에만 노출 — `supportsContentData` 가드).

## 9. 체크리스트 — 자주 놓치는 지점

- [ ] Entity에 `tableName`을 입력했는가? (없으면 파일생성 버튼 비활성화)
- [ ] 파일생성 저장 후 **bo-api를 재기동했는가?** (안 하면 API 동기화에서 신규 컨트롤러가 안 잡힘 — `compileJava`만으로는 떠 있는 프로세스에 반영 안 됨, 프로세스 자체를 다시 띄워야 함)
- [ ] `/admin/system/api`에서 **API 동기화를 눌렀는가?** (파일생성만으로는 api_info에 등록 안 됨)
- [ ] 동기화된 API가 **사용(active=true) 상태인가?** (미사용이면 빌더 드롭다운에 안 보임)
- [ ] local 프로파일에서 작업 중인가? (파일생성 기능은 dev/운영에서 차단됨)
- [ ] entity 연결 API(mode2, apiInfoId 선택)에 Form 컨텐츠를 연결했는가? → 필드 key 표기(snake_case든 camelCase든) 그대로 둬도 된다. `handleApiCall`이 자동으로 codegen 규칙과 동일하게 camelCase 변환한다.
- [ ] mode1(apiInfoId 미선택)에서 SubList를 연결했는가? → main과 다른 entity라면 **"부모 연결 필드"(parentIdField)를 반드시 지정**해야 저장된다. 안 하면 조용히 스킵되고 경고 토스트만 뜬다.
- [ ] mode1에서 MultiSelect를 연결하려 하지 않았는가? → mode1은 Form/SubList만 지원한다(체크박스 목록에도 안 뜬다). MultiSelect가 필요하면 mode2(apiInfoId 선택 + BE에 별도로 만든 API)로 전환해야 한다.
