## 0. ⭐ 핵심 아키텍처 원칙 — 반드시 숙지

### 0-1. slug = 데이터 네임스페이스

`page_data` 테이블의 모든 데이터는 `template_slug` 컬럼으로 격리된다.  
하나의 운영 페이지에서 여러 위젯이 각자 다른 `connectedSlug`를 가질 수 있다.

```
/admin/widgetSub/user-form
  └── Form 위젯 (connectedSlug: 'user-info')     → page_data WHERE template_slug = 'user-info'
  └── SubList 위젯 (connectedSlug: 'user-roles') → page_data WHERE template_slug = 'user-roles'
  └── Table 위젯 (connectedSlug: 'user-list')    → page_data WHERE template_slug = 'user-list'
```

> **slug 없이는 어떤 저장도 불가능하다.** `connectedSlug`가 비어있으면 해당 위젯의 저장 로직은 전부 건너뜀.

### 0-2. widgetId = FE 상태 키

동일 타입의 위젯이 한 페이지에 여러 개 있어도 `widgetId`로 구분한다.  
FE의 모든 상태 맵(formValuesMap, tableDataMap, …)은 widgetId를 키로 사용한다.

```typescript
formValuesMap[widgetId]        // 해당 Form 위젯의 입력값
tableDataMap[widgetId]         // 해당 Table 위젯의 데이터
subListRowsMap[widgetId]       // 해당 SubList 위젯의 행 목록
```

### 0-3. contentKey = dataJson 내부 섹션 구분자

하나의 `page_data` 레코드에 여러 위젯 데이터를 통합 저장할 때 `contentKey`로 섹션을 분리한다.

```json
{
    "id": 1,
    "userInfo": { "name": "홍길동", "email": "hong@test.com" },
    "orderHistory": { "rows": [{ "product": "노트북", "qty": 1 }] },
    "relatedTags": [3, 7, 12]
}
```

| contentKey 유무 | dataJson 저장 방식 |
|---|---|
| `contentKey` 없음 | 필드값을 최상위에 flat하게 저장 |
| `contentKey` 있음 | `{ [contentKey]: { 필드값들 } }` 중첩 구조로 저장 |

### 0-4. groupId = 다중 slug 묶음 키

한 페이지에서 서로 다른 `connectedSlug`에 동시 저장할 때 FE에서 UUID를 생성하여 모든 레코드에 동일한 `groupId`를 심는다.  
수정/삭제 시 `groupId`만 알면 연관 레코드 전체를 한 번에 처리할 수 있다.

```
신규 저장 (다중 slug):
  crypto.randomUUID() → groupId 생성
  POST /page-data/user-info    { dataJson, groupId: "uuid-abc" }
  POST /page-data/user-roles   { dataJson, groupId: "uuid-abc" }
  POST /page-data/user-address { dataJson, groupId: "uuid-abc" }

수정 모드 진입:
  URL: ?group_id=uuid-abc
  GET /page-data/user-info/group/uuid-abc
  GET /page-data/user-roles/group/uuid-abc
  GET /page-data/user-address/group/uuid-abc

삭제:
  DELETE /page-data/user-info/group/uuid-abc  (← 첫 번째 slug만 호출, BE에서 전체 삭제)
```

## 1. 데이터 모델 (dataJson 구조)

### 1-1. 단순 구조 (Form 단일, contentKey 없음)

```json
{
    "id": 1,
    "name": "홍길동",
    "email": "hong@test.com",
    "status": "active",
    "profileImage": [101, 102],
    "createdAt": "2026-05-27T10:00:00"
}
```

> - `id`는 BE에서 INSERT 후 자동 주입 (FE에서 보내지 않아도 됨)
> - 파일 필드(`file` / `image`)는 파일 ID 숫자 배열로 저장
> - `createdAt` 등 감사 컬럼은 테이블 조회 시 FE에서 flat하게 병합

### 1-2. 중첩 구조 (Form + SubList, contentKey 있음)

```json
{
    "id": 1,
    "userInfo": {
        "name": "홍길동",
        "email": "hong@test.com",
        "profileImage": [101]
    },
    "orderHistory": {
        "rows": [
            { "product": "노트북", "qty": 1, "attachFile": [201] },
            { "product": "마우스", "qty": 2, "attachFile": [] }
        ]
    },
    "relatedTags": [3, 7, 12]
}
```

> - `userInfo` : Form 위젯의 contentKey
> - `orderHistory` : SubList 위젯의 contentKey
> - `relatedTags` : MultiSelect 위젯의 contentKey (ID 배열)

### 1-3. BE INSERT 후 id 자동 주입

```java
// PageDataService.create() — INSERT 완료 후 dataJson에 id 즉시 추가
Map<String, Object> dataJsonWithId = new LinkedHashMap<>(request.getDataJson());
dataJsonWithId.put("id", newId);
UPDATE page_data SET data_json = CAST(:dataJson AS jsonb) WHERE id = :id;
```

> **이유**: 카테고리 위젯처럼 자신의 id를 참조해야 하는 케이스 지원.  
> FE에서 id를 보내지 않아도 저장 후 항상 `data_json` 안에 id가 포함된다.

## 2. 데이터 호출 흐름 (읽기)

### 2-1. 페이지 초기 로딩 순서

```
[브라우저] /admin/widgetSub/{slug} 진입
     │
     ▼ useEffect (마운트 1회)
     │
     ├─① fetchGroups()
     │     GET /api/v1/code-groups         ← 공통코드 전체 캐시 (select/radio 옵션용)
     │
     ├─② GET /api/v1/page-templates/by-slug/{slug}
     │     └── configJson 파싱 → widgetItems[] → setWidgetItems()
     │
     └─③ flatWidgets() → type==='table' 위젯 추출
              └── fetchTableData() (위젯별 병렬 실행)
                   └── GET /api/v1/page-data/{connectedSlug}
                       ?page=0&size=10
                       └── setTableDataMap(wid, rows)
```

### 2-2. 수정 모드 데이터 복원

URL 파라미터에 따라 3가지 모드로 동작한다.

```
URL 파라미터       진입 모드        동작
─────────────────────────────────────────────────
?id=123           단일 수정 모드   단일 Form connectedSlug로 1건 조회
?group_id=uuid    다중 수정 모드   slug별로 개별 API 조회 후 각 위젯에 복원
(없음)            신규 입력 모드   hidden defaultValue + URL params 적용
```

#### 단일 수정 모드 (`?id=123`)

```typescript
// 첫 번째 Form 위젯의 connectedSlug 사용
GET /api/v1/page-data/{connectedSlug}/123
└── restoreFromDataJson(dataJson, formWidgets, sublistWidgets, multiSelectWidgets)
```

#### 다중 수정 모드 (`?group_id=uuid`)

```typescript
// 페이지의 모든 Form+SubList+MultiSelect 위젯에서 connectedSlug 수집
const slugSet = new Set([
    ...formWidgets.map(fw => fw.connectedSlug),
    ...sublistWidgets.map(sw => sw.connectedSlug),
    ...multiSelectWidgets.map(mw => mw.connectedSlug),
].filter(Boolean));

// slug별 개별 조회 후 복원
slugSet.forEach(s => {
    GET /api/v1/page-data/{s}/group/{groupId}
    └── restoreFromDataJson(dataJson, slugForms, slugSublists, slugMultiSelects)
});
```

#### `restoreFromDataJson` 내부 동작

```
dataJson 수신
  │
  ├── Form 위젯별
  │    contentKey 있으면 dataJson[contentKey] 섹션에서 필드 매핑
  │    contentKey 없으면 dataJson 최상위에서 fieldKey 기준 매핑
  │    → setFormValuesMap(widgetId, { fieldId: 값 })
  │
  ├── SubList 위젯별
  │    dataJson[contentKey].rows → _rowId 부여 → setSubListRowsMap()
  │
  ├── MultiSelect 위젯별
  │    dataJson[contentKey] (number[]) → setMultiSelectValuesMap()
  │
  └── 파일 메타 로드
       └── dataJson 재귀 순회 → 숫자 배열 → fileIds 수집
            GET /api/v1/page-files/meta?ids=1,2,3
            └── setExistingFileMetaMap(widgetId, { fieldId: [{id, origName, fileSize}] })
                 + image 타입 필드: GET /api/v1/page-files/{id} (blob) → setImgBlobUrls()
```

#### 신규 모드 초기화

```typescript
// hidden 필드: URL 파라미터 우선, 없으면 defaultValue
fw.fields.forEach(f => {
    if (f.type !== 'hidden') return;
    const key = f.fieldKey || f.label || '';
    const urlVal = key ? searchParams.get(key) : null;
    vals[f.id] = urlVal ?? f.defaultValue ?? '';
});
```

> **활용 예시**: 카테고리 등록 팝업에서 `?depth=2&parentId=5`를 전달하면 hidden 필드에 자동 세팅.

### 2-3. 검색 흐름

```
[검색 버튼 클릭]
  handleSearch(searchWidgetId)
       │
       ├── searchValuesRef.current 참조 (stale closure 방지)
       └── Table 위젯 중 connectedSearchIds에 searchWidgetId 포함된 것 찾기
            └── fetchTableData({ ..., sv: 현재검색값, page: 0 })
                 └── GET /page-data/{slug}?필드키=값&page=0&size=10
```

#### 검색 파라미터 → BE 동적 쿼리 매핑

```
FE 파라미터 형식              BE 쿼리 조건
─────────────────────────────────────────────────────────────────────
name=홍길동                  data_json->>'name' ILIKE '%홍길동%'
eq_status=active             data_json->>'status' = 'active'
createdAt=2026-01-01~2026-12-31
  → createdAt_start/end      data_json->>'createdAt' >= '2026-01-01'
                              AND data_json->>'createdAt' <= '2026-12-31'
```

> - 일반 파라미터 (`name=값`): ILIKE 부분 일치 + contentKey 1단계 중첩도 동시 검색  
> - `eq_` 접두사: 정확 일치 (카테고리 parentId 필터 등에 사용)  
> - `~` 구분자: 범위 검색 (날짜 범위, 숫자 범위)

#### 검색 파라미터 SQL Injection 방지

```java
// 키 유효성 검증 — 영문자/숫자/언더스코어만 허용
if (!key.matches("[a-zA-Z0-9_]+")) return;
// eq_ 접두사도 동일 규칙 적용
String fieldKey = key.substring(3);
if (!fieldKey.matches("[a-zA-Z0-9_]+")) return;
```

### 2-4. 정렬·페이지이동·무한스크롤



| 동작 | 핸들러 | API |
|---|---|---|
| 정렬 클릭 | `handleSortChange(widgetId, accessor, dir)` | `?sort={accessor},{dir}&page=0` |
| 페이지 이동 | `handlePageChange(widgetId, page)` | `?page={n}&size=10` |

#### 무한스크롤 ref 패턴

```typescript
// tableDataMapRef — 콜백 클로저에서 최신 상태 참조 (stale closure 방지)
const tableDataMapRef = useRef<Record<string, PageTableData>>({});
useEffect(() => { tableDataMapRef.current = tableDataMap; }, [tableDataMap]);

const handleLoadMore = (tableWidgetId: string) => {
    const td = tableDataMapRef.current[tableWidgetId]; // ref 경유
    if (!td || !td.hasMore || td.loading || td.appendLoading) return;
    fetchTableData({ ..., page: td.nextPage, append: true });

```

### 2-5. popup 모드 데이터 복원 (`handleInternalPopupOpen`)

팝업 모드는 page 모드와 동일한 QUICK_DETAIL 템플릿을 레이어로 표시한다.  
표시 방식만 다를 뿐 데이터 읽기·저장 구조는 page 모드와 **완전히 동일**하다.

#### `fetchAndMapFieldValues` 내부 유틸 (WidgetRenderer.tsx)

```typescript
fetchAndMapFieldValues(
    connectedSlug: string,
    editId:        number | null,
    fields:        FormFieldItem[],
    initialValues?: Record<string, string>,  // URL 파라미터 초기값 (최우선 적용)
    contentKey?:    string,                  // 있으면 dataJson[contentKey] 섹션에서 읽기
): Promise<{ values, existingFileIds, sourceData }>
```

```
GET /page-data/{connectedSlug}/{editId}
 └── sourceData = dataJson

contentKey 유무에 따른 읽기 섹션 결정:
  없음 → section = sourceData              (flat root에서 읽기)
  있음 → section = sourceData[contentKey]  (중첩 섹션에서 읽기)

fields.forEach:
  file/image → existingFileIds[f.id] = section[fieldKey] as number[]
  hidden     → values[f.id] = section[fieldKey] ?? defaultValue
  그 외       → values[f.id] = String(section[fieldKey])

initialValues 있으면 → values[f.id] = initialValues[fieldKey]  (덮어쓰기)
```

#### `handleInternalPopupOpen` 전체 읽기 흐름

```
팝업 오픈 트리거 (카테고리 카드 버튼, 테이블 액션 버튼 등)
     │
     ▼
① 팝업 상태 전체 초기화
   popupFormValuesMap={}, popupFileValuesMap={}, popupExistingMetaMap={}, ...

② GET /page-templates/by-slug/{slug}
   → cfg: { widgetItems, outputMode, layerType, layerWidth, ... }

③ outputMode='page' 이면 → router.push('/admin/widgetSub/{slug}?id=...') 로 이동 (팝업 없이 페이지 전환)

④ form 위젯 목록 추출
   cfg.widgetItems.flatMap(contents).filter(type==='form')

⑤ [form 위젯별 반복] — widgetId별 분리 구조로 수집
   fetchAndMapFieldValues(connectedSlug, editId, fw.fields, initialValues, fw.contentKey)
   → formValuesAccum[fwWidgetId] = fwValues       ← 필드 값
   → fileIdsAccum[fwWidgetId]    = fwIds           ← 파일 ID 배열
   setPopupFormValuesMap(formValuesAccum)

⑥ SubList rows 복원
   sourceData[sw.contentKey].rows → _rowId 부여 → setPopupSubListRowsMap

⑦ 기존 파일 메타 조회 — widgetId별 구조로 저장
   GET /page-files/meta?ids=...
   → popupExistingMetaMap[widgetId][fieldId] = [{ id, origName, fileSize }]
   → image 타입: GET /page-files/{id} (blob) → popupImgBlobUrls[id] = blobUrl

⑧ setPopupCfg(cfg) → setPopupOpen(true)
```

#### page 모드 vs popup 모드 구조 비교

| 항목 | page 모드 (`widgetSub/[slug]/page.tsx`) | popup 모드 (`WidgetRenderer.tsx`) |
|---|---|---|
| 읽기 함수 | `restoreFromDataJson()` | `fetchAndMapFieldValues()` × form 위젯 수 |
| 값 상태 | `formValuesMap[widgetId][fieldId]` | `popupFormValuesMap[widgetId][fieldId]` |
| 파일 상태 | `fileValuesMap[widgetId][fieldId]` | `popupFileValuesMap[widgetId][fieldId]` |
| 기존 파일 메타 | `existingFileMetaMap[widgetId][fieldId]` | `popupExistingMetaMap[widgetId][fieldId]` |
| 저장 함수 | `handleContentAction` | `handlePopupContentAction` |
| dataJson 구성 | `buildDataJson` (공통) | `buildDataJson` (공통) |
| 다중 slug | O (groupId 지원) | X (단일 slug만) |
| 저장 후 동작 | `router.back()` 또는 `onRefresh()` | 팝업 닫기 + `onRefresh()` |

## 3. 데이터 저장 흐름 (쓰기)

### 3-1. handleContentAction 전체 순서 (`action: 'save'`)

```
[Space 버튼 클릭]
connectedContentWidgetIds = ['form-wid-1', 'sublist-wid-2', 'ms-wid-3']
action = 'save'
     │
     ▼
① 대상 위젯 수집
   allFlat에서 widgetId 매칭 → (Form | SubList | MultiSelect)[]

② slug별 그룹핑
   Map<connectedSlug, [widget1, widget2, ...]>
   ex: { 'user-info': [Form], 'user-roles': [SubList, MultiSelect] }

③ 유효성 검사 — Form 위젯 전체
   validateFormFields(fw.fields, formValuesMap[fw.widgetId], fileValuesMap[fw.widgetId], existingFileMetaMap[fw.widgetId])
   실패 시 → toast.warning + return

④ 다중 slug면 confirm 팝업 (신규 저장 시에만)

⑤ groupId 결정
   다중 slug 신규  → crypto.randomUUID()
   다중 slug 수정  → URL ?group_id 유지
   단일 slug       → undefined (groupId 없음)

⑥ [slug 그룹 반복]
   각 slug에 대해 아래 순서 실행:
   │
   ├─A. 파일 업로드 (Form의 file/image 필드)
   │    fileValuesMap[fw.widgetId][fieldId] → File[]
   │    POST /page-files/upload (파일 1개씩 순차 업로드)
   │    → newFileIdsByFieldId[fieldId] = [id1, id2]
   │
   ├─B. dataJson 구성 — `buildDataJson` 공통 함수 사용 (§3-6 참고)
   │    SubList 파일 업로드 → processedSubListRowsMap 완성 후 호출
   │    Form  : fieldKey → 값, 파일 필드 → [기존ID + 신규ID]
   │    SubList: contentKey → { rows: [...] }
   │    MultiSelect: contentKey → [선택 ID 배열]
   │    contentKey 있으면 중첩, 없으면 flat
   │
   ├─C. 수정 대상 id 결정
   │    group_id 있음 → GET /page-data/{slug}/group/{groupId} → id 추출
   │    id 있음       → URL ?id 파라미터 사용
   │    없음          → 신규 (POST)
   │
   ├─D. 저장 API 호출
   │    신규: POST /page-data/{slug} { dataJson, pkKeys?, groupId? }
   │    수정: PUT  /page-data/{slug}/{id} { dataJson }
   │
   ├─E. 파일 dataId 연결
   │    새로 업로드된 파일 ID들이 있으면:
   │    PATCH /page-files/link { fileIds: [...], dataId: savedDataId }
   │
   └─F. 파일 메타 재조회
        dataJson 재귀 순회 → fileIds 수집
        GET /page-files/meta?ids=...
        → setExistingFileMetaMap 갱신 (UI 즉시 반영)

⑦ toast.success('저장되었습니다.' | '수정되었습니다.')
⑧ goBackAfterAction=true → router.back()
```

### 3-2. 파일 처리 2단계 구조

```
[STEP 1] 파일 업로드 (저장 전)
POST /page-files/upload
  FormData {
    file:         <File>,
    templateSlug: 'connected-slug',
    fieldKey:     'profileImage',
  }
  → 응답: { id: 101, ... }
  → 파일은 파일시스템에 저장되지만 data_id는 null (임시 상태)

[STEP 2] dataId 연결 (저장 후)
PATCH /page-files/link
  { fileIds: [101, 102], dataId: 1234 }
  → page_file.data_id = 1234 로 업데이트
  → 이제 파일이 해당 데이터 레코드에 귀속됨
```

> **왜 2단계인가**: 저장 전에 파일을 먼저 업로드해야 파일 ID를 dataJson에 넣을 수 있다.  
> 저장 실패 시 업로드된 파일은 orphan 상태가 되지만, 추후 정리 배치 등으로 처리 가능하다.

### 3-6. `buildDataJson` 공통 함수

**위치**: `utils.ts`  
**사용처**: `handleContentAction` (page 모드), `handlePopupContentAction` (popup 모드)

#### 함수 시그니처

```typescript
buildDataJson(
    widgets:        Array<{ type, widgetId?, fields?, contentKey? }>,
    formValuesMap:  Record<widgetId, Record<fieldId, 값>>,
    formFileIdsMap: Record<widgetId, Record<fieldId, number[]>>,  // 기존+신규 합산 완료
    subListRowsMap: Record<widgetId, 처리된 행[]>,               // _rowId 제거, 파일ID 완성
    multiSelectMap: Record<widgetId, number[]>,
): { dataJson: Record<string, unknown>, pkKeys: string[] }
```

#### contentKey 처리 규칙

```typescript
// Form 위젯
if (fw.contentKey) dataJson[fw.contentKey] = section;  // 중첩 저장
else Object.assign(dataJson, section);                   // flat 저장

// SubList 위젯
if (sw.contentKey) dataJson[sw.contentKey] = { rows };
else dataJson.rows = rows;

// MultiSelect 위젯
if (mw.contentKey) dataJson[mw.contentKey] = selectedIds;
```

#### 호출 전 사전 처리 순서 (필수)

```
1. Form 파일 필드 업로드
   fileValuesMap[widgetId][fieldId] 의 신규 File[] 를 업로드
   → formFileIdsMap[widgetId][fieldId] = [...기존MetaId[], ...신규업로드Id[]]

2. SubList 파일 컬럼 업로드 + rows 처리
   각 row의 파일 컬럼 신규 File[] 를 업로드
   _rowId 제거 + 파일 컬럼 ID 배열 완성
   → processedSubListRowsMap[widgetId] = processedRows[]

3. buildDataJson 호출
   → { dataJson, pkKeys }
```

> **왜 순서가 중요한가**: `buildDataJson`은 순수 동기 함수다.  
> 비동기 파일 업로드가 완료된 후에 호출해야 파일 ID가 dataJson에 포함된다.  
> SubList 파일 처리 전에 호출하면 해당 파일 컬럼이 빈 배열로 저장된다.

### 3-7. popup 모드 저장 흐름 (`handlePopupContentAction`)

page 모드(`handleContentAction`)와 저장 로직이 동일하다. `buildDataJson` 공통 함수를 공유한다.

```
[팝업 내 Space 버튼 → 저장]
     │
① 유효성 검사 — form 위젯별 (page 모드와 동일)
   validateFormFields(fwFields, popupFormValuesMap[fwId], popupFileValuesMap[fwId], popupExistingMetaMap[fwId])

② Form 파일 업로드 → formFileIdsMap[widgetId][fieldId]
   popupExistingMetaMap[fwId][f.id].map(m => m.id) + 신규 업로드

③ SubList rows 처리 → processedSubListRowsMap[widgetId]
   파일 컬럼 업로드 + _rowId 제거

④ buildDataJson(allWidgets, popupFormValuesMap, formFileIdsMap, processedSubListRowsMap, {})
   → { dataJson }

⑤ PUT  /page-data/{popupListSlug}/{popupEditId}  (수정)
   POST /page-data/{popupListSlug}                (신규)

⑥ PATCH /page-files/link  (신규 파일 있을 때만)

⑦ onRefresh() → 목록 재조회
   setCategoryRefreshTick() → 카테고리 위젯 재조회
   handlePopupClose()
```

> **page 모드와의 차이**: popup은 단일 slug만 처리 (다중 slug·groupId 없음).  
> 저장 후 `router.back()` 없이 팝업만 닫힌다.

### 4-5. 카테고리 위젯 조회 방식 — dot notation 필드 읽기

테이블과 카테고리는 contentKey 중첩 구조를 처리하는 방식이 다르다.

```
테이블 (fetchTableData):
  dataJson의 모든 object wrapper를 자동으로 펼침 (flat-map)
  → { lv1pop: { title: 'A' } } → flat에 title 직접 추가

카테고리 (fetchItems):
  readField() 유틸로 dot notation 경로를 명시적으로 지정
  → widget.fieldTitle = 'lv1pop.title' 이면 dataJson['lv1pop']['title'] 접근
```

**`readField()` 동작 원리** (`CategoryRenderer.tsx` 내부):

```typescript
const readField = (dataJson: Record<string, unknown>, fieldKey: string): unknown => {
    const dotIdx = fieldKey.indexOf('.');
    if (dotIdx !== -1) {
        const parentKey = fieldKey.slice(0, dotIdx);   // 'lv1pop'
        const childKey  = fieldKey.slice(dotIdx + 1);  // 'title'
        const section   = dataJson[parentKey];
        if (section && typeof section === 'object' && !Array.isArray(section)) {
            return (section as Record<string, unknown>)[childKey];
        }
        return undefined;
    }
    return dataJson[fieldKey];  // dot 없으면 root 직접 접근
};
```

**동작 예시**:

```
dataJson = { "lv1pop": { "title": "대분류A", "code": "A-001" } }

readField(dataJson, 'lv1pop.title')  → '대분류A'   ✓
readField(dataJson, 'lv1pop.code')   → 'A-001'     ✓
readField(dataJson, 'title')         → undefined    (root에 없음)
```

> 카테고리 위젯 빌더에서 카드 필드 키를 `폼key.컬럼명` 형식으로 설정하는 이유가 이것이다.  
> 설정값이 연결 폼의 contentKey와 다르면 해당 필드는 빈값으로 표시된다.

### 금지 8: popup 상태를 widgetId 없는 flat 구조로 관리

```typescript
// ❌ 금지 — flat 구조 (contentKey 중첩 저장 불가, 폼 2개 이상 시 fieldId 충돌)
const [popupValues, setPopupValues] = useState<Record<string, string>>({});
```

> **이유 2**: form 위젯이 2개 이상일 때 fieldId가 겹치면 나중 폼의 값이 이전 폼의 값을 덮어쓰는 충돌이 발생한다.

- [ ] 필수 필드에 `required: true`가 설정되어 있는가? (`validateFormFields` 검사 대상)
- [ ] 수정 모드 진입 시 URL에 `?id=` 또는 `?group_id=`가 올바르게 전달되는가?
- [ ] 다중 slug 저장 시 연관된 모든 위젯의 `connectedSlug`가 다른가?
- [ ] 파일 필드가 있을 때 `PATCH /page-files/link` 호출이 누락되지 않는가?
- [ ] SubList rows 저장 시 `_rowId`가 제거되는가?
- [ ] hidden 필드에 `defaultValue`가 설정되어 있거나 URL 파라미터로 전달되는가?

- [ ] 검색 파라미터 키가 영문/숫자/언더스코어로만 구성되어 있는가? (SQL Injection 방지 규칙)

- [ ] `buildDataJson` 호출 전에 Form 파일 업로드와 SubList 파일 업로드가 모두 완료되었는가?
